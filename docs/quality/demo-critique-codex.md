**結論**
製品成熟度は **2/5未満**。XMLはwell-formedで、小節内の`duration`合計も4/4としては破綻していません。しかし「楽譜として読ませる」段階には達していません。初デモとして見せるなら、現状は **AI採譜の成果物ではなく、MIDIイベントをMusicXML容器に押し込んだだけ** です。

**記譜品質のダメ出し**
1. **`turkish_pd`の三連符はほぼ誤爆です。** 177個の音符/休符に対して`time-modification`が145個、つまり約82%が変則音価扱いです。通常のトルコ行進曲の冒頭をここまで三連化する合理性はありません。`79.5 * 1.5 = 119.25`なので、実態は倍半ではなく **3:2格子選択ミス** です。ベンチにも「トルコ行進曲は2分系120.5」とあり、XML出力がその診断と矛盾しています。

2. **大譜表なしは致命傷です。** 両ファイルとも単一パート・単一ト音記号で、`<staff>`がありません。MusicXMLチュートリアルでもピアノのような複数段譜では`staff`を使うべきとされています。現状だとE2/D2までト音記号上に置かれ、`pd`は64%、`real`は47%の音が加線3本以上です。これは「読みにくい」ではなく「ピアノ譜として成立していない」。

3. **声部が音楽的声部ではなく、帳尻合わせです。** `pd`は全20小節が2声部、`real`は全120小節が5声部。しかも大量の全小節休符で埋めています。MusicXML上の`voice`は線形に進む音楽イベント列ですが、ここでは声部推定ではなく小節充足のための水増しに見えます。`voice`が文字列なので`0`自体は規格違反とは断定しませんが、譜面エディタ上の可読性は悪いです。

4. **休符が細切れすぎます。** `real`は休符932個、休符連鎖585本、最大15連続。連鎖分布は`1:395, 2:94, 3:55, 4:33, 5:6, 7:1, 15:1`。これは採譜ノイズと声部分割の副作用で、譜面清書の後処理が入っていません。

5. **調号/臨時記号が曲を理解していません。** 作品解説上、第三楽章はA minorからA majorへ展開する2/4曲です。`pd`は五度圏0固定で転調なし、`real`は五度圏+3固定で、ナチュラル臨時記号210個を発生させています。特に`real`はC natural 112個、F natural 41個、G natural 30個で、A major固定の副作用が露骨です。

**MusicXML規格・NF-011**
`xmllint --noout`では両方well-formedでした。`duration`もMusicXMLの仕様どおり、`chord`以外の`note`/`backup`/`forward`でカーソルを動かす前提で見ると小節充足はOKです。ただし、MusicXML 4.0 XSD検証はローカルXSD/catalogなしで未達。公式XSDはローカルcatalog検証を想定しており、現状のHTTP DTD参照はオフラインCIで失敗します。NF-011の「XSD妥当性100% + music21 roundtrip + MuseScore警告ゼロ」には届いていません。

**ベンチ解釈**
`results-pd.md`自身がかなり正直です。平均F1@100msはBP素点0.709に対しspike0.676で悪化。`rescue`だけが`score_rhythm`で0.402 vs BP 0.387と僅差で上回りますが、precision平均0.547の代償が大きい。トルコ行進曲は通常構成でF1 0.228、rescueで0.666まで上がるが、これは「過検出で拾った」側面が強く、譜面XMLでは1745音/932休符の過密なゴミとして出ています。

研究水準と比べると距離は大きいです。近年のピアノAMTでは、PiJAMA論文の整理でMAESTRO系のonset F1が0.953-0.968程度、ジャズ実録音でも100ms note F1平均0.891-0.925が示されています。これはMIDI転写指標であって、MusicXML清書はさらに難しい。現状の0.23-0.90のばらつきは「曲によって使える」ではなく、**高密度ピアノで壊れる未安定エンジン**という評価です。

**必須修正リスト**
P0:
- トルコ行進曲を2/4または等価な正しい小節設計へ戻す。4/4固定をやめる。
- 三連格子を採用する前に、2分系/3分系のAIC/BIC的な罰則を入れ、`time-modification`過密を失敗判定にする。
- ピアノは大譜表必須。`<staves>2</staves>`、`<staff>1/2</staff>`、ト音/ヘ音分割を実装する。
- 調号をA minor開始、A major部で転調、または最低でも臨時記号コスト最小化で選ぶ。
- 休符結合、声部削減、全小節休符の整理を入れる。
- ローカルMusicXML XSD/catalog検証とMuseScore import CIを追加する。

P1:
- stem/beamを拍境界ベースで生成する。短音符855個beam欠落はデモ不可。
- `normal-type`が不自然な6:5、12:7系休符を禁止または警告化する。
- `rescue`は密度適応に限定し、疎な曲で自動OFFにする。

P2:
- F1だけでなく、加線数、休符断片数、臨時記号密度、声部数、小節数差分を譜面品質KPIに入れる。
