## 【stem】 field.py — 修正の頑健性: 部分的に破綻

### R1修正の再確認(持った): P0-S1/S2は回帰済み
`_estimate_snr_db`(field.py:33)は#45で無音率検出器→スペクトル雑音床分離に作り直し済み。変異再攻撃で回帰を確認:
```
clean continuous 440Hz 2s -> est_snr=60.00 clean   (R1: 0.0/very_noisy)  ← 治った
sparse notes(gaps)         -> est_snr=42.41 clean   (R1: 182.3)          ← 非物理値解消
```
白色雑音での既知SNR較正も妥当(pink 20/10/5/0dB → 27.1/17.3/12.7/9.1、単調で符号正しい)。

### 【新P0】 R2-S1: 有色(非白色)雑音で雑音床推定が崩壊 — 純雑音を`clean`と誤判定
`_estimate_snr_db`(field.py:37-40 docstring)の中核仮定は「**音程成分は少数ビンに集中・広帯域雑音は全ビンに広がる → 周波数ビンのパワー中央値=雑音床**」。この仮定は**白色雑音でのみ成立**する。実測(`/tmp/r2_verify.py`):
```
PURE white noise (no tone): est_snr= 0.00 very_noisy   ← 正しい(全ビン均等)
PURE brown noise (no tone): est_snr=44.06 clean         ← 誤り(信号ゼロなのにclean)
```
褐色/低域偏重雑音はエネルギーが低域ビンに集中し**大半のビンが静か**なため、ビン中央値≈0となり雑音床が激しく過小評価される。信号がゼロでも `signal = total - noise` が大きく残り、SNRが発散する。混合実測でも:
```
brown noise mix target=5dB -> est_snr=47.82 clean   (真5dBを47dBと誤報)
reverb rt60=1.0/2.5s        -> est_snr=60.00 clean   (残響尾=低域偏重広帯域も雑音と認識できず)
```
- 実環境の雑音(空調・交通・ハムのうなり・残響尾)は**ほぼ全て低域偏重=有色**。白色雑音は実験室にしか無い。
- 害の方向はR1と逆(R1=クリーンを`very_noisy`、R2=雑音まみれを`clean`)。`clean`判定は下流 field_select の過絞りを**発動させない**方向なので致命度はR1より低いが、**降噪モードの起動判断(noise_profile駆動)を誤り、雑音下pYIN崩壊の救済機会を逃す**。
- docstringは「限界: ドラム等の広帯域音楽成分も雑音側に数える」とは書くが、**有色雑音そのものを雑音と認識できない**逆の欠陥は台帳に無い。**主張(スペクトル雑音床分離で実SNR推定)と実態(白色専用)が乖離** → 錨2抵触。
- 判定 **P0**。再現: `PYTHONPATH=$PWD .venv/bin/python /tmp/r2_verify.py`

### R2-S2: 位相反転ステレオの無音を`very_noisy`表示(軽微)
L=-R のステレオはモノ畳込みで完全無音(max abs=0.00e+00)。field は `very_noisy` を返す(field.py:85 無音時の既定ラベル)。物理的には「無音」であり `very_noisy` は不正確だが、下流影響は無い(イベント0で一致)。判定 **P2**(表示の正直性のみ)。なお sig+n/sig-n のステレオ(雑音が畳込みで相殺)は正しく `snr=60/clean` を返す(頑健)。

## 【ear】 mono.py — 修正の頑健性: おおむね持つ / 限界記述が不正確

### R1修正の再確認(持った・強い): ビブラート統合は速度非依存で堅牢
`detect_events`(#46)を速度・非対称で変異再攻撃。**0.5半音ビブラートは4/5/6/7/8Hz 全速度で1音統合**(midi=[69]):
```
vib depth=0.5 rate=4.0〜8.0Hz -> 全て 1 events midis=[69]   ← 速度違いに頑健
```
非対称ビブラート(下方偏位を0.3〜0.5に圧縮)も **depth1.0でも1音**(むしろ振幅が減り安定)。ここは**完成水準**。

### 【P1】 R2-E1: ダッシュボードの「±1.5半音で8断片」は不正確 — 断片化は0.75半音から始まり速度依存
ダッシュボード ear行は「ビブラート±0.5半音まで1音統合 / ±1.5半音で8断片前後」と書くが、実測(`/tmp/r2_ear.py`)は**閾値がもっと低く、かつ4Hz(遅い)側がむしろ悪い**:
```
depth=0.6 rate=6Hz -> 1 event      ← ここまで統合
depth=0.75 rate=4Hz -> 8 events    ← 0.75で既に8断片(4Hz)
depth=0.75 rate=6Hz -> 2 events    ← 同depthでも6Hzは2断片(速度で3倍差)
depth=1.0 rate=4Hz -> 8 events / rate=6Hz -> 8 events
depth=1.1 rate=6Hz -> 10 events
```
- 「±0.5→±1.5の間は緩やかに劣化」という台帳の含意と食い違う。**実測の断片化開始は≈0.7半音**(SPLIT_GAP_SEMITONES=0.8 と整合的)、**遅いビブラート(4Hz)は速いビブラート(6Hz)より断片化しやすい**(窓長 `_SPLIT_WINDOW_FRAMES=7`≈81ms が6Hz半周期前提のため、4Hz=125ms半周期で局所中央値が振動を平均化しきれない)。
- 台帳の「±1.5半音で8断片」は**速度と開始点の両方で楽観的**。錨2(主張と実態の一致)に軽度抵触。判定 **P1**(限界記述の是正=ダッシュボード数値の書き換えが必要)。

### 【P1】 R2-E2: 高速トリルの二重崖 — 音価消失 + 音高マージ誤り
同音連打の高速トリル(A4↔B4交替)を変異攻撃(`/tmp/r2_newangle.py` N3):
```
@200ms/note -> 10 events (69×5, 71×5)   正常
@120ms/note -> 16 events (69×8, 71×8)   正常
@ 80ms/note -> 10 events (71×7, 70×3)   音高マージ誤り(A4=69とB4=71が70に融合)
@ 60/50/40ms -> 0 events                 全消失
```
- 崖1(消失): 音符長 < MIN_DUR_SEC=0.08 で全破棄。60ms音符は emit の `end_t-onset>=min_dur` を割る(`/tmp/r2_verify.py`で確定: 80ms境界)。速いトリル・装飾音・機関銃奏法は常態的にこの下。
- 崖2(音高マージ): 80msでは pitch_tol/STEP_TOL の連続性許容が隣接半音(A4-B4=2半音差だが遷移フレームで中間値)を跨ぎ、midi=70(中間)を生む。
- 弦のトレモロ・トリルは演奏の常態。判定 **P1**(既知の min_dur 限界の延長だが、台帳に「高速トリル/装飾音は下限以下で消失」の明記なし)。

### R2-E3: ポリリズム3:2 はモノ検出で1音に潰れる(既知のモノ限界)
3対2の同時2声(`/tmp/r2_newangle.py` N4)をモノ混合 → ear は **1 event** のみ。単旋律(mono.py)は同時発音を扱わない設計(多声はpoly側)なので**仕様通り**。判定 **—**(mono契約内)。多声は別途poly.pyの範疇。

## 【rhythm】 quantize.py — 修正の頑健性: 堅牢 / 限界記述はむしろ悲観的

### R1修正の再確認(全て持った): 境界値で正しく動作
境界テンポ 59/60/61 と 179/180/181/182 で変異再攻撃(`/tmp/r2_rhythm.py`):
```
eighths@59 -> est=118.0 warn=True    ← 範囲外(<60)を検出し警告発火(正しい)
eighths@60 -> est=120.0 warn=False   ← 境界ちょうどは倍テンポだが警告なし
eighths@61/62 -> est=122/124 warn=False
quarters@179〜182 -> est=89.5〜91.0 warn=False   ← 速い側は数学的に検出不能(台帳通り)
```
- **偽警報ゼロを維持**: 範囲内の真テンポ(90/108/120/140)は warn=False で確定(`/tmp/r2_rhythm.py`)。遅い側(45/50/59)は全て warn=True。警告の線引きは正確。
- quantize_events の境界bpm(59.9/60/180/181)は全て例外なく処理。R1のbpm=0/負ガード(quantize.py:256)も維持。

### R2-R1: 三連符の有効域は「96-120」でなく実測「**94-120**」— ダッシュボードが悲観的
ダッシュボード rhythm行は三連有効域を「96-120(72/90は2分系エイリアス)」と書くが、境界を細かく走査すると **94BPMも三連検出成功**:
```
triplet@90 -> (135.0,4) binary-MISS
triplet@94 -> (94.0,3)  TRIP        ← 台帳の下限96より下でも成功
triplet@96 -> (96.0,3)  TRIP
triplet@120 -> (120.0,3) TRIP
triplet@122 -> (92.0,4) binary-MISS  ← 上限は120と122の間
```
- 実態は台帳より**わずかに良い**(下限94)。誠実性の観点では悲観記述は害が小さいが、**線が正確でない**(錨2)。判定 **P2**(ダッシュボードの数値を94-120に更新推奨)。

### R2-R2: 速い側テンポの静かな倍半誤り(台帳済みだが実害の再確認)
`quarters@179-182 → est≈90, warn=False`。179bpmの4分音符曲が**警告なしで90bpmの2分音符**に化ける。台帳は「速い側は 4分@200≡8分@100 で数学的に検出不能」と正直に記録済み(quantize.py:130)で、警告不能も設計上の帰結。判定 **—**(限界が台帳化済み・主張と実態一致)。ただし遅い側は警告するのに速い側は無警告という**非対称**は、ユーザーから見ると「なぜ警告が出ないのか」の混乱源になりうる(改善候補: 範囲上端付近のテンポには弱い注記を出す案)。

## 最重要発見(1つに絞るなら)

**stem の #45 SNR作り直しは「白色雑音専用」の限界を隠している。** `_estimate_snr_db` の中核仮定「周波数ビン中央値=雑音床」は白色雑音でのみ成立し、実環境で支配的な**有色雑音(褐色・残響尾・低域ハム)では純雑音すら `est_snr=44dB → clean` と誤報**する。R1はクリーン持続音を`very_noisy`にする欠陥を直したが、その代償として逆方向の欠陥(雑音を`clean`)を白色前提の較正で作り込んだ。docstringが「実SNR推定」を主張する以上、**白色専用という限界を台帳化するか、有色雑音に頑健な床推定へ再設計する**まで stem は未完成。214テストはこの欠陥を素通し(有色雑音の純雑音入力を検査していない)。
