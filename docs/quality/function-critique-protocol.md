# 機能別ダメ出しラウンド規約

**目的:** エンジンの各機能（stem / ear / rhythm / notate / engrave / quality）を個別に厳しく批判し、機能ごとの誠実な現在地を確定する。

## 基準（3つの錨）
1. **受入条件** — core-requirements-v3 の該当項目・関連要件（仕様との一致）
2. **封緘済み実測** — AIの耳・正解つきPDベンチ・XML/信号解剖の客観指標
3. **外部水準** — 研究SOTA・競合の公表水準との距離

好みの指摘・基準外の指摘は不採用。

## 証拠義務（批判者の3行動）
- コードを実際に読む（file:line引用必須）
- 実測を自分で再実行 or 再解剖する（数字の引用必須）
- **未テスト入力で新規攻撃する**（反例の新規生成 — 最重要）

引用のない指摘は無効。

## 完成の定義（機能ごと）
- (a) P0/P1指摘が修正され**テストで固定**（テスト緩め禁止）
- (b) **品質ダッシュボード行の確定**: 現在スコア / 既知の限界の正直リスト / 次の改善候補
- (c) 全テスト緑

「未達だが理由・限界が記録済み」も完成の一形態。完成＝満点ではなく、**主張と実態が一致し数字と限界が台帳化された誠実な現在地**。

## ループ規則
- 各機能: 批判→修正→再確認の**最大2巡**。収束しない課題はIssue化して次へ
- ラウンド終了条件: 全6機能で (a)(b)(c) 充足
- ダッシュボード: docs/quality/function-dashboard.md に機能×スコア×限界×次候補を集約
