## Phase 6: Electron デスクトップアプリ MVP 開発

### 方針決定
- WebUI形態: Electron（ローカル処理C7必須）
- TABは後回し。まず「音源を渡したら楽譜が出る」体験を作る

### エンジン調査で判明した統合設計の要点
- CLI: `python -m earpipe.pipeline transcribe <input> -o <musicxml> --pdf <pdf>`
- JSON stdout 出力 → `child_process.spawn` で簡単にパース可能
- PDF生成はMusicXMLの後工程（2ステップ）
- 処理時間: mono約5秒 / poly約30秒

### 実装内容
`app/` ディレクトリを新規作成:

| ファイル | 役割 |
|---|---|
| `package.json` | Electron 32 + スクリプト |
| `main.js` | Mainプロセス・Pythonサブプロセス管理・IPC |
| `preload.js` | contextBridge（セキュリティ境界） |
| `renderer/index.html` | UI骨格（3ステート: idle/processing/done） |
| `renderer/style.css` | ダークテーマ・波形アニメーション |
| `renderer/app.js` | レンダラーロジック |

**UIの3ステート:**
1. **idle**: 波形アニメーション付きドロップゾーン
2. **processing**: プログレスバー + ログストリーミング
3. **done**: PDF表示 + 採譜統計 + エクスポートボタン

**起動確認**: `npm start` でウィンドウ表示まで確認済み

### バグ修正: Electron 32 の file.path 問題
Electron 32 で `file.path` が非推奨・空文字を返すケースがある。`webUtils.getPathForFile(file)` に修正:
- `preload.js`: `webUtils` を import して `getPathForFile` を expose
- `app.js`: ドロップ時に `window.earpipe.getPathForFile(file)` を使用

### 採譜テスト結果
音源（複数言語タイトルのポップス）でテスト → **結果がスカスカ（数音符のみ）**。原因: モノラル（pYIN）エンジンは単旋律を想定しており、複雑な楽曲から音を拾えていない。polyモード（Basic Pitch）が必要。

**Issue #61** を作成（事後だったが Issue 駆動の原則に従い作成）
