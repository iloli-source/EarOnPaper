# ADR-001: エンジンのサービス分割（ローカル・マイクロサービス方針）

**日付:** 2026-07-19
**状態:** 採択（ユーザー決定: 「メインエンジンの機能はマイクロサービス化してそれぞれの品質を上げていくイメージで」）
**関連要件:** NF-007（部品交換可能性）/ NF-050（二層構造）/ NF-045（プラグイン出力層）/ NF-001（完全ローカル動作）

## 決定

メインエンジンを**サービス境界で分割**し、各サービスが独立に品質改善できる構成とする。

| サービス | 責務 | 品質KPI（AIの耳/独自指標との対応） |
|---|---|---|
| stem（分離） | 音源→ステム分離・前処理 | 分離品質・下流採譜への寄与（note-wise評価） |
| ear（耳） | 音声→音程イベント+信頼度（楽器非依存・NF-050） | 音高一致度・多声検出F1 |
| rhythm（律動） | 拍・テンポ推定→量子化 | 出だし一致・テンポ整合・拍子妥当性（NF-022） |
| notate（整譜・記譜） | 音符列→譜面（出力プロファイル層NF-045へ接続） | 譜面健全性・MusicXML妥当性（NF-011） |
| quality（品質） | 全サービスの審判（AIの耳ハーネス） | 自己検証スイート |

## 設計翻訳（ネットワークマイクロサービスとの違い）

- 完全ローカル前提（NF-001/成果物非関与）のため、**HTTP/gRPCの常駐サービス群ではなく、プロセス/モジュール境界＋型付き契約インターフェース**で実現する（配布・オフライン・プライバシーの制約に適合）
- 各サービスは: ①スキーマ定義された入出力契約 ②独自のテストスイート ③独自の品質KPIとベンチ ④独立したバージョン、を持つ
- 差し替え可能性が本体: 例）ear を pYIN→Basic Pitch ONNX→自社モデルへ、他サービス無変更で入れ替え、qualityサービスで前後比較して採否判定
- 将来クラウド版を作る場合も同じ契約でサービスを再配置できる（境界は保存される）

## 帰結

- 改善ループ: 1サービスを変更→そのサービスのテスト＋契約テスト＋qualityの前後比較→採否。全体の作り直しなしに部品ごとに品質を上げる
- spike/ear-pipeline は本ADRに従い ear / rhythm / notate / quality クライアントのサービス構成へ再編する（多声化v0.2完了後に実施）
- 各サービスの品質履歴（バージョン×KPIスコア）を台帳化し、回帰評価CI（NF-021）に接続する
