# ADR-002: 開発手法の採用とコンフリクト解消

**日付:** 2026-07-19
**状態:** 採択（ユーザー確認: 「開発手法のコンフリクトは起きないか・issue駆動になっているか」への回答として明文化）

## 採用する手法（確定）

| 手法 | 役割 | 実装状況 |
|---|---|---|
| **EDD（評価駆動開発）** | 背骨。品質は事前封緘した評価基準で測る | AIの耳ハーネス・封緘判定・ベンチ群（稼働中。ONNX破損・指標欠陥を実際に検出） |
| **TDD** | 関数・モジュールの正しさ。テスト先行・緩め禁止・カバレッジ80%+ | spike 37テスト・cov95%（RED→GREEN記録あり） |
| **Issue駆動** | 全開発作業はIssue起票→着手→進捗/検証記録→結果つきクローズ（devラベル） | 2026-07-19運用開始（それ以前の作業は遡及記録#24-26） |
| **ADR** | アーキテクチャ決定の記録 | ADR-001（サービス分割）〜 |
| **サービス分割** | エンジンをstem/ear/rhythm/notate/qualityに分割、各自の契約IF・テスト・KPI | ADR-001。コード再編はレビュー(#29)完了後 |

## 未採用・保留（正直な記録）

- **DDD戦術パターン・クリーンアーキテクチャの依存規律**: 二層構造・サービス境界はDDD的な境界づけと親和するが、戦術パターン（エンティティ/リポジトリ等）の採用は本実装段階で判断
- **ジョブキュー**: 長時間処理（NF-004/NF-040、24時間音源対応）の実装段階で必要になる見込み。spike段階では不要（YAGNI）

## コンフリクト検査の結果

| 組み合わせ | 判定 | 解消方法 |
|---|---|---|
| マイクロサービス × 完全ローカル（NF-001） | **衝突あり→解消済み** | ADR-001の翻訳: ネットワークサービスでなくプロセス/モジュール境界＋契約IF |
| TDD × EDD | 共存 | 役割分担: ユニットテスト=関数の正しさ／ベンチ=品質スコア。スイート物理分離（tests/ vs bench/） |
| TDD × Issue駆動 | 共存 | Issue→テスト先行→実装→Issueに検証記録 |
| EDDの封緘 × TDDの「テストを直すな」 | 共存 | 共通原則「基準・テストを緩めて通すの禁止。変更は理由記録つき手続きのみ」 |
| サービス分割 × TDD | 共存 | サービスごとのテストスイート＋契約テスト |

## 運用フロー（1つの開発タスクの標準形）

Issue起票（devラベル・受入基準記載）→ テスト先行（RED）→ 実装（GREEN）→ ベンチ/AIの耳で品質確認（EDD）→ code-reviewerレビュー → Issueに検証結果を記録してクローズ → コミット（conventional形式）→ push
