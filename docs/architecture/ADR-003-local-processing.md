# ADR-003: 完全ローカル処理型アーキテクチャ（成果物非関与構造の技術設計・Issue #10)

**日付:** 2026-07-19
**状態:** 採択（設計検証の結果を記録。法務最終確認はG3弁護士4問に依存）
**関連:** NF-001/NF-013/NF-023/NF-050 / ADR-001 / gate-execution-spec（G3）

## 1. 背景

ロクラクII「枢要な行為」論により、サーバー側で複製・変換を行うSaaSは事業者=複製主体と認定されるリスクがある（round1-legal-caselaw）。一方、免責の判断軸は「ローカルかクラウドか」ではなく**「①コンテンツを利用者が用意 ②事業者が成果物に非関与 ③自動処理に留まる」の三点セット**（conclusions-v4 §2.2）。ローカル処理は③を最も立証しやすい実装として採用する。

## 2. データフロー（音源が端末外に出ないことの設計証明）

```
[ユーザー音源] → stem → ear → rhythm → notate → [楽譜/MIDI/MusicXML]
      │      （全サービスがユーザー端末内のプロセス。ADR-001の境界）      │
      └──────────── ネットワーク境界を一度も越えない ────────────┘
外部通信が発生しうるのは: アプリ/モデルの更新取得（下り専用）・オプトインの匿名テレメトリ（NF-009: 音源由来データ0件）のみ
```

- **上り通信ゼロが既定**（NF-013: 既定で外部送信なし・送信機能は明示同意制）
- 検証で実証済み: spike v0.2（Basic Pitch TFLite・pYIN・量子化・記譜）は全段オフラインで動作。AIの耳（計算の耳）もローカル完結

## 3. 技術選択肢の整理（Issue #10「やること」①）

| 形態 | 三点セット適合 | 制約 | 判定 |
|---|---|---|---|
| デスクトップアプリ（現行路線） | ◎（全処理端末内） | 配布に署名/公証（NF-014/037）・モデルサイズ（NF-030） | **主線** |
| ブラウザ内WASM/WebGPU | ◎（端末内実行） | モデルサイズ・性能・ブラウザ差。NF-043のランタイムゲート対象 | 将来オプション（配布容易性） |
| サーバー側SaaS | △〜×（③が崩れやすい） | ロクラクIIリスク直撃 | 不採用（G3回答までオープンだが計画に置かない） |

## 4. モデル更新配信の継続関与性への対処（G3付随論点）

- リスク: モデル更新の継続配信が「管理・支配」の徴表として関与性を補強しうる（conclusions-v4 legal穴4）
- 設計対処: 更新は**汎用ソフトウェアアップデート**として実装（特定ユーザーの特定成果物に対する個別関与を構造的に不可能にする）。更新サーバーはユーザーの音源・成果物・利用内容を受信しない（NF-009/013と同一の規律）。この整理の妥当性はG3弁護士4問の付随論点として確認する

## 5. 外部AI利用の境界（AIの耳・第二の耳の規律）

| データ | 外部AI（Gemini等）送信 | 根拠 |
|---|---|---|
| ユーザー音源・ユーザー成果物 | **禁止** | 成果物非関与・私的複製の外部化回避（G0'台帳の運用で実施済み） |
| 合成データ（自前生成・正解既知） | 可 | 権利者不存在 |
| PD素材（pd-corpus等、PD確認済み） | 可（出所記録つき） | PD |
| 公式配信音源（検証用DL分） | 禁止（ローカル計算の耳のみ） | 私的利用の範囲維持 |

## 6. 「忠実採譜=複製の範囲」の機能線引き（Issue #10「やること」②）

- エンジン既定は**忠実採譜**（聞こえた音程イベントの記譜）。これは原曲の複製/翻案の範囲に留まる
- **創作的編曲の自動生成はエンジンに置かない**（F-026難易度アレンジ等はWon't済み）。編曲権（round1-legal-henkyoku）の主体論点を製品側に持ち込まない
- ユーザーによる手動編集は利用者の行為（ソフトは道具）。エディタ機能は三点セットを崩さない
