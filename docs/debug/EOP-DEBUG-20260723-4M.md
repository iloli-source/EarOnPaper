## 1. 結論

現行ZIPには、単体テストが緑でも実利用で事故になる不具合が残っていた。特に、古い非同期処理が新しい入力へ上書きする競合、子プロセスと一時ファイルの無制限化、symlink経由の管理外ファイル保存、URL取込結果の過信、長尺分割の非有限値入力、任意依存1個の欠落で無関係な機能までimport不能になる構造は、放置できない。

今回、重大経路を修正し、4ジョブ各100万件、合計400万件の決定的な境界値・プロパティ実行は全件通過した。初回修正版の梱包後再監査では、今回追加した資源追跡がURL取得音源を分離直前に削除する退行も発見した。初回成果物と旧400万件証跡は採用せず破棄し、資源種別分離・進捗トークン・旧処理停止を追加した現行ソースで400万件を最初から再実行した。ただし、この400万件は音源400万本を採譜した結果ではない。パス、URL、数値設定、TAB音域折畳み、長尺分割入力の実装関数を400万回実行した証跡である。実曲の採譜精度、Demucs/basic-pitchの実動作、Electron GUIのE2E、2時間・24時間音源の完走は証明していない。

そのため、ソース修正は受領可能なB判定だが、製品出荷はC判定で停止する。music21/verovioを含む正式依存環境と実音源コーパスでの再検証が必要である。

## 4. 初回修正版を採用しなかった理由

初回の修正・400万件実行・梱包まで進めた後、コードをURL取込から分離処理まで手作業で再追跡した。その結果、D4M-024を発見した。400万件ジョブは純関数・境界ガードを検査していたため、Electron IPCをまたぐ「URL元音声ルートの寿命」までは検出していなかった。

この事実は、件数が多くても対象範囲外の統合不具合は検出できないことを示す。初回ZIP、初回patch、旧圧縮結果ID `EOP4M-5E74674CD274C3EB` は納品対象から除外した。D4M-024〜030を修正し、Node回帰テストを追加した後、変更後の`platform-utils.js` SHA-256を含む400万件証跡を再生成した。中間ID `EOP4M-CEDEB179D222881C` も、新規の特殊URL表記とURL元音声保持シナリオを400万件ジョブ内へ追加するため破棄し、最終ID `EOP4M-128BC952C365ACB9` で再々実行した。

## 6. 残存リスク

### R1. 実曲精度は未検証

合成・純関数テストが通っても、音漏れ、残響、ライブ録音、歪みギター、ピアノ、歌声、テンポ揺れでのNote F1や小節整合は保証できない。実曲コーパスでの合格基準が必要。

### R2. 2時間・24時間は未達扱い

`chunk`機能は存在するが、現行Electronの標準フローは音源全体を分離・採譜へ渡す。2時間保証・24時間対応を名乗れる接続と実測証跡はない。

### R3. URL取込のDNS再解決

文字列上のlocalhost・プライベートIPは拒否したが、公開ドメインが後から私設IPへ解決されるDNS rebindingをyt-dlp側まで固定する仕組みはない。高信頼環境でURL取込を公開するなら、ネットワークサンドボックスまたは機能無効化を検討する。

### R4. Windows/macOS実機未確認

パス・taskkill・sandbox・PDF表示・ファイルダイアログは実機差がある。Node純関数だけではElectron実機の挙動を証明できない。

### R5. 要件正本と実装の衝突

URL直接入力は要件書でWon't、実装では有効。法務・権利方針を含むため、単なる文書修正ではなく意思決定が必要。
