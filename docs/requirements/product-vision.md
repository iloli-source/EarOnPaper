# プロダクトビジョン: 絶対音感エミュレータ

**記録日:** 2026-07-19（ユーザーの言葉から）
**位置づけ:** プロダクトの核心仮説の候補。要件一覧・UI/UX方針の上位にある「なぜ作るか」の言語化

## ユーザーの原文（感覚値）

> 日常に溢れている雑音を含めた音を録音してこのアプリに渡してあげたら、絶対音感を持っている人と同じように音符として出力されるのが良い

## 翻訳: これは何を仕様として言っているか

絶対音感の人は雨音そのものを音符として聴くのではなく、**雑音の中から音程を持つ成分だけを拾って音名に変換**している。したがってこのビジョンは:

1. **入力前提の転換** — 「クリーンな音楽ファイル」前提を捨て、雑音混じりの日常録音（フィールド録音・スマホ録音・ライブ録音）を第一級の入力として扱う
2. **音程イベントの選択的抽出** — 音程を持つ成分（ジングル・鳥・鼻歌・楽器・声の抑揚）を音符化し、音程のない成分（雨・足音・ノイズ）は無視または分類して正直に報告する
3. **確信度の正直な提示** — 拾えたものだけを信頼度つきで出す。「進化する下書き」（UI/UX方針の差別化核30%）と直結

## 市場的意味

- 2巡の網羅調査で競合は全社「単一のクリーン音楽ソース前提」。日常音採譜はどの言語圏にも存在しない空白
- 「街の音が楽譜になる」はSNS共有性の高い感情的フック。耳コピ学習層の「世界が音名で聴こえる体験」への憧れに直結
- 既存要件との接続: F-002音質診断 / F-003ステム分離 / F-005録音入力 / NF-032頑健性軸 の延長線上

## 技術的な正直な見立て

- **狙える**: 音程が明確な日常音（駅ジングル・鳥・ハミング・単一楽器の生活音）の音符化。頑健な前処理＋音高抽出で現実的
  - ※「準備（前処理）が最終精度の8割を決める」という共有動画由来の説について: **学術的裏づけは未確認** — 根拠があるのは「SNR低下・残響・録音環境差がAMT精度を大きく落とし、前処理・augmentationが頑健性を改善する」という範囲まで（waveform-codex-papers 2026-07-19）
- **研究水準超え**: 雑踏など複雑なシーン全体の完全音符化。ここは「拾えたものだけ正直に出す」設計が現実解
- ベンチ（G1）の頑健性軸（クリーン/歪み/ライブ録音別）に「フィールド録音」区分を追加する価値あり

## 第二の洞察: 楽器にこだわらない（二層構造）

**ユーザーの言葉（2026-07-19）:** 「それだったら楽器にこだわらなくてよいじゃん」

このビジョンの帰結として、アーキテクチャは二層に分かれる:

| 層 | 性質 | 内容 |
|---|---|---|
| **耳（コア）** | 楽器非依存 | 音→音程イベント＋信頼度。ギターも鳥の声も同じ処理。NF-007のパイプライン前段 |
| **記譜（出力）** | 楽器・用途依存 | TAB（弦/フレット）・五線・简谱・ドラム譜・歌声合成…はすべて**出力プロファイルの選択肢**。NF-045プラグイン型出力層がこの思想の実装 |

- 懸案とされた「プロダクトコアの物語（TAB特化 vs 二本柱）」はこの見方で解消: **コアは楽器非依存の耳、楽器は出力オプション**であり、二択ではなかった
- **ユーザー明確化（2026-07-19確定）:** 「TAB特化は要求していない。**デフォルトは五線譜、TABはオプション、他の全形式もオプションで網羅的に出力**」— これがプロダクトの出力形の確定仕様（F-104/NF-045）。TAB特化案は批判ループ内のgrok提案であり不採用。G0'後に決めるのは出力プロファイルの磨き込み順序のみ
- **表示デフォルトの裁定（2026-07-19確定）:** 批判第3巡grok「非音楽家は五線が読めない」指摘に対し、ユーザーは**一次画面を含めデフォルト五線譜**と裁定。音名表示・鍵盤・ピアノロール・再生などの非音楽家向け表示はオプションとして提供（F-040/F-042等の既存要件でカバー）。批判は記録として保存し、G2需要検証B系統（非音楽家）の実データで再評価可能とする
- ただし「演奏できる譜面」には楽器文脈が必須（弦・フレット選択、キット割当、左右手分割）。**耳は普遍、財布は楽器別** — 設計思想は楽器非依存、売り方・出力の磨き込みは楽器プロファイル単位、が両立解
- G0'/G1実測はこの仮説の検証を兼ねる（競合の崩れ方が「耳」起因か「記譜」起因かの切り分け）

## 判定哲学: 音楽家の基準に引っ張られない（2026-07-19）

**ユーザーの言葉（要旨）:** 「音楽家の判定基準（手直しの苦痛）に依存する枠組みは間違い。その音楽家の方々の基準に引っ張られてしまう」

- 本製品のターゲットは非音楽家（絶対音感エミュレータ）。したがって品質判定の主軸は二段: **①事前封緘した客観指標**（AIの耳ハーネス `tools/ai-ears/`: 音高一致度・音の出だし一致・テンポ整合・譜面健全性。閾値は実測前に封緘し、続行バイアスは「第三者の耳」ではなく「基準の事前固定」で抑える） **②ターゲットユーザーの聴感**（非音楽家が元録音と採譜結果の再生音を聴き比べ「同じに聞こえるか」を3段階判定 — ターゲットユーザー本人が最適な判定者であり、楽譜を読む必要はない）
- 音楽家による評価（手直しの苦痛・演奏可能性）は廃止ではなく**適用範囲の限定**: 演奏者向け出力オプション（TAB・五線PDF等）のQA専用。音楽関係者のつてによるレビューは、五線譜PDF出力の実現後の製品レビューに温存する
- gate-execution-spec §2/§3・g0-prime-kit はこの哲学で改訂済み（手続き改訂2）

## 調査による裏づけ（2026-07-19、3方向網羅調査）

出典: `docs/research/absolute-pitch-codex-papers.md`（論文）/ `absolute-pitch-grok-x.md`（X日英中＋印）/ `absolute-pitch-web.md`（Web一般）

**科学的実態（確立知見）**: 絶対音感は「全ての音を常時音名化する超能力」ではなく、**音高カテゴリと言語ラベルの連合記憶**。音色・音域・文脈で精度が変わり（ピアノ音は高精度・純音や非楽音では低下）、当事者も「混ざった音・中間の音は音符化できない。聞こうと思えば聞ける（選択的注意）」と証言。**つまり F-108 の「拾えたものだけ正直に出す」設計は、絶対音感の実態の忠実なエミュレーションであり過大主張ではない**。成人習得研究は「不可能→一部可能」に転じているが自然APと同質かは論争中。

**物語のクリアランス**: 「絶対音感エミュレータ＝日常音採譜」を使う既存プロダクトは不在（空白）。既存の Perfect Pitch 系アプリは全て耳トレ・テスト用途で別物。ただし「絶対音感/perfect pitch」は一般名詞で商標識別力がないため、**製品名には使わず造語候補（naming-clearance参照）を使い、説明コピーとして用いる**。

**市場通用度（grok判定）**: 条件付きで通用、総合 B+（日本 A- / 英語圏 B / 中華圏 B- / 南アジア C）。
- **勝ち筋フレーミング**: 「絶対音感者のように、雑音の海から“音程のあるもの”だけをつまみ出せる耳 — 注意と選択のエミュレーション」。当事者の科学的訂正と整合し、非当事者の憧れとも接続する
- **負け筋（禁止）**: 「AIが本物の絶対音感を獲得」等の生物学的誇称、「生活音がすべて美しいメロディに」等の全量音名化の約束

**先行例と初期ターゲット**: Tune Smithy が鳥の歌の採譜を実装済み（先行例）。鳥・駅ジングルなど音程が明確な日常音が初期実証ターゲットとして妥当。

**わからなかったこと（正直な報告）**: ヒンディー語圏話者を対象にした絶対音感の直接研究（保有率・訓練）は論文・X需要シグナルとも今回の調査では確認できず。インド古典音楽は相対音感文化（sa=可動）であり、進出時は shruti 文化に合わせた別設計が必要という示唆のみ得られた。
