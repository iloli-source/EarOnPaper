**作成日:** 2026-07-19
**位置づけ:** フル構想カタログ（functional-requirements.md v2.7・117件）から選抜した「最初に作る核」の実装者向け要件定義。カタログは地図として存続し、本書が工程表。
**決定の性格:** ユーザーの包括自動化委任（2026-07-19「全部自動でやってよ」）に基づく**AI決定。全項目にユーザー拒否権あり**（一言で差し戻し・変更可能）。
**判断材料:** G0'判定確定（台帳§10・続行）／PD正解つきベンチ15曲・リズム解剖（rhythm-autopsy.md）／spike v0.2実績（74テスト・score_rhythmでBP素点超え）／ビジョン（絶対音感エミュレータ・非音楽家・デフォルト五線）。

## 3. コア8件の受入条件（実装レベル）

**共通の測定基盤:** AIの耳ハーネス（tools/ai-ears・pytest 49件）＋PD正解つきコーパス15曲（pd-corpus-catalog.md）＋score_rhythm（正解あり時の正指標）。ベースライン数値は2026-07-19実測値。

### C1 多重音高検出
- 入力: 22.05kHz mono wav（PDコーパス・レンダリング／実録音）。出力: 音程イベント列（onset/offset/midi/confidence、JSON契約=bp_worker互換スキーマ）
- 合格: PD15曲でNote F1@100ms ≥ BP素点平均0.709を非劣化維持しつつ、オンセット検出（旧F-011）は許容窓±50msでのマッチ手順を固定。無音・ノイズのみ入力で音符ゼロ（既存ネガティブテスト維持）
- 基準ピッチ: A=440±30centsのズレを推定し補正（3曲以上の変則ピッチ合成ケースで検証）

### C2 テンポマップ生成
- 入力: C1のイベント列。出力: 区間別テンポ系列（一定テンポ曲は単一値に退化）
- 合格: 合成マトリクス（60-150BPM×単音/密多声/幽霊混入）誤差±5%（#34で13/13達成済み=回帰維持）。追加: 三連符曲（Romanze）の正解96BPM復元をP1出口条件に含める（現状不正解=既知の山）。説明不能時はデフォルト退避を正直に返す
- ルバート: 緩急のある実録音（real-audio照合ペア）でテンポ系列の区間分割が破綻しない（クラッシュ・発散ゼロ）

### C3 リズム量子化（二重表現）
- 構造: **実タイミングを保持したまま**格子表現を併記する二重データモデル（格子スナップで原データを破壊しない。API/保存形式で両方に到達可能）
- 合格: PD15曲 score_rhythm 平均で**BP素点(0.387)を上回る**（現状 rescue構成0.402=達成済みの回帰維持）。拍子整合（旧F-022）: 4/4曲に11/8等を出力しない検証をPD全曲で0件。密度適応（疎な曲での劣化≤0.05）を新規に達成する
- 音価クラス一致率をscore_rhythm内で計測・記録（現状両者低=改善余地の追跡指標）

### C4 調整合ピッチスペリング
- 入力: 音符列＋推定キー。出力: 調号と整合する異名同音の綴り
- 合格: PDコーパスの正解MusicXML（調号既知）に対し臨時記号の綴り一致率≥90%。転調を含む曲（Romanze: Em→E）で調号切替に追従。キー推定（旧F-013内部化）はPD15曲で主調正解率≥80%

### C5 五線譜生成
- 入力: C3の格子表現＋C4の綴り。出力: 大譜表を含む五線譜（music21 Score）
- 合格: PD15曲全てで小節数が正解と一致（±1小節以内）・拍子/調号ヘッダ正しい・連桁が拍境界を跨がない（自動検査）。一次画面のデフォルト表示（ユーザー裁定の実装確認）

### C6 MusicXMLエクスポート
- 合格: 出力がXSD妥当性100%＋music21ラウンドトリップで音高/音価/小節の差分ゼロ＋MuseScore（無料）でのインポート警告ゼロを対象アプリ検証の最小セットとする（旧F-052統合）。圧縮.mxl対応

### C7 完全ローカル処理モード
- 合格: 採譜〜出力の全経路でネットワーク送信ゼロ（パケット監視での実証をCI化・ADR-003のデータフロー図と一致）。オフライン（ネットワーク遮断）状態での全コア機能の動作テスト

### C8 フィールド録音モード
- 合格: v2.7の受入条件（分類タグ6種・受入デモ3種=ノック/雨+口ずさみ/エアコンhum・単音抽出優先）をテスト化。フィールド録音合成セット（PD曲+雑音ミックス、SNR 3段階）で「音程成分の選択抽出率」と「非音程の誤音符化率」を計測しベースライン記録（初回は記録のみ・閾値は実測後封緘）

## 4. Mustラベル改訂表（カタログへの反映は次回改訂時）

**Must維持（コア+構造/法務）:** 機能=C1〜C8＋F-001（計9件）／非機能=NF-001・009・011・013・022・023・024・025・026・027・029・045・050（計13件）
**Must→Should降格（25件）:** 機能16件=F-003・013・014・078・023・024・025・028・076・077・032・038・039・042・043・045・046・049・050・052・068・071のうちコア統合済み（F-013/052等）を除く実質16件（TAB系5件は「第3陣で再昇格予定」を注記）／非機能6件=NF-004・032・037・038・043・046・047（署名NF-037等は「リリース時必須に再昇格」注記つき）
**効果:** Must 54件 → **22件**（コア9＋構造/設計原則13。NF-027は2026-07-19廃止のため実質12）。「全部がMustなら何もMustでない」批判（grok 41点・Must過積載）への回答。

## 5. 完了の定義

コア8件の受入条件が全て自動テスト/ベンチとして実装され、グリーンかつベースライン非劣化を2週連続維持した時点で「コア完成」。その時点で第2陣（編集UI最小核）の分冊を作成する。
