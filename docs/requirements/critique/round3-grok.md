## 1. 【ビジョンとカタログの整合】構成比は破綻している

### 1.1 ビジョン直結を数える

`product-vision.md` が明示するコアは次の3点だけだ。

1. 雑音混じり日常録音が第一級入力  
2. 音程成分の選択抽出＋非音程の正直な報告  
3. 非音楽家がターゲット、評価は客観指標＋非音楽家聴感

この定義で **117件を分類**するとこうなる。

| 層 | 定義 | 件数（機能） | 代表ID |
|---|---|---:|---|
| **A. ビジョン本体** | 日常音・フィールド・魔法の可視化を名指し | **2** | **F-108**, **F-109** |
| **B. ビジョンの前提部品** | 録音・音質・信頼度・聴き比べ・非楽音除外 | **約8〜10** | F-001/002/005/010/038/042/043/054/082 |
| **C. 音楽家向け生産装置** | 整譜・TAB・奏法・エディタ・MusicXML方言・計器 | **約90+** | 編集UI25 + 整譜17 + TABクラスタ11 + 出力専門 ほか |
| **D. 運用・法務・試用** | ローカル・権利・課金 | **約10** | F-066/071/073 等 |

**厳密に「F-108ビジョン直結」と呼べる新規の物語機能は 2件（F-108, F-109）**。  
関連部品を最大に寄せても **10件前後 ≒ 全体の 8〜9%**。  
残り **90%超は音楽家が楽譜を仕上げる製品**の語彙である。

### 1.2 MoSCoWがビジョンを否定している

ここが第3巡の新傷の中心だ。**優先度の階層が物語と逆立している。**

| 対象 | 現行 | 意味 |

| **F-108** フィールド録音（ビジョン直結と自己宣言） | **Should / Phase2** | コア仮説が「後でやる便利機能」 |

ビジョン文書は「コアは耳、記譜は出力オプション」（`product-vision` / **NF-050**）と書いた。

これは二層構造を**言語では採用し、優先度では破棄**している状態だ。

## 2. 【非音楽家UXの実需検証】五線をデフォルトにするのは製品事故

### 2.1 「音符になった」のあと、非音楽家は何をするのか

ビジョンはターゲットを非音楽家に置いた。  
だがカタログは **結果の消費形態を「五線（F-031 Must）＋PDF（F-050 Must）＋MusicXML（F-048 Must）」** に固定している。

楽譜が読めない人が五線を受け取った瞬間に起きるのは:

1. 魔法の一瞬（「音符になった！」）  
2. **読めないので何もできない**  
3. 共有するか捨てる  
4. 翌日開かない

つまり **ジョブが「変換のワンショット」で死ぬ**。  
カタログは「編集して仕上げる」前提の成功指標（uiux-direction: 1000音符60fps / UndoでAI再解析前に戻る / 初見10分で修正）を置いており、**非音楽家のジョブが未定義のまま編集KPIだけが完成**している。

### 2.2 非音楽家が欲しい出力は足りているか

| 欲しい出力（仮説＋X言説） | カタログ上の位置 | 判定 |
|---|---|---|
| **音名表示（ド/C4 等が第一級）** | **不在**。F-100は移動ド・Could/P3。絶対音名のタイムライン表示要件なし | **致命欠落** |
| **ピアノロールを「結果ビュー」として** | F-040は **編集**（Should）。デフォルト結果ではない | 不足 |
| **光る鍵盤 / 鍵盤ハイライト** | **不在**。F-101は運指推定Could/P3で別物 | **致命欠落** |
| **耳コピ再生（結果を聴いて分かる）** | F-054 Should、F-042 Must（だが検証UI＝編集者向け） | 半分あるが主画面でない |
| **コードカードで合わせ弾き** | F-087 Should | あり（ただしギタリスト/学習寄り） |
| **共有カード** | F-109 Should | あるが、その先の体験が無い |
| **五線譜** | F-031 Must | **ターゲットが読めない形式がデフォルト** |

**「デフォルト五線＋全形式オプション」確定は、ターゲットが非音楽家であるという宣言と正面衝突している。**  
オプションの奥に正しいUIがあるなら、非音楽家は辿り着かない。  
オプションを探す人はすでに音楽家だ。

### 2.3 X上の非音楽家・楽譜非読者の言説との照合

Xで見えるのは、楽譜を読めない／使わない人の行動は **譜面編集ではない** という点で一貫している。

- 楽譜が読めないままプロ活動している、耳と記憶が主、という声。  
  「譜面が読めなくても演奏は成立する」側の文化。  
- 楽譜が読めないから **階名で歌う／自分なりの謎言語で起こす／動画を何度も再生** する実務。  
  欲しいのは五線のエングレービングではなく **「音が何か分かる外部記憶」**。  
- 「絶対音感の人に譜面にしてほしい」「音の合致ができるアプリが欲しい」——  
  需要は **判定と可視化** であり、Guitar Pro互換やMusicXML TABプロファイルではない。  

対照的に、高エンゲージの「耳コピ」言説の多くは **すでに音楽制作を始めた層**（ピアノ音色で打ち込み、コード進行）であり、ビジョンが言う「街の音が楽譜になる非音楽家」とは **別セグメント** である。  
カタログは後者の言語で前者の機能リストを埋めており、**セグメント取り違え**が起きている。

### 2.4 UI/UX方針書が非音楽家を置いてきぼりにしている

`uiux-direction.md` の確定方針は:

- A層60%: **静かな工房・長時間編集・キーボード第一**  
- C層30%: 進化する下書き（編集者向けAI正直さ）  
- B層10%: 弾ける歓び（調味）

これは **楽譜を直す人のUI** だ。  
絶対音感エミュレータの第一画面は本来:

> 録音 → 拾えた音程が **音名・鍵盤・再生** で返る → 共有  
> （五線は「譜面が読める人向け詳細」）

であるべきなのに、方針書の成功指標は:

- 初回30秒で**譜面**が見える  
- 1000**音符**表示で操作落ちしない  
- 初見10分以内に最初の**修正**を完了  

「修正」を成功と呼ぶ時点で、ターゲットは編集者にすり替わっている。  
**UI/UX方針はビジョン転換を未消化のまま、第2巡以前の「気持ちいい楽譜エディタ」を確定してしまっている。** これが第3巡の新しい構造傷。

## 3. 【二兎のリスク】1カタログに同居する2製品

### 3.1 実際に同居している2つの製品

| | 製品α: 絶対音感エミュレータ | 製品β: AI採譜工房 |
|---|---|---|
| ユーザー | 非音楽家・日常録音 | 耳コピ勢・ギタリスト・DTM |
| 入力 | 雑音混じりスマホ録音 | クリーンな曲ファイル・ステム |
| 成功 | 「同じに聞こえる／音名が分かる／シェアした」 | 「手直し少なく弾ける／書き出せる」 |
| UI | ワンショット魔法・聴覚中心 | 静かな工房・キーボード第一 |
| コアKPI | 客観指標＋非音楽家聴感（vision） | NF-018手直し比率・NF-032 TAB KPI |
| 代表要件 | F-108, F-109, （欠落している音名・鍵盤） | F-039, F-076, F-078, F-048, F-115 |

カタログは **βを本体、αをShouldの飾り** にしている。  
ビジョン文書は **αを本体、βを出力オプション** と言っている。  
**同じリポジトリ内で製品定義が分裂**している。

### 3.2 二兎が生む具体的リスク（新規）

1. **G0'/G1の解釈が汚染される**  
   ビジョンは「耳起因か記譜起因か」を切り分けよと言う。  
   だがPhase1候補プールとMustは記譜・TAB・MusicXMLで埋まっており、**スパイクがβの検証になる確率が極めて高い**。αは検証されないまま「構想にある」状態で死ぬ。

2. **判定哲学の自己矛盾**  
   vision は「音楽家の手直し苦痛に引っ張られるな」と宣言した。  
   しかし **NF-018 手直し比率KPI は依然 Must**、**NF-032 TAB品質KPI も Must**。  
   哲学は更新、KPIは旧世界。**測るものが作るものを決める**ので、実装は必ずβに引きずられる。

3. **デザインシステムの敵味方逆転**  
   計器UI・波形LOD・FL級ピアノロール・コマンドパレットはβを最高にする投資。  
   αに必要なのは「3タップで音名と再生」「子どもでも分かる結果画面」。  
   **同じカラーシステム3系統（NF-047）で両方を最適化できない。** 片方の最適解がもう片方のクラッターになる。

4. **「全形式オプション」（F-104 / NF-045）がフォーカス破壊装置になる**  
   網羅はエンジニアの安心剤だが、非音楽家向け製品では **選択肢＝迷子**。  
   「全部出せる」はβの差別化、「一発で分かる」はαの差別化。両立をMust/Shouldで同時に掲げると、**どちらの第一画面も中途半端**になる。

5. **組織的リスク（1人開発想定でも）**  
   αのPMFに必要なのはフィールド録音デモ3本（F-108受入）と共有ループ。  
   βのPMFに必要なのはTAB弦割当とMusicXML方言。  
   **週次の実装選択で毎回βが勝つ**（要件数・Must数・既存調査量がβ側に偏っているため）。αは永遠に「来四半期」。

### 3.3 二兎の評決

二兎は「将来オプションとして書く」こと自体が悪ではない。  
悪なのは **どちらが北か不明なまま、両方を同一Must重力圏に置いていること**。  
現状のカタログは **βの重力井戸にαが落ちて蒸発する構造** になっている。

**33/100 — 「絶対音感エミュレータを作るための文書」としては不合格。**  
「音楽家向けAI採譜のフル構想カタログ＋昨夜のビジョン覚書」としては、調査品質だけ見ればまだ70点台の残骸がある。

（参考: 第2巡は「調査カタログとしては優秀、実行要件としては未成立」。第3巡はさらに厳しい。**ビジョンが明確になった分、カタログのズレが隠蔽不能になった。**）

### 4.2 最大の欠陥 3つ（新傷のみ）

#### 欠陥1 — デフォルト出力がターゲットに読めない（製品定義の自己矛盾）

- **F-031 五線 Must / Phase1** と vision の「非音楽家ターゲット」「聴感で判定（楽譜を読む必要はない）」が両立しない。  
- 非音楽家の一次消費に必要な **絶対音名表示・光る鍵盤・結果としてのピアノロール** が要件として存在しない／末端Could。  
- **「変換できた」ことと「使えた」ことを混同**している。

#### 欠陥2 — ビジョン本体が Should、周辺の楽譜産業が Must（優先度の倒錯）

- **F-108 / F-109 が Should** のまま、TAB・整譜・MusicXML・楽譜エディタが Must 席を独占。  
- NF-050 で二層を謳っても、**Must一覧が「記譜層の百科」** になっている時点でアーキテクチャ原則は紙だけ。  
- 実装リソースは必ずMustに吸われる。**αは設計図上は存在するが、出荷経路上は存在しない。**

#### 欠陥3 — 判定哲学を書いたのに、測るKPIが音楽家のまま

- vision: 主軸は客観指標＋非音楽家聴感。音楽家評価は演奏者向け出力のQAに限定。  
- 現実: **NF-018 手直し比率 Must**、**NF-032 TAB KPI Must**、uiux成功指標は編集完了。  
- **非音楽家聴感の受入プロトコルが要件IDとして独立していない**（哲学メモ止まり）。  
- 測らないものは改善されない。**言葉では音楽家から離れたが、計器は音楽家の工房に残っている。**

## 5. 第3巡で「今すぐ刺すべき」是正方向（提案のみ・実装指示ではない）

批判に止めず、次巡が噛める粒度だけ置く。

1. **製品を一文で再固定**  
   「非音楽家が日常音から音程をつまみ出し、音名・再生・共有で理解する」か、「音楽家が下書き譜を仕上げる」か。**両方を同じMVPに載せない。**

2. **αを選ぶなら、デフォルト結果ビューを要件化**  
   例: `F-xxx 音名タイムライン` / `F-xxx 鍵盤ハイライト再生` を **Must**、五線はオプション。  
   F-031 の Must/デフォルトを撤回または「βプロファイル限定」。

3. **F-108 を Must / Phase1 候補の中心に格上げ**し、F-076系TAB Mustを「β出力プロファイル」へ降格。  
   NF-050 と優先度表を一致させる。

4. **NF-018 を「β出力QA専用」にスコープ限定**し、α用に `NF-xxx 非音楽家聴感一致率` を主KPIとして新設。  
   哲学とKPIの二重帳簿をやめる。

5. **uiux-direction をビジョン後に再承認**  
   「静かな工房60%」はβ用。α用は別方向（結果ファースト・編集最小化）を書かない限り、UI投資が敵になる。
