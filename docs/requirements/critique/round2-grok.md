**対象:** 機能102件 / 非機能44件（Must 33+16）  
**レビュアー立場:** 1人開発で出荷可能なプロダクト定義を見たい投資家兼PM

## 0. 総評（先に結論）

v1→v2→v2.1の2巡は「欠落を埋める」には成功した。  
**失敗したのは「捨てる」ことだ。**

- 出典規律・受入条件化・Won'tの増加・TABエンジン層の追加は、ドキュメント品質としては高い。
- しかし件数は **75→102**。第2巡は「飽和点」と言いながら **機能+5 / 非機能+3** を足した。  
  飽和点を宣言した直後に食べ続けるのは、ダイエット中の深夜ラーメンと同型である。
- 文書は冒頭で「フル構想版」「事業推進の判断を意味しない」と逃げ道を確保している。  
  その逃げ道がある限り、**Mustは“実装義務”ではなく“調査員の安心バッジ”**になりやすい。

**一意の物語は崩壊している。**  
「ローカル完結の弾けるギターTAB下書き」なのか、「ピアノ整譜の勝負所」なのか、「中華简谱入口」なのか、「SynthV用MIDI」なのか——**全部がShould/Mustの席に座っている。**

## 1. 【肥大化検査】102+44は焦点を失っていないか

### 1.1 数字の異常さ（1人開発者視点）

| 指標 | 数値 | 辛口コメント |
|---|---|---|
| 機能 | 102 | カタログとしては厚い。ロードマップとしては消化不能 |
| 非機能 | 44 | 品質/法務/モデル運用が「会社3部署分」 |
| Must合計 | **49**（33+16） | 1人が並行で抱えられるMustは通常 **8〜15** |
| Phase1 | **23** | スパイクではなく **半MVP** |
| Phase2 | 45 | Mustがまだ大量に残る |
| Could+観測 | 多 | 「捨てたフリの保留」が厚い |

**判定:** フル構想としても焦点は薄い。  
「何を作らないか」がWon't 6件にしか表れていない。Could 25 + 観測5 は、**判断保留の倉庫**である。

### 1.2 Must 33+16は本当に全部Mustか（X実需と突合）

X上で繰り返し見える需要は、だいたい次の3つに収束する。

1. **耳コピ/採譜AIが欲しい**（生成AIより耳コピAI）
2. **ギター/ベースのTABが精度不足で萎える**（音抜け・運指ゴミ）
3. **リズム/拍が崩れると使い物にならない**

高エンゲージ寄りの信号も、「フルスコア全部盛り」ではなく **コード進行プレイアロング** や **ステム→MIDI/TAB** のような**狭いジョブ**側にある。

## 2. 【プロダクトの物語】誰のための何のプロダクトか

### 2.1 今の文書から読める物語（複数）

同時に同居している物語:

| 物語 | 代表ID | 状態 |
|---|---|---|
| A. ギター/ベース「弾けるTAB」下書き | F-032, F-076, F-078, NF-032 | v2でMust昇格 |
| B. ピアノ整譜・多声・ペダル | F-023, F-024, F-102, NF-019 pedal | 旧U1/U2主戦場が残存 |
| C. バンドフルスコア | F-019, F-015 | Couldだが受入条件が肥大 |
| D. ボーカル/歌詞/歌声合成 | F-020, F-098 | Phase3とShouldが混在 |
| E. 中華简谱・移動ド教育 | F-033, F-100, F-069 | 市場拡張オプション |
| F. ローカル/権利/非関与 | F-066, NF-023〜028 | 制約は明確（良い） |

**一意に読めない。**  
「TAB必須化でv5のU3を上書き」と書きつつ、「ピアノ主戦場は維持・五線とTAB二本柱」も維持している。  
これは戦略ではなく **和集合** だ。

### 2.2 中途半端化の兆候（既に要件文面に出ている）

1. **Phase1に五線多声(F-023/024/031)とTABエンジン(F-076/028/032)が同居**  
   → スパイクが「どちらの仮説を殺すか」を決められない。
2. **F-003ステム分離Mustなのに、ギター/ピアノ6-sourceは実験的品質と自己注記**  
   → 中核工程が「まだ無理」と宣言しているのにMust。
3. **F-078をギター+ベース固有奏法まで拡張**  
   → 「TAB」から「全弦楽器奏法百科」へ。
4. **F-098(SynthV)をPhase2 Should**  
   → ギターTAB製品のMVPに歌声合成納品を混ぜた瞬間、サポート対象が爆発する。
5. **F-033简谱はShouldのままPhase3**  
   → 落とさないからカタログが太る。採用しないならWon't/観測へ。

### 2.3 X実需が指す「狭い楔」

日本語圏の生声はかなり率直:

- 「**耳コピ生成AIが欲しい**」（生成AI不要論）
- 「ベース抽出→TAB自動生成、**精度が悪すぎて萎えた**」
- 「AIにTAB作らせたら**欲しい音がごっそり抜けた**」
- 「**ベース耳TAB**が欲しい」アプリ自作投稿
- Songsterrは「ある」が **riffが違う/ass** という精度不満も併存

つまり市場が求めているのは:

> **混合音源（またはベース/ギター寄せ）から、修正可能な“弾けるTAB下書き”をローカルで出すこと**

ではない:

> 合唱SATB + 简谱 + 移動ド + 鍵盤運指 + SynthV + LLMバンドル + ペダルCC64

**全部中途半端になる兆候: 強い。**  
特に「ピアノ整譜の研究課題」と「ギターTABのプロダクト課題」を同一Must集合に閉じ込めた点が最悪。

## 3. 【Phase1の現実性】23件はスパイクか

### 3.1 結論: **スパイクではない。検証を装った半製品**

スパイクの定義は「仮説を最短で殺す最小構成」。  
23件は **実装ロードマップの第1四半期** に見える。

Phase1 Must群を要約すると:

- 音声I/O・ローカル
- ステム分離ゲート
- ピッチ/オンセット/拍/テンポマップ
- 量子化・拍子・**声部分離・連桁・演奏可能性**
- TAB割当・運指・チューニング・五線・TAB
- 下書きUX
- MusicXML/MIDI/妥当性検証

これは「半日〜数週のスパイク」ではなく、**エンジンチームの最初の四半期**。

### 3.2 Phase1から外すべき（スパイク汚染）

| ID | なぜ不要か |
|---|---|
| **F-023** | ピアノ多声仮説。TAB仮説と同時検証するな |
| **F-024** | エングレービング。手直し比率の主因が連桁かは未検証 |
| **F-031** | TAB成果物なら表示はTAB優先。五線は後 |
| **F-052（フル）** | 多アプリ往復はP2品質ゲート |
| **F-081** | 重要だがTAB弦位置より後段。P1 Shouldでよい |
| **F-083** | 有用だがP1 Must化は過剰（現行Shouldでも重い） |
| **F-003（フル内蔵）** | スパイクは **既分離ステム入力(F-088)で代替** 可能。分離品質とTAB品質を同時に測るな |
| **F-025のピアノ側** | ギター左手制約に限定せよ |

### 3.3 Phase1に欠けている（X実需ベース）

| 欠け | 推奨 | 根拠 |
|---|---|---|
| 信頼度ハイライト **F-043** | P1へ前倒し | 「AI slop」「音抜け」への即時応答。下書き(F-038)の歯 |
| 原音同期 **F-042**（最小） | P1へ | 耳で検証できない下書きは信用されない |
| 最小修正UI **F-039** のサブセット | P1 | 手直し比率(NF-018)を測るなら編集がないと測れない |
| 既分離入力 **F-088** | P1 Should→Must級 | ステム問題と採譜問題の変数分離 |
| 「音抜け」検出の受入条件 | 新設 or F-038拡張 | Xで明示された失敗モード（ごっそり抜け）が要件に薄い |

**あるべきPhase1（例: 10〜12件）**

1. F-001 入力  
2. F-088 既分離（またはF-003を簡易）  
3. F-010/011/012 解析基礎  
4. F-017 or 固定テンポ+F-021/022  
5. F-076 + F-028 + F-077  
6. F-032 TAB  
7. F-038 + F-043  
8. F-042 最小同期  
9. F-048（TABプロファイル最小）  
10. F-066 ローカル  
11. NF-018 手直し計測  
12. 最小編集（F-039 subset）

これ以外は **全部P2以降**。今の23件は検証ではなく保険の積み上がり。

## 4. 【調査反映の副作用】証拠と優先度の不釣り合い

第2巡の病名は明確:

> **「証拠があるから追加」≠「今やるべき」**

調査は網羅性に報酬を出し、優先順位には罰を出さない。  
そのインセンティブでドキュメントは必ず太る。

### 4.1 弱い証拠なのに重い優先度

| ID | 現行 | 証拠の質 | 辛口判定 |
|---|---|---|---|
| **F-098** 歌声合成エクスポート | Should / P2 | ニッチは実在するが別ジョブ | **Could / P3**へ。TAB本線を汚染するな |
| **F-102** ペダル検出 | Should / P2 | MIREX/論文強い・Xギター実需弱い | **Could**。ピアノ研究所機能 |

| **NF-043** ランタイム別ゲート | **Must** | 実装工学の一次資料 | 優先度釣り合い崩壊の代表。**Should** |
| **NF-044** MusicXML歌詞/コード profile | Should | 規格ギャップ | リードシートを捨てるなら不要 |

### 4.2 強い証拠なのに相対的に弱い扱い

| 信号 | 現状 | 問題 |
|---|---|---|
| TAB精度・音抜け・運指 | TAB群はMust化済み（良い） | だがピアノ/歌声のShouldが同じP2テーブルに並び、**焦点が薄まる** |

| ローカル・非アップロード | Must（良い） | その軸で切れる機能がまだ多い（F-098等） |

### 4.3 プロセス批判

- Webスキャンが「飽和点」と言ったのに採用8件。  
  **飽和点宣言と追加行動が矛盾**している。
- 「不採用0」は健全ではない。  
  良い調査ラウンドは **不採用の山** を作る。
- 観測リスト5件は評価できる。  
  だが **Shouldに上げたものこそ観測へ戻すべき** だった。

## 6. 最後の一言（辛口）

v2.1は「賢い要件が増えた文書」であって、「勝てるプロダクトがシャープになった文書」ではない。

Xが言っていることは単純だ。

- 耳コピAIが欲しい  
- TABは精度がゴミだと使わない  
- リズムが死んだら終わり  
- 生成AIより、直せる下書きが欲しい  

なのに要件セットは、論文・規格・中華・歌声合成・鍵盤運指・LLM可読まで**記念撮影**している。

**次の改訂でやるべき唯一の仕事は追加ではない。**  
**Mustを半分にして、ピアノとギターのどちらの戦争に行くか宣戦布告することだ。**

両方やるなら、1人開発者ではなく、最初からチーム編成を要件に書け。
