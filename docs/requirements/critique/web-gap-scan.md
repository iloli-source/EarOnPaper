## 1. 欠落要件リスト

既存106件のどのIDにも該当しない、証拠のあるニーズ。

### G-01: 異名同音（エンハーモニック）綴りの正規化
- **根拠:** Songscription公式ブログおよびMusicRadarレビューが「enharmonic chaos（D♯ where it should be E♭）」を主要な手直しコストとして繰り返し指摘。musicnotationhub.com の内部テストでは「AI出力の修正が最初から書くより2.25倍長い（45分 vs 20分）」で、その要因として**リズム書き直し・リビーミング・異名同音綴りの修正**を筆頭に挙げている。F-013（キー推定）やF-021（量子化）はあるが、「調に整合した音名綴り（key-aware pitch spelling）」を独立要件として持たない。
- **提案要件名:** 調整合ピッチスペリング（key-aware enharmonic spelling）。推定キー・和声文脈に基づき音名を選び、五線に調号と矛盾しない綴りを出力。臨時記号の散乱を抑制。
- **提案カテゴリ:** リズム・整譜
- **提案MoSCoW:** Must（手直し工数KPI NF-018に直結する最頻不満の一つ。競合共通の弱点）

### G-02: グレースノート・装飾音の意図保持
- **根拠:** musicnotationhub.com「AI treats grace notes as literal 32nd notes」、MusicRadar「flutist breathing marked as longer notes（演奏要素を譜面要素と誤認）」。F-024（連桁・符尾整理）はあるが、装飾音・ブレス・アーティキュレーション由来の微小音価を「装飾記号」として解釈し直す層がない。
- **提案要件名:** 装飾音・演奏ノイズの記譜解釈。32分音符の羅列でなくグレースノート/前打音等の装飾記号候補として提示。ブレス・弦のこすれ等の非楽音を音符化しない。
- **提案カテゴリ:** リズム・整譜
- **提案MoSCoW:** Should

### G-03: MIDIドラフト先行編集ワークフロー（記譜前クリーンアップ）
- **根拠:** RiffToTab（rifftotab.com）の中核設計＝「まずMIDIドラフトを掃除してから譜面/TABにマップ」。ユーザーは pitch/timing/density/tempo/cleanup を1つのMIDIワークスペースで整えてから記譜へ渡す。musicnotationhub.com も「XMLを解きほぐすより音源から作り直す方が速い」と述べており、**記譜化する前の中間MIDI段での修正UI**への需要を示す。既存はF-040（ピアノロール編集）を持つが、これは「記譜後のMIDIレベル編集」であり、RiffTabが提供する「記譜化前にドラフトを確定させる工程」とは順序が異なる。
- **提案要件名:** 記譜前MIDIクリーンアップ工程。量子化・音価・密度・不要音削除を記譜化の前段で確定させ、確定後に整譜する順序制御。
- **提案カテゴリ:** 編集UI
- **提案MoSCoW:** Should（NF-007部品交換・NF-008中間出力検査と整合し、手直し順序の非効率を解消）

### G-04: 楽器制約を尊重したTABマッピング（記譜の一般化でなく）
- **根拠:** RiffToTab「mapping respects tuning, capo, fret range, and guitar constraints instead of treating MIDI like generic notation」。F-032（TAB譜）とF-028（運指最適化）はあるが、両者の中間にある「チューニング・カポ・フレット可動域を制約として与えた上でのポジション割当」が明示要件でない。F-028は"最適化"、F-032は"生成"だが、制約入力（このカポ・このチューニング前提でマップせよ）が抜けている。
- **提案要件名:** 演奏制約付きTABマッピング。ユーザー指定のチューニング・カポ位置・使用フレット範囲を入力制約として運指を割当。
- **提案カテゴリ:** リズム・整譜（F-028/F-029の隣接）
- **提案MoSCoW:** Should（F-029変則チューニングと統合可能だが、入力制約UIとして独立の価値）

### G-05: 手直し工数の実測ベースライン提示（from-scratch比の可視化）
- **根拠:** musicnotationhub.com の「45分 vs 20分（AI修正 vs スクラッチ2.25倍）」は製品の存在意義を揺るがす定量的批判。NF-018は「手直し比率KPI」を内部品質指標として持つが、これは開発側の指標。**ユーザーに対し「この曲はAIドラフト採用が得か、耳コピが得か」を事前提示する機能**は要件化されていない。信頼度ハイライトF-043の延長線上にある「そもそもドラフトを使うべきか」の判断支援。
- **提案要件名:** ドラフト採用可否の事前判断支援。音質診断（F-002）・信頼度分布から「この曲は手直しが多い可能性」を採譜前/直後に警告し、耳コピ学習モード（F-064）へ誘導。
- **提案カテゴリ:** 編集UI（下書きUX F-038の具体化）
- **提案MoSCoW:** Should（「AIが学習を奪う」批判への製品的応答であり、正直さが堀になる）

### G-06: リビーミング／リバーリング（小節線・拍頭の再割当）
- **根拠:** musicnotationhub.com が手直し内訳として「rewriting rhythms, **rebeaming, rebarring**, and fixing time signatures」を列挙。MusicRadar「everything is shifted an eighth note late（全体が8分ずれる）」。F-024は連桁・符尾整理を含むが、**小節線位置そのものの系統的オフセット補正（rebarring）**と、拍頭からのグローバルなずれ補正は明示されていない。F-012拍推定の下流で「拍グリッドは正しいが記譜上の小節割当が半拍ずれる」ケースへの対策層。
- **提案要件名:** 小節・拍オフセット補正。全体的な系統オフセット（例: 8分遅れ）を一括検出・補正し、小節線と拍頭を再整列。
- **提案カテゴリ:** リズム・整譜
- **提案MoSCoW:** Should

## 2. 過小評価リスト

既存要件だが、Web上の需要の強さに対して優先度が低すぎるもの。

### U-01: F-017 テンポマップ生成 — Should → **Must** 昇格を提案
- **現状:** Should / Phase2
- **根拠:** ルバート・表情的タイミングは**複数の独立情報源で最頻の崩壊点**。MusicRadar「any kind of expressive timekeeping で timing goes completely off track」、musicnotationhub.com「rubato is notated as alternating accelerations and decelerations」、Songscription公式も自認。手直し工数の主因であり、NF-018 KPIとNF-022拍子ゲートの前提。競合が全社失敗している最大の差別化ポイントであることを踏まえると、テンポ揺れ追従は「あれば良い」ではなく製品の生死を分ける層。
- **提案:** Must / Phase1スパイクへ前倒し（F-012拍推定と一体で検証すべき勝負所）

### U-02: F-059 テンポ変更再生（ピッチ維持減速） — Should維持だがPhase前倒しを提案
- **現状:** Should / Phase2
- **根拠:** Transcribe!（seventhstring.com）がプロ耳コピの事実上の標準ツールである最大理由が「1/20〜2倍速のピッチ維持スロー再生＋ループ」。プロWF記事（Flypaper/Musical U）も「50%以下に落として聴く」を必須手順とする。AIドラフトが不完全である以上、**ユーザーが自分の耳で検証・補完する再生環境**はコア機能に近い。F-042原音同期再生（Must）とセットで扱うべき。
- **提案:** 優先度はShould維持で妥当だが、F-042とバンドルしてPhase2内で最優先化。単独のCould扱いに沈めない。

### U-03: F-071 無料試用 — Should維持だが「試用の実用性」を要件に明記すべき
- **現状:** Should / Phase2（「プレビュー制限方式」）
- **根拠:** Klangioの20秒デモは**Trustpilot/比較記事で繰り返し「短すぎて何も完成できない」と批判**。Songscriptionは「30秒無制限＋長尺トライアル」で明確に差別化し比較記事で優位に立つ。「無料枠がある」だけでなく「無料枠が意思決定に足る長さ・回数か」が競争軸。F-071の説明が「20秒プレビュー」を出典に引くのは、むしろ競合の弱点を踏襲する危険がある。
- **提案:** 優先度は据え置きだが要件説明を「意思決定に足る試用（回数無制限の短尺 or 長尺トライアル）」へ改訂。20秒モデルを踏襲しない設計制約を明記。

## 3. 競合実装済みで欠落

競合が既に実装しているのに75件のどのIDにも見当たらない機能（AnthemScore v6.0.0/v5.x 変更履歴 lunaverus.com、Moises、RiffToTab等）。

### C-01: 譜面の簡略化スライダー（AnthemScore v5.2「Simplify sheet music by any amount」）
- 既存F-026「難易度別アレンジ」（Could/Phase3）は"初級〜上級の別譜面生成"だが、AnthemScoreの Simplify は**同一譜面の音符密度を連続的に間引く軽量機能**で位置づけが異なる。より手軽な「読みやすさのための間引き」は独立要件化の余地。**提案:** F-026の下位に「密度簡略化スライダー」（Could）。

### C-02: 精度/速度トレードオフスライダー（AnthemScore v6.0「Accuracy/speed slider 1-10」）
- 既存はF-067（処理速度）とNF-004（処理時間）を持つが、**ユーザーが精度と速度を明示的にトレードオフするUI**は要件化されていない。**提案:** F-067に「精度/速度の手動調整」を追記（Should）。

### C-03: ドラッグ&ドロップ入力（AnthemScore v6.0）
- F-001音声ファイル入力に含意されるがUX要件として明示なし。軽微だが競合が明示的に追加した点。**提案:** F-001の受け入れ条件へ追記（Must内）。

### C-04: 別プロジェクトからの拍・音符インポート（AnthemScore v5.1/v5.2「Import beats/notes from another project」）
- 手直し済みの拍グリッドや音符を別ファイルへ再利用するワークフロー。版管理F-046やバッチF-008とは別。**提案:** 中間資産の再利用要件（Could）。

### C-05: サウンドフォント差し替え・楽器シンセ（AnthemScore v5.0「Synthesize instruments using changeable sound fonts」）
- F-054音声プレビューはあるが、**任意サウンドフォントでの試聴**は未記載。検証時の聴き分け精度に寄与。**提案:** F-054に追記（Could）。

### C-06: 700+コードタイプ・テンション対応のコード検出（AudioJam/Chord ai）
- F-014コード進行解析はあるが、**テンション・分数コード・複雑なジャズ和声の粒度**は明示されていない。Chord ai/Moisesの主要不満が「ファンク/ジャズで根音を取り違える・到達不能なコードを出す」。**提案:** F-014に「テンション・オンコード・和声粒度の制御」を追記（Should）。競合の主要失敗領域＝差別化余地。

### C-07: マイク録音入力（AnthemScore v5.3「Record audio from the microphone」）
- 既存F-005（マイク/ライン録音）はCould/Phase3。AnthemScoreは2025年に標準搭載済み。過小評価の疑い。**提案:** F-005をShouldへ昇格検討（競合が標準化した機能）。

## 付記: 既存要件の網羅性評価

75+31件は**Web上の主要不満をほぼ捕捉できている**。特にルバート崩壊・無意味な拍子・非現実的運指・想定外請求・英語UIといった「最頻の不満」は明確に要件化済み。本スキャンで見つかった真の欠落は、(a) 異名同音綴り G-01（最重要）、(b) 装飾音の記譜解釈 G-02、(c) 記譜前MIDIドラフト編集の順序制御 G-03 の3点が中心で、いずれも「手直し工数を最初から作るより長くする」根本原因（musicnotationhub.com の2.25倍問題）に直結する。優先度の観点ではテンポマップ F-017 のMust昇格が最も影響が大きい。
