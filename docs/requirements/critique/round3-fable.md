## 1. 前提カスケード監査（最重要）— 転換で無効化されたのに残る記述

### 転換③（音楽家依存の評価枠組み撤廃）の残骸 — これが最も重い

- **NF-018「手直し比率KPI」がいまだMust・主要品質KPI扱い。** non-functional §85: 「フル耳コピ比の手直し工数比率を**主要品質KPI**とする」。転換③は「音楽家の判定基準（手直しの苦痛）に依存する枠組みは間違い」と明言し（product-vision §49）、gate-spec §72では手直し比率を**参考データに降格**した。ところがNF-018は要件文書上「主要品質KPI」の肩書きのまま。**gate-specは降格・requirementsは主KPIという正面衝突**。NF-018はMust降格＋「演奏者向け出力QA専用指標」への役割変更が必須。
- **NF-032「TAB品質KPI」に「TAB手直し時間比率」がMust指標として残る。** non-functional §90。同じく手直し時間ベースで、非音楽家二段判定と非整合。TAB系はそもそも「演奏者向け出力オプションのQA専用」（gate-spec §79）に降格した領域であり、Must（全ゲート判定を縛る主軸）に置き続けるのは転換③・②の両方と矛盾。
- **gate-spec §3.6判定表の脚注が本体を上書きしきれていない。** §98で「本表の『手直し比率』は主指標に読み替える」と注記するが、表本体（§100-104）の帯条件は「手直し比率 ≤ 校正済み下限」等の旧語彙のまま。読み替え注記で運用するのは封緘直前文書として危うい。表本体をハーネススコア＋聴感語彙に書き換えるべき。

### 転換①（絶対音感エミュレータ＝非音楽家ターゲット）の残骸

- **「クリーン音源前提」の名残。** F-003ステム分離は「採譜前の標準工程として内蔵」（func §129）で、雑音混じり日常録音を第一級入力とするビジョン（product-vision §14）と接続が弱い。F-108（フィールド録音）とF-003の関係（日常録音でステム分離は前提化できるのか、pitchedイベント抽出が先か）が未定義。**耳コア（楽器非依存）の入口処理としてF-003を強制すると二層原則NF-050と衝突しうる。**
- **G0'/G1の選曲がビジョン非整合。** gate-spec §3.2・g0-prime-kit §2の固定3曲は「U1ピアノカバー・U2弾き語り・U4バンド」＝**全てクリーンな音楽ソース**。絶対音感エミュレータの中核実証（日常音＝鳥・駅ジングル・ノック）が初手検証にゼロ。product-vision §29自身が「G1頑健性軸にフィールド録音区分を追加する価値あり」と言い、NF-021も「ベンチにフィールド録音区分を含める」とするのに、実行仕様（gate-spec/g0-kit）の選曲に降りていない。**ビジョンの一次検証機会がG0'にない。**

### 転換②（TAB特化撤回・五線デフォルト＋全形式オプション）の残骸

- **gate-spec §3.2にU3ギターTABが「スコープ外判定の材料」として残存。** §63-64・§3.1表の脚注。これはv5のTABスコープ外判定を前提にした旧枠組み。func v2は「TAB必須化＝Must」に昇格させ（F-032）、さらに転換②で「TAB特化は要求していない・全形式オプション」に再々転している。gate-specだけ**v5世代の「TABはスコープ外」語彙**が化石化しており、requirements（F-032=Must）ともvision（全形式オプション）とも三者三様。
- **func冒頭注記の地層が読者を惑わす。** func §7・§37に「TAB必須化はv5のU3スコープ外判定を上書き」（v2）と「TAB特化は選択肢から除外」（v2.2訂正）が併存。両立はするが（必須≠特化）、初読者には矛盾に見える。転換②確定後の要約を先頭に一本化すべき。

### 転換④（固定は証拠が来るまで最低段）の適用漏れ

- 転換④はW-06（RX型領域選択）を観測リスト最低段に置く根拠として明示されたが（func §297）、**同じ原則を他のMust群に遡及適用した形跡がない。** 下記4節参照。原則導入と既存行の再監査が非対称（＝traceability-review §72が指摘した「追加は即日・削減はv3送り」の続行バイアス症状と同型）。

## 3. Phase1候補プールとビジョンの整合

Phase1候補プール24件（func §12）を絶対音感エミュレータの最小検証として見ると、**中核と周辺の優先が転倒している。**

- **F-108（フィールド録音モード＝ビジョン直結機能）がPhase2。** func §136。product-vision §57が「F-108＝絶対音感エミュレータの中核実証」と位置づけるのに、最小検証プール（Phase1）に入っていない。二層アーキの「耳」を検証するならF-108こそPhase1候補筆頭のはず。**ビジョン中核がPhase2にいる矛盾は存在する。**
- 一方でF-032（TAB）・F-076（弦フレット割当）・F-028（運指最適化）・F-081（異名同音）などTAB/整譜寄りがPhase1に4件超。転換②で「TABはオプションの一つ」に降格した以上、最小検証にTAB中核エンジンを4件積むのは配分として重い。これらは「演奏者向け出力QA」領域（gate-spec §79で後回し確定）と整合しない。
- Phase1に「耳コア（楽器非依存の音程イベント抽出＋信頼度）」を直接検証する行が薄い。F-010（多重音高）・F-011（オンセット）・F-017（テンポマップ）はあるが、**日常音からのpitched成分選択抽出**という新規性の核はF-108頼みで、それがPhase2。

**判定:** Phase1候補プールは「旧TAB必須化時代の重心」を引きずっており、転換①②後の「耳を最小検証する」構成に組み替えられていない。F-108のPhase1昇格（少なくとも候補プール入り）と、TAB系Phase1件数の削減が必要。

## 4. 最低段原則（v2.6.1）の一貫性 — Mustサンプル検査

「固定は証拠が来るまで最低段」に照らし、証拠が弱いのにMust（最上段）で固定されている行を数件サンプル。

- **NF-045「出力層プラグイン型アーキ」＝Must。** 根拠は「ユーザー決定（2026-07-19）」のみ（non-functional §29）。G0'実測もスパイクも未実施の段階で、アーキ層をMust固定するのは最低段原則に反する。実装土台の妥当性は実測後にしか分からない。**Should妥当、あるいは判定保留付与が筋。**
- **NF-050「二層アーキ原則」＝Must。** vision由来の設計仮説（non-functional §30）。G0'は「崩れが耳起因か記譜起因か」でこの二層仮説を**検証する**段階（g0-kit §8）。検証前の仮説をMustで固定するのは、証拠が来る前の固定＝原則違反。**「検証対象の仮説」であってMust要件ではない。**
- **NF-006「対応楽曲長2時間保証/最大24時間」はCould**（non-functional §26）だが、これを受けるNF-046（波形3段描画・24時間音源で破綻しない）が**Must**（§31）。**Could前提の上にMust実装規律**という強さ逆転。24時間対応という未検証の上限にMust級の描画予算を紐づけるのは過剰固定。
- **NF-022「拍子妥当性ゲート」＝Must・判定保留。** これは妥当（判定保留付きMustは「固定していない」ので原則整合）。**適合例として記録。**

**所見:** ユーザー決定を根拠にした「2026-07-19 Must」群（NF-045/050、F-103/104関連）は、決定＝要望であって実測証拠ではない。最低段原則を遡及適用するなら、この層こそ真っ先にShould/保留へ落とす対象。traceability-review §72の「Must約54件・削減はv3送り」という続行バイアス警告と完全に符合する。
