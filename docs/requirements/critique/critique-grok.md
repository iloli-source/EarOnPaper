## 総評（1段落で殴る）

フル構想75件は「調査を網羅した機能カタログ」としては悪くない。だが **製品として誰のどの痛みを最初に潰すか** が決まっていない。v5は譜面系（U1/U2）主戦場・U3 TABはベンチ1曲のスコープ外、と明言しているのに、一覧は **ピアノ整譜の勝負所（F-021等）を Must/Phase1 に置き、ギター層が死活で欲しがる運指・チューニング・奏法・.gp を Could/Phase3 に放置** している。TAB必須化はこの歪みを表面化させる。今の優先度のままでは「五線の下書きAI＋TABは後付け表示」になり、**Songsterr AIの劣化コピーかつオフライン版** にしかならない。

## (3) TAB譜必須化の妥当性 と 欠けているTAB要件

### 妥当性：条件付きで正しい。だが「必須にした瞬間に競合定義が変わる」

**賛成材料**
- YouTube: ギター層の要求は明確。日本語のTAB AIデモは希少＝空白認識は調査と一致。
- X: Songsterr需要は実在。一方で「AI生成TAB全部間違い」「精度甘いので耳コピ併用」「特定リフが違う」と **品質不信が濃い**。ここが参入余地。
- X: 自作曲を読ませ「大体合っててビビる」系の驚き体験は、**クリエイター自分の音源→TAB** ユースケースを示す（URL違法コピー需要だけではない）。

**反対・リスク材料（直視せよ）**
- v5は Songsterr を用途別支配者と明記。クラウドURL・カタログ・ソーシャル修正という **ネットワーク効果** がある。
- 当プロダクトは [F-006](F-006) Won't + [NF-023](NF-023) 成果物非関与で、**「好きな曲をURLで」ができない**。必須化しても **「手元音源のTAB化」** に戦場を限定する必要がある。
- フル構想は相変わらず五線・拍子・MusicXML中心。TAB必須は **方針メモの変更** に過ぎず、要件構造が追従していない。

**結論:**  
「TABをMustにする」はユーザー観測と整合。ただし **Mustの中身を「TAB表示」ではなく「弾けるギター譜の生成」と定義しないと失敗する。** Songsterrにカタログで勝つ話ではない。勝つなら:

1. ローカル・権利クリーン（[F-066](F-066)/[NF-013](NF-013)/[NF-023](NF-023)）  
2. 運指・チューニング・奏法で「slopに見えない」  
3. 日本語UI（[F-068](F-068)）と手直しの速さ  
4. .gp 出口  

この4点を Must セットに再編成せよ。

### TAB対応に必要なのに欠けている／弱い要件（チェックリスト）

**エンジン層（新規IDが必要）**
- [ ] pitch→string/fret 割当（左手移動・ポジション帯制約）
- [ ] チューニング推定 + プリセット（Standard / Drop D / Drop C / 半音下げ / Open …）
- [ ] カポ
- [ ] 弦数・楽器プロファイル（Gt6/Gt7/Ba4/Ba5）
- [ ] 連続ピッチからの奏法推定（slide/bend）と discrete fret への写像
- [ ] ミュート・ノイズ・ゴーストノートの扱い
- [ ] 同一音の異弦異フレット候補のスコアリング

**記譜・UX層**
- [ ] 五線+TAB連動（[F-031](F-031)+[F-032](F-032) の統合要件）
- [ ] TAB上でのフレット直接編集（[F-039](F-039) は音符中心で弦指定が不明）
- [ ] 奏法記号の自動付与 + 編集（[F-045](F-045) 拡張）
- [ ] 運指候補の切替UI（複数解）
- [ ] チューニングを譜面ヘッダに必ず印字（X: チューニング書き忘れで「弾けない」事故）

**出力層**
- [ ] .gp（[F-051](F-051) 昇格）
- [ ] MusicXMLのTAB方向・奏法の妥当性（[F-052](F-052)/[NF-011](NF-011) にTAB検証を追加）
- [ ] テキストTAB（練習共有用）は任意だが需要はある

**品質・評価層（NF）**
- [ ] 「人間がそのポジションで弾けるか」自動指標
- [ ] ベンチに **変則チューニング曲・PMリフ・低音ベース** を必須曲として入れる（v5のU3を1曲サンプルで済ませない）
- [ ] 手直し時間の定義に「フレット修正・奏法付与」を含める

**現状IDの再解釈（最低限のパッチ）**
- [F-025](F-025) 演奏可能性制約 → **ギター左手制約を明示**（同時に押さえられないフレット幅など）
- [F-009](F-009) 解析ヒント → **チューニング・カポ・弦数・ピック/フィンガー** を追加
- [F-015](F-015) 楽器分類 → ギター/ベース識別をTABパイプラインの前置 Must に
- [F-032](F-032) → 生成定義を「音高のタブ化」ではなく「演奏可能タブ」に書き換え

## (5) 横断ダメ出し：文書としての欠陥

1. **v5と要件の二枚看板が再発**  
   一覧は「フル構想でゲートと独立」と逃げつつ、v5制約を継承すると書く。TAB必須化は **v5のU3スコープ外判断の撤回** なのに、一覧側の優先度が旧前提のまま。

2. **「出典がある＝作るべき」の混同**  
   調査で言及された機能をフラットに75件化した結果、**ユーザーの支払理由の順位** が消えた。中華圏简譜・sargam・インド内部表現・DAWプラグインが、運指やDrop Dと同列に並ぶのはプロダクトではなく百科事典。

3. **F-006 Won't と競合比較の不誠実**  
   URL入力を捨てるなら、比較表から「Klangio/Songsterrと同じ土俵」を外せ。要件上も **代替価値のMustセット** を書かないと、レビューのたびに「URL無いじゃん」で終わる。

4. **非機能が研究室向け**  
   [NF-019](NF-019) は立派。[NF] に **「初見で30秒フレーズが弾ける確率」** が無いのは、ギタリスト製品として知性の見せ方を間違えている。

5. **練習支援（F-059〜061）は実はTABと相性が良い**  
   減速・ループ・パートミュートは耳コピ層の本音。ここを Phase2 Should に置くのは悪くないが、**「採譜結果の検証装置」としてP1に引き上げた方がTAB必須と噛み合う**。Moises化ではなく「TABの正しさを耳で殺すUI」と再定義せよ。

## (7) 一句で締め

**75件は「調査に出てきた機能の在庫管理」であって、TAB必須化後の製品要件になっていない。**  
ユーザーが金を払うのは「五線が正しいこと」より先に、**変なフレットを押さされず、チューニングが合い、PMとbendが書いてあり、Guitar Proで開いて30秒で音になること** だ。  
Xが既に言っている: SongsterrのAIは便利だが **slopで精度が甘い**。そこに「表示としてのTAB」を後付けしても勝たない。**弾けるTABをMustの定義に書き換えろ。さもないと、Couldに沈めた運指とチューニングが、そのままプロダクトの墓標になる。**
