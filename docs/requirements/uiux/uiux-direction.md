# UI/UXデザイン方針書 — AI採譜アプリ

**確定日:** 2026-07-19
**成立過程:** 3モデル調査（grok=X実ユーザー実証 / codex=設計理論一次資料 / Fable=競合・受賞デザイン分解）→ 相互レビュー会議（meeting-grok.md / meeting-codex.md）→ 本書で統合
**ユーザー要望:** 直感的で、何をするのか色で分かりやすく区分され、マイクロインタラクションが入り、「操作するのが気持ちよい〜！」となるUI

---

## 1. 打ち合わせ議事録サマリー

### 1-1. 全員一致の合意

- **方向性はハイブリッド「A 静かな工房（土台60%）× C 進化する下書き（差別化核30%）× B 弾ける歓び（調味10%）」**。3者とも同配分に収束（配分の数値はgrok提案、codexは「達成点だけに限定」と同旨、Fableは原案提示者）
- 気持ちよさは装飾ではなく**応答時間・吸着・試聴音・短いモーションの整合**で作る（codex原則#10、grok F17/F19、Fable juice分解）
- 信頼度表示は**色+形状の二重符号化・常時可視・掘らせない**（3者一致。WCAG 1.4.1、MuseScore 4の反面教師）
- **モードロック禁止**。Dorico型のWrite/Engrave往復はX上の最大摩擦源（grokアンチパターン#1）。単一キャンバス＋文脈ツールで
- 「分離→MIDI→コード→楽譜」の**ワンパス導線**（Logicが口コミベンチマーク。grok Top10#1、codex賛成）

### 1-2. 会議で補正された点（相互レビューの成果）

| 元の主張 | 補正 | 補正者 |
|---|---|---|

| Fable: 案B「弾ける歓び」は要望に直球 | 連続祝祭は長時間編集層とズレる（haptic過多でアプリ離れの実証F18）。調味10%に格下げ | grok・codex |

## 2. 確定方針: 静かな工房60% × 進化する下書き30% × 弾ける歓び10%

### A層: 静かな工房（土台60%）— 長時間編集に耐える集中の道具

**やる:** 低彩度キャンバス（テキスト/アイコンは40-60%不透明度に沈め、状態・優先度・操作可能要素のみフル彩度）/ 4pxグリッドの秩序 / キーボード第一（全主要操作にショートカット＋コマンドパレット）/ 遷移<100msの楽観的UI（先に見た目を変え確定は裏で）/ 起動即ワークスペース（スプラッシュ・広告・余計な初回画面ゼロ）/ 単一キャンバス＋文脈ツール＋薄いステータス表示（確認中/編集中/書き出し前）
**やらない:** モードによる操作ロック / 事務ソフト風リボン・クラッター / 常用操作への演出

### C層: 進化する下書き（差別化核30%）— AIの正直さを体験の中心に

**やる:** 部分結果の先出し（確定セクションからskeleton→実譜面を逐次描画、全曲完了を待たせない）/ 停滞しない段階ラベルつき前進バー（ステム分離中→拍・小節推定中→整譜中）/ 信頼度の二重符号化（琥珀・破線/ハッチ=要確認、緑・実線=確定）を常時可視 / 触るたびに「確定」へ色が育つ0.2s遷移 / 小節単位の「採用・別候補・手修正・この範囲だけ再解析」/ 非破壊3レイヤー（原音・AI下書き・人間編集）
**やらない:** 全音符への信頼度%数値の常時表示 / 要確認箇所を階層の奥に隠す

### B層: 弾ける歓び（調味10%）— 祝祭は達成の「点」だけ

**やる:** 解析完了（先頭小節が見えた瞬間の"ready"チャイム1音）/ 要確認の区間クリア / 書き出し成功 — この3点のみ祝祭 / スナップ成功の音ゲーJUST的手応え（1回の軽い音）
**やらない:** 連続ストリーク演出 / 全操作haptic / 全面パーティクル

## 3. カラーシステム設計（意味体系・3系統と衝突ルール）

| 系統 | 用途 | 表示位置（衝突ルール） |
|---|---|---|
| **パート色** | 楽器/声部の識別（自動色分け+命名。DTM民の"fun"の前提） | **左帯・トラックヘッダ**に限定 |
| **状態色** | 青=再生/ナビ、緑=確定済み、琥珀=要確認、赤=エラー | 音符・UIの前景。意味を固定し別用途に使わない |
| **信頼度/AI状態** | AI下書き・低信頼・再解析差分 | **形状オーバーレイ**（枠線・下線・破線/ハッチ・バッジ）。色レイヤーを占有しない |

- 全系統でWCAG 2.2 AA準拠（通常文字4.5:1・非テキストUI 3:1）。**色だけで情報を伝えない**（形状・アイコン・ラベル併用）
- キャンバスは低彩度を保ち、高彩度は帯とステータスのみ（クラッターがX上の主敵。虹色UIより過密が嫌われる）

## 4. マイクロインタラクション設計（統合優先表）

**原則: 常用は静かで速い / 成功は点で打つ / 音は非侵襲（デフォルト極小・個別オフ可・原音再生中は抑制）**

| 優先 | 操作 | フィードバック | 実装手段 | 60fps注意点 |
|---:|---|---|---|---|
| 1 | 音符ドラッグ（拍/ピッチ吸着） | 吸着位置ゴースト＋軽いスナップ＋極小クリック音（成功時1回） | 制約スナップ・transform移動 | DOM再配置しない。確定時のみ譜面再計算 |
| 2 | 音符選択・半音上下・ドラッグ中 | そのピッチを短く試聴（和音は変更後和音を即発音）。「見る・聴く・触る」同時 | WebAudio | 発音はレイテンシ最優先・視覚は最小限 |
| 3 | 低信頼音符クリック | 原音の該当区間ループ＋波形/譜面の同期ハイライト＋視覚的"吸い付き" | 時間マップ・Audio loop・Canvasオーバーレイ | 波形は差分レイヤー描画。全体再描画禁止 |
| 4 | 要確認を確認/修正して確定 | 破線→実線・琥珀→緑＋チェック形状の0.2s遷移。区間クリア時のみ淡い祝祭 | 状態更新・CSS opacity/transform | 大量音符の一括アニメは範囲単位で間引く |
| 5 | 解析待ち | 段階ラベルつき前進バー＋部分譜面の逐次立ち上がり＋ready1音 | Worker/別プロセス・逐次レンダリング | IPCを詰まらせない。描画キュー間引き |
| 6 | 原音同期再生（F-042） | 生きた波形＋再生ヘッドの慣性追従＋現在音符ハイライト＋自動スクロール | Canvas/WebGL | transform/opacity中心 |

## 5. 成功指標（会議合意）

- **初回30秒で譜面が見える**（部分結果先出しの効果測定）
- **低信頼箇所へ1操作で飛べる**（掘らせないの定量化）
- **1000音符表示で操作落ちしない**（60fps維持・仮想化の受入基準）
- **UndoでAI再解析前に戻れる**（音楽的意味単位のUndo履歴）
- 補完: 初回起動→ワークスペース到達がクリック0 / モードによる操作ロック箇所ゼロ / 初見ユーザーが10分以内に最初の修正を完了

## 6. アンチパターン禁止リスト

1. モード往復必須の設計（Write/Engrave地獄）— ツール・操作可能範囲のモードロック禁止
2. 連続祝祭・全操作haptic — 成功フィードバックは点で打つ
3. 掘らせるUI — 信頼度・要確認を階層の奥に隠さない
4. 思想の違う自動化のデフォルト強制 — 自動整譜/レイアウトには**ワンクリックのロック/手動優先**を必ず併設
5. 事務ソフト風クラッター＋起動摩擦（スプラッシュ・広告・余計な初回画面）
6. 急激なUI改変で機能地図を失わせる — 将来の刷新時は旧操作マッピング表示を原則化
7. UIスレッドでのAI処理・波形解析・差分計算 — Worker/別プロセス分離
8. 全音符への信頼度%常時表示 — 低信頼箇所だけを目立たせる

## 7. ユーザー要望との対応表

| 要望 | 応える方針 |
|---|---|
| 直感的 | 起動即ワークスペース / 単一キャンバス＋文脈ツール / 分離→MIDI→コード→楽譜のワンパス導線 / D&D→即プレビュー |
| 何をするのか色で分かりやすく区分 | 3系統カラーシステム（パート色=左帯・状態色=意味固定・AI状態=形状オーバーレイ）。装飾でなく意味体系として |
| マイクロインタラクション | 統合優先表6件（吸着スナップ・ピッチ試聴・色の育ち・区間ループ吸い付き・部分譜面立ち上がり・生きた波形） |
| 操作するのが気持ちよい | 遷移<100ms・楽観的UI・音ゲーJUST的スナップ手応え・達成の点の祝祭。「solid / organized / interfaceが消える」というX上の"気持ちいい"言語に整合 |
