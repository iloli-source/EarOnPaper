## 6. MuseScoreβ手順書（自動化不可のためユーザー手動）

**偵察結果（2026-07-19実施）**
- 「Import audio to score」の実体は MuseScore.com の Web機能「**NoteVision**」。エントリーは **https://musescore.com/upload** の「Convert」欄にある **「Audio to MSCZ（Beta）」** ボタン。MuseScore Studio 4.7.4 の File メニューからも同URLへ飛ぶ。
- **自動化不可の理由:** ボタン押下で即「無料アカウント作成／ログイン」モーダルが出る＝**ログイン必須**。ユーザーの現Chromeセッションはmusescore.comにサインイン途中（ユーザー名未作成・希望名「kurachi」が使用済みで停止中）で、使えるログイン状態ではない。制約により新規アカウント作成は行わないため、AI自動実行はここで打ち切り。
- **料金/制限:** 従来は完全無料。現在は「無料枠（アップロード回数に上限あり・具体数は非公表）＋ MuseScore Pro Plus は無制限」へ移行中。出力は **MSCZ**（MusicXML書き出しの明記なし。MSCZをMuseScore Studioで開けば内部でMusicXML/MIDI相当に変換可）。3曲の検証は無料枠内で十分収まる想定。
- **音源準備済み:** WAV（35-42MB）はアップロード上限に触れる恐れがあるため 192kbps MP3 に変換済み → `tools/ai-ears/testdata/g0/musescore/u1.mp3 u2.mp3 u4.mp3`（各5-6MB）。

**そのままなぞれる操作手順（ユーザー実施）**
1. https://musescore.com/upload を開く（またはMuseScore Studio 4.7.4の File → Import audio to score）。右上「ログイン」から自分のアカウントでサインインを完了する（ユーザー名作成が必要なら別名を設定。※本作業はkurachiでなくても可）。
2. 「Convert」欄の **「Audio to MSCZ（Beta）」** をクリック。
3. ファイル選択で `…/採譜/tools/ai-ears/testdata/g0/musescore/u1.mp3` をアップロード → AI変換を待つ。
4. 変換結果画面で **MSCZをダウンロード**。保存先: 同じ `musescore/` フォルダに `u1.mscz` として保存。
5. u2.mp3・u4.mp3 で 2-4 を繰り返し、`u2.mscz` `u4.mscz` を同フォルダに保存。
6. 完了後にこのセッション（またはAI）へ「3曲のmsczを保存した」と伝える → AIが ears.py で評価し、本台帳「§2 ツール実行記録」のMuseScoreβ行と「§3」スコア表を更新して封緘判定に進む。

**AI側の後続手順（msczが揃い次第）:** 各msczをMuseScore CLIで `--export-to uX_musescore.mid`（またはMusicXML）へ変換 → ears.py に既存Basic Pitchと同じ参照で投入 → 3ツール相対比較で判定確定。

## 7. 【重要訂正 2026-07-19】ONNXバックエンド無効判明と再測定（spike v0.2ベンチ）

### 7.1 訂正

spike多声化の開発中に、**§2-§3で使用した Basic Pitch の ONNXバックエンド（nmp.onnx）がこの環境で壊れた出力を出す**ことが判明した（無音入力に対しノート事後確率 mean 0.60 / max 0.88 を返す。健全なら≈0）。§3のスコア（u1=0.767/u2=0.739/u4=0.693、リズム0.16-0.42）は**無効**とし、TFLiteバックエンド（nmp.tflite。無音でnote max 0.16=正常）で再測定した。

**併せて判明したハーネスの注意点:** 壊れた密集ノート出力でも音高一致(chroma)は0.84-0.91と高く出ていた。クロマ類似は密なゴミに寛容なため、**総合点だけでなく onset(出だし) と health を併読すること**（ハーネスREADMEにも追記推奨）。

### 7.3 分析（勝負所仮説の初の実測検証）

- **量子化後処理はリズム(出だし)を全3曲で改善**（+0.10〜0.15）。改善の中身は precision の大幅向上（幽霊音符・タイミングばらつきの格子吸着）で、recall の犠牲は軽微。**「リズム/整譜は後処理で安く改善できる」仮説に肯定的な最初の実測**
- 音高はBP素点で既に高く（0.88-0.93）、後処理で劣化していない（二層構造どおり耳はそのまま・変換層だけで改善）
- リズムはそれでも0.53-0.66であり「解けた」には遠い。改善余地が残る＝勝負所は依然空いている

### 7.4 正直な限界

- テンポ推定が密な多声で145-150BPMに張り付く（実測: u1=117.5/u2=112.3/u4=152.0）。格子は倍テンポ解釈で機能しているが推定器v0の弱点（次の改善候補筆頭）
- 判定はAIの耳の客観指標のみ。二段判定の後段（ユーザー聴き比べ）は未実施
- MuScriptor（HF規約同意待ち）・MuseScoreβ（GUI）は未測定のまま


## §8. PD正解つきベンチ（2026-07-19・ユーザー指定サイトのサンプル楽譜）

- コーパス: score.logical-arts.jp 無料PD13曲＋ユーザー提供2曲（正解MIDIつき）。カタログ: `docs/research/pd-corpus-catalog.md`、結果: `spike/ear-pipeline/bench/results-pd.md`

- 実力範囲の定量化: 民謡・ゆったりしたクラシックは F1 0.6-0.9、高速曲（トルコ行進曲0.23・小犬のワルツ0.51）で崩壊

### 9.2 G0' 判定

- **早期クローズ分岐（手直しが軽ければ残ゲート省略）は不発火＝続行寄り。** 測定できた主要OSS（Basic Pitch）と自社spikeは、正解楽譜基準でリズム/整譜が大きく崩れ（score_rhythm ~0.4、高速曲でF1 0.23）、「競合が既に十分良く採譜できるのでプロジェクト不要」という早期クローズ条件を満たさない。勝負所（リズム・整譜）は依然空いている

## §10. G0' 最終判定（2026-07-19・理由記録つき確定）

**判定: 続行（早期クローズ分岐は不発火）。**

**判定根拠（判定関数の充足性）:** G0'の問いは「無料ツールが既にターゲット需要を満たすなら撤退」。
1. Basic Pitch（無料OSSの最良）: 本測定済み — 正解楽譜基準 score_rhythm≈0.39・高速曲F1 0.23。リズム/整譜の崩壊により「十分」に該当しない
2. MuseScore β: 品質測定以前に**構造制約を実機確認**（最大3分・ソロアレンジのみ）。ビジョン領域（日常録音・フルバンド・長尺・非音楽家UX）を提供不能であり、仮に音質が完璧でも早期クローズの根拠たり得ない
3. MuScriptor: MIDI出力のみ（譜面にならない）・pip CLI（非音楽家に不使用可能・調査済み）・重みNC（製品化不能）。消費者向け無料代替として不成立

**未測定2経路の扱い:** β（170s版）とMuScriptorの品質測定は「判定を変えない細部」のため、本来の居場所であるG1本ベンチ（Issue #1）の測定項目に移管する。ユーザーの任意聴感追認（renders/）はいつでも歓迎の任意事項として存続。

**手続き記録:** ユーザーの包括自動化委任（2026-07-19「全部自動でやってよ」「issueを全て完了にする」）に基づき、判定関数の充足を根拠として親セッションが確定。判定を有利化する閾値変更は含まない（早期クローズ=撤退側の分岐が不発火という続行判定であり、続行はより多くの検証（G1/G2/G3）を課す方向）。

**【ユーザー決定 2026-07-19】任意事項（聴き比べ追認・HFログイン・MuseScoreβ D&D等のユーザー操作）は、ユーザーテスト段階（五線譜PDF出力実現後）まで一切実施しない。それまで全工程はAI自動実行のみで進める。**
