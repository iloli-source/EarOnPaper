## 最終判定: **条件付き封緘可（3項目の追記後に封緘）**

Round 3 の7指摘は**5件が完全処理・2件が概念採用のみで運用粒度に未達**。設計思想は投資判断に耐える水準まで到達した。ただし**このままでは「1日目の作業手順書」が書けない**。封緘をブロックするのは新規欠陥ではなく、既存設計の「数値と手順の空欄」である。下記A・B・Cを埋めれば封緘してよい。逆に空欄のまま封緘すると、Round 3 指摘5（ゴールポスト後動）が別の形で再発する。

## 封緘条件A: 3帯の帯境界（25%/60%）の根拠が空欄（CRITICAL）

指摘5への処理は「事前固定する枠組み」は作ったが**数値の根拠が入っていない**。25%と60%はRound 3で私が批判した「15分/30分」と同じ構造の恣意的閾値であり、v3の欠陥をv4がラベルだけ替えて温存している。

- **なぜ問題か:** 手直し比率25%（AI下書きでフル耳コピの1/4に短縮）と60%（半分強しか削れない）の境界が、なぜその値か。フル耳コピ比という指標自体は正しいが、**閾値の校正手続きが無い**。校正なしに封緘すると、測定後に「25%は厳しすぎた/緩すぎた」と動かせる余地が残る。
- **封緘前に埋めるべき:** (1) 基準線曲を**2〜3本先行測定し、既存OSS（Basic Pitch）の手直し比率の実測値でアンカリング**する。「無料下限が既に手直し比率X%」という実測点から相対的に帯を引けば、25/60の恣意性が消える。(2) それが無理なら、25/60を**単一点でなく「校正待ちのプレースホルダ」と明記**し、フェーズA先頭3曲の実測後に確定→封緘、という順序を仕様に書く。「Round 4で確定」は誰がどの数字を根拠に確定するかが不在。

## 封緘条件B: グレー帯（25-60%）に落ちた場合の「次の一手」が循環（HIGH）

指摘6のGO側は埋まったが、**最頻結果であるグレー帯の出口が自己参照している**。§5判定表は「グレー帯→フェーズB移行。B後もグレーなら『安く勝てるか』列で判定」とするが——

- **なぜ問題か:** 「安く勝てるか」列（§5・G0接続）は「崩れが後処理3-6人月で埋まる種類か」を誤り種別分解から判定する定性基準であり、**帯判定とは独立の軸**。つまりグレー帯の出口は「別の定性判断に委ねる」であって、実質「フェーズBをやってもグレーなら人間が総合判断」＝宙吊り回避になっていない。Round 3指摘6で私が批判した「中間結果で意思決定が宙吊り」がB段階に先送りされただけ。
- **封緘前に埋めるべき:** グレー帯かつ「安く勝てる」判定が出た場合の**具体的遷移先を1本の矢印で固定**する。案: 「グレー帯 ∧ 崩れが規則的（後処理可能）→ G2需要ゲートへ暫定進行し、G1は§5スパイク（3人月）で最終決着」。グレー帯 ∧ 崩れが非規則的（モデル刷新要）→ 即撤退。この2分岐を判定表に1行追加すれば循環が切れる。

## 封緘条件C: 手直し比率の測定手順が「誰が・何で・何を数えるか」の粒度に未達（HIGH）

指摘1・3で「手直し比率」「2名測定」の概念は採用されたが、**1日目に測定者が机で再現できる手順書の粒度に達していない**。フル耳コピ比の分子・分母の定義が曖昧。

- **穴1（何を数えるか）:** 「手直し時間」の開始点・終了点が未定義。AI出力を開いた瞬間からか、内容を読み込んでからか。「納品可能」の停止条件（＝終了点）は§5に無い（指摘5の操作的定義が結局残っている）。停止条件が無いと手直し時間は測定者裁量で伸縮する。
- **穴2（何のツールで）:** 手直しをどのソフト（MuseScore / Finale / スコアメーカー等）で行うか未固定。ツールが違えば同一出力でも手直し時間が2倍変わる。測定者2名で別ツールを使えば時間差がスキル差かツール差か分離不能。
- **穴3（誰が測るか）:** 時間計測の主体（測定者の自己計時か第三者の傍観計時か）と、ブラインド化の実装（出力ファイル名から製品名を除去する具体手順、誰がマスキングするか）が未記述。自己計時＋ブラインドは両立が難しい（自分でツールを開けば製品が分かる）——この矛盾の解決策が仕様に無い。
- **封緘前に埋めるべき:** 測定プロトコルに以下を1枚追記。(a) 手直しの開始＝「AI出力をエディタで開いた時刻」、終了＝「操作的定義を満たした時刻」で、**終了の操作的定義**（例: Note F1が基準線起譜と同水準 ∧ 拍子・小節線に構造誤りゼロ）を数値で固定。(b) 手直しツールを1つに統一固定（国内文脈ならスコアメーカー or MuseScore）。(c) ブラインド化＝「第三者（代表以外）が出力を無名IDにリネームし台帳と対応表を分離保管、測定者は製品名を知らずに作業」の実装手順。

## 曲ID台帳ひな形の列提案（§5「曲ID付き台帳」の具体化）

§5は「曲ID付き台帳で公開」とあるが列が未定義。追試可能性を担保する最小列セットを提案する。

**曲メタ列:** 曲ID / 曲名 / アーティスト / ジャンル（ソロP・弾き語り・ポップバンド） / 難度（編曲密度3段階） / 楽器編成（単声/多声・パート数） / テンポ帯（BPM区分） / 調 / 尺（小節数or秒） / 音源品質（サンプルレート・ステレオ/モノ・ラウドネス・生演奏/打込み） / 著作権状態（PD・自作・許諾済み・引用の別。**測定に使う音源の権利根拠を必ず1列**——G3法務ゲートと矛盾しないため）

**測定結果列（曲×製品ごとに1行）:** 製品名（対応表側にマスク） / 測定者ID（熟練/準初級） / 手直しツール / 基準線耳コピ時間 / AI手直し時間 / **手直し比率** / 誤り種別内訳（音高・音価・拍子小節線・声部分離・その他の各修正所要） / mir_eval Note F1 / Onset F1 / 帯判定 / 「安く勝てるか」判定（規則的/非規則的） / 生出力ファイルパス

著作権状態列は装飾でなく必須。ベンチ音源自体が権利処理されていないと、台帳公開（§5追試可能性）がG3の法務リスクを自ら作る。

## 1日目作業手順書が書けるか — 残る空欄の総括

現状で書ける: 対象5本の選定・役割分け・2段階の全体像・台帳を作る意図。
現状で**書けない**（＝封緘ブロッカー）:
1. 帯境界25/60の確定値（条件A）→ 何%で撤退の紙を測定者に渡せない
2. グレー帯の遷移先（条件B）→ 最頻結果の出口が未定
3. 手直しの開始/終了・ツール・ブラインド実装（条件C）→ 測定者が時計を止める瞬間を定義できない
4. 基準線曲の選定（先行測定用2-3本）→ 条件Aの校正に必須だが§5に「先に測る」順序が無い

**封緘可否の結論:** 設計思想は封緘水準。ただしA・B・Cの3枚（帯校正手順・グレー遷移1行・測定操作定義1枚）を追記するまでは「測定開始」の封は押せない。この3枚は新規調査でなく**既存仕様の数値埋めと順序明記**のみで、Round 4内で完了可能。3枚が入れば1日目手順書が書け、封緘してよい。
