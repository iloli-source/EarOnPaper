## 総評

v4は「事前基準の固定・帯判定・封緘・ゲート列」というRound 3の指摘（本レビュアーの§2-5）を実装した点で、意思決定装置として大きく前進した。だが「設計」から「運用」に移す段で、**誰が・いつ・何を確定するか**の手続き粒度がまだ足りない。以下、6観点で辛口に指摘する。最大の火種は「G2需要ゲートのバイアス汚染」と「最終判定会の主体が未定義」の2点である。

## 1. 実施順序と並行関係 — G2並行の設計は正しいが「順序バイアス」の防御が抜けている

v4はG0を判定済み（条件付き通過）とし、G1（技術ベンチ）とG2（需要）を並行、G3（法務）をG1・G2が撤退帯でなかった場合に後置する。順序自体は費用最小化の観点で合理的だ。G3を最後に置いて弁護士費用を条件付き支出にするのは正しい。

**しかし本レビュアーが指摘したい残存リスクは「G2インタビューがG1ベンチ結果を知った後に行われるとバイアスが乗る」点である。** v4はG1とG2を「並行」と書くが、実務上インタビューの実施は数日〜1週間かかり、ベンチのフェーズA（3日）が先に結果を出す可能性が高い。ベンチで「競合が崩れている＝勝機あり」と見た測定者が、そのままインタビュー設計・質問・解釈に「欲しい答え」を混入させる。逆にベンチがグレーだと、インタビューの熱量が下がり過小評価に振れる。

- **是正1:** G2のインタビュー質問文・撤退基準（5件中0件で撤退）を**G1着手前に封緘**する。質問設計をベンチ結果から独立させる。
- **是正2:** できれば**インタビュー担当とベンチ測定者を分離**し、相互に中間結果を共有しない「情報隔壁」を敷く。1人運用なら、G2の質問実施をG1フェーズA結果の開封より前に着手する順序固定で代替する。
- **是正3:** 「並行」の定義を「同一カレンダー期間に走らせる」ではなく「**相手のゲートの中間結果を見ずに各自の事前基準で判定する**」と運用定義せよ。カレンダー並行よりも情報並行が本質。

## 2. 3値判定の網羅性 — G1のみ3帯定義済み、G0・G2・G3は実質2値

v4を精査すると、通過/撤退/グレーの3値が明示的に定義されているのは**G1（明確撤退帯／グレー帯／勝負所濃厚帯）のみ**。他は2値以下に留まる。

| ゲート | 現状の判定値 | グレー帯の有無 |
|---|---|---|
| G0 経済性 | 「条件付き通過」= 実質グレー（ベンチ依存に委譲） | ○（ただしG1へ委譲する形の変則グレー） |
| G1 技術 | 3帯（撤退/グレー→B/勝負所） | **○ 完備** |
| G2 需要 | 「5件中0件なら撤退」= **通過/撤退の2値のみ** | **✕ 欠落** |
| G3 法務 | 弁護士3問の回答（黒/白）＝ 暗黙2値 | **✕ 明示なし** |

- **G2のグレー欠落が最も危険。** 「5件中0件で撤退」の裏返し（何件で通過か）が未定義。1件でも前向きなら通過なら基準が緩すぎ、5件全員でないと通過しないなら厳しすぎる。**3値化案:** 撤退=前向き0件、グレー=1〜2件（追加5件でフェーズB相当の再測）、通過=3件以上。グレー時の追加インタビュー数も事前固定せよ。
- **G3も3値が要る。** 弁護士回答は「白/黒」の二分では返らない。現実の回答は「グレー（条件付き適法・契約でクリーン化可能）」が最頻値だ。3値化案: 通過=3問すべて成果物非関与構造で適法性が立つ、グレー=契約設計・利用目的限定で回避可能、撤退=構造的に30条の4ただし書or変換型該当が覆せない。**グレー時に誰がクリーン化設計の可否を判断するか**まで決めておく。
- **G0は「グレーをG1に委譲する」変則構造。** これ自体は許容だが、「G1が勝負所濃厚でも“安く勝てる列”がNGならG0で撤退」という**再判定ループ**が発生する。G0の最終確定タイミング（G1判定と同時か、その後か）を明記せよ。

**指摘:** 全ゲートで撤退/グレー/通過の3値と、各グレー帯の「次に何をするか（追加測定・追加ヒアリング・契約設計検討）」をセットで定義せよ。2値ゲートは境界事例で判定者の願望に流れる（Round 3 §2で指摘した崖の再来）。

## 3. 封緘運用 — 「封緘する」とは書いてあるが手続きが未定義

v4は「Round 4レビューを通してから封緘し、封緘後に測定を開始」（§5冒頭・判定表）と明記した。方針は正しい。だが**封緘の実務手続き（誰が・いつ・どこに・どう改訂するか）が空欄**である。

- **誰が:** 封緘の署名主体が未定義。1人プロジェクトでも「封緘者＝ユーザー本人」を明記し、AIは封緘案を提示するのみ、と役割分離せよ。判定表を書いた者と封緘承認者が同一だと封緘の意味（後知恵防止）が半減するため、最低でも「日付＋バージョン＋変更不可宣言」を記録に残す。
- **いつ:** 「Round 4レビュー完了後、G1・G2着手前」の1点に固定。着手後の封緘は無効（着手自体がコミットメントになるため。Round 3 §5指摘）。
- **どこに:** 封緘版を独立ファイル（例: `sealed/gate-criteria-sealed-v1.md`）として切り出し、以後の編集を禁止。conclusions本体に混ぜると事後編集の履歴に紛れる。
- **改訂手続き（重要・v4に完全欠落）:** 封緘は「禁止」ではなく「理由記録つきで改訂可」とすべき。実測中に測定プロトコルの欠陥（例: 選曲の偏り）が判明することは十分ありうる。ただし**改訂は「基準の数値」ではなく「測定手続き」に限定**し、①改訂理由 ②改訂日時 ③改訂が判定結果を有利化していないことの自己申告 を封緘ファイルに追記する。**判定閾値（≤25%等）の実測開始後の改訂は原則禁止**（これを緩めると封緘が無意味化する）。

**指摘:** 封緘は「凍結の儀式」ではなく「変更を追跡可能にする仕組み」。改訂可否を手続き（変数=手続き改訂可／閾値改訂原則不可）で線引きし、改訂ログの様式を事前に用意せよ。

## 4. 最終判定会の設計 — 主体が未定義、Round 5との関係も曖昧

v4 §7は「Round 5で最終判定文書を確定」とするが、**誰が最終判定を下すか**が書かれていない。これは意思決定装置の心臓部が空欄ということだ。

- **判定主体:** 経営判断（Go/クローズ）はユーザー本人が下す。AIレビュアー群（Round 5）は判定材料を提示するのみで、**投票権を持たない**ことを明記せよ。Round 3で本レビュアーが指摘した「合議の一致は独立試行でなく相関の高いレビュアー」問題（§4）が、最終判定でも再発する。AI合議を判定の根拠に使わないこと。
- **Round 5との関係:** Round 5は「実測結果への最終批判」であって「判定そのもの」ではない。順序は **①全ゲート結果が出揃う → ②Round 5でAI群が実測結果を批判（バイアス・測定不備の洗い出し） → ③ユーザーが最終判定文書に署名**。Round 5を判定会と混同すると、AIの批判が判定を代行してしまう。
- **判定会の入力物を事前定義:** 判定会には「4ゲートの結果表＋封緘版判定表との照合＋各ゲートで実際に撤退帯/グレー/通過のどこに落ちたかの記録」を必須入力とする。口頭やその場の印象で判定させない。
- **全ゲート撤退帯ゼロでも自動Goにしない:** どのゲートも撤退帯でなくても、それは「必要条件クリア」に過ぎず十分条件ではない（Round 3 §3で指摘済み）。判定会では「進める積極的理由が立証できるか」を問い、立たなければデフォルト撤退（v4 §5末尾の原則）を最終判定会にも適用せよ。

**指摘:** 最終判定会の「主体＝ユーザー／材料提供＝Round 5 AI群」の分離を明記。判定会の入力物と、Go成立に必要な積極的立証を事前定義せよ。

## 5. 中間報告・中断基準 — ゲート内の途中撤退が未設計

各ゲートに時間箱と中止条件は付いた（§5）が、**ゲート実行の途中経過での中断基準**（ゲート完了を待たずに撤退する条件）が無い。

- **早期撤退トリガー:** 例えばG1フェーズAの初日で、撤退判定用3本すべてが手直し比率≤10%（明確撤退帯を大きく下回る）と出たら、残りの選曲を測るまでもなく即撤退してよい。「明白な足切りは全曲測定前に発火させる」early-kill条件を各ゲートに1つ設けよ。逆に途中で「勝てる」兆候が出ても走り切る（活用側への早期確信は封緘破りの温床）。
- **中間報告の様式:** フェーズA完了時（3日目）に「各帯にどれだけ寄ったか」の中間報告を1枚出す。これはユーザーへの進捗共有であると同時に、フェーズB移行の要否判定の入力になる。報告に「判定者の主観コメント」を書かせない（数値と誤り種別分解のみ）。
- **予算超過アラート:** 各ゲートの時間箱・現金上限（G1スパイクは3人月・10万円）に対し、80%到達で黄色・100%で赤（強制中断）のトリップワイヤーを引く。サンクコストによるエスカレーション（Round 3 §5）の構造的防止。

**指摘:** 各ゲートに ①early-kill条件（完走前撤退）②中間報告様式（数値のみ）③予算トリップワイヤー を追加せよ。

## 6. タイムライン見積 — 開始から最終判定まで

各ゲートの時間箱を積み上げ、G1・G2並行・G3後置の前提で見積もる。営業日ベース。

| フェーズ | 内容 | 所要（営業日） | 依存関係 |
|---|---|---|---|
| Round 4 完了・封緘 | v4批判＋実行系設計レビュー→判定表/質問文/インタビュー設計を封緘 | 2〜3日 | 全ゲートの前提。ここが律速 |
| G1-A ＋ G2 並行 | ベンチ フェーズA（n=9・3日）／需要インタビュー5件（2.5h＋日程調整） | 3〜5日 | 情報隔壁下で並行 |
| G1-B（条件付き） | フェーズAがグレーの場合のみ n=25-30 拡大 | +5日 | G1-Aがグレー時のみ発生 |
| 中間判定 | G1・G2の帯照合、撤退帯なら即終了 | 1日 | G1-A/B・G2完了後 |
| G3 法務 | 弁護士3問発注→回答（往復・先方リードタイム含む） | 5〜10日 | G1・G2が撤退帯でない場合のみ発注 |
| GO側スパイク | Basic Pitch＋後処理プロト1本（上限3人月） | 最大60日 | 全ゲート通過時のみ。通常はここまで到達しない |
| Round 5・最終判定会 | 実測結果への最終批判→ユーザー署名 | 2〜3日 | 全結果出揃い後 |

**代表シナリオ別の所要:**

- **早期撤退（G1-Aで明確撤退帯）:** Round 4封緘含め **約6〜8営業日**でクローズ。最頻・最安。
- **グレー進行→法務まで（スパイク未着手）:** フェーズB＋G3往復込みで **約20〜28営業日**（3〜5.5週）。
- **全ゲート通過→スパイク:** 上記＋最大60営業日。**約4ヶ月**。ただしv4自身が「確率の低い1シナリオ」と認める経路。

**律速はRound 4の封緘とG3弁護士の先方リードタイム。** ベンチ・需要は自力で3〜5日に圧縮できるが、弁護士往復は外部依存で伸びやすい。G3発注を「G1・G2が撤退帯でない」と判明した瞬間に即発注する運用にすれば、往復時間を他ゲートと重ねられる（現設計は「後置」だが、撤退帯でないと分かった時点での前倒し発注を許容すべき）。

## 結論（運用設計としての評点）

| 観点 | 評価 |
|---|---|
| 実施順序・並行 | 良（順序は最適）／要改善（情報隔壁・順序固定でバイアス防御） |
| 3値判定の網羅 | 不足（G1のみ完備、G2・G3が2値、グレー時の次アクション未定義） |
| 封緘運用 | 方針良／手続き未定義（誰・いつ・どこ・改訂ルール） |
| 最終判定会 | 未設計（主体未定義、Round 5と混同リスク） |
| 中間報告・中断 | 不足（early-kill・中間様式・予算トリップワイヤーなし） |
| タイムライン | 明示なし（本レビューで補完・6〜8日〜4ヶ月） |

**一行で:** v4は「良い判定表」を持つが「運用手順書」がまだ無い。バイアス隔壁・全ゲート3値化・封緘改訂ルール・判定主体の明記・early-killの5点を足せば、プランBは「実行可能なゲート運用」になる。最優先はG2バイアス隔壁と最終判定主体の明記。
