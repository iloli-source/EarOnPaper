## 0. 一行評決（先に結論）

**プランB採用は「机上永久ループよりマシ」だが、v4のまま走ると「儀式付き延命」になる確率が高い。**  
Round 3 redteamの核心——「ベンチ結果がどう出ても事業No-Goが動かないなら儀式」——は**未反駁のまま**。その上で、v4が「唯一の残存デスク調査」とした **MuseScore AMT参入は、すでにベータ出荷済み**であり、「オプション脅威の定点観測」という構図自体が**1〜2週間遅れで破綻**している。

| 観点 | 判定 | 一言 |
|---|---|---|
| ベンチ選曲 vs 実需要 | **不合格（要再設計）** | 学術軸（テンポ×多声×録音品質）であり、X上の実需要分布とズレる |
| 需要ゲート5–10件 | **設計未成熟** | 対象は見つかるが、撤退閾値とペルソナ混合が甘い |
| 競合再定義 | **重大漏れ** | MuseScore audio2score / Songsterr / Chordify / MuScriptor / 中国系が欠落 |
| MuseScore定点観測 | **前提が古い** | 「参入したら詰み」ではなく「参入が始まった」 |

## 1. ベンチ対象曲の選定基準は、実ユーザーの典型ユースケースと合っているか

### 1.1 v4の現状

v4 §5 の選曲は次の通り。

> テンポ帯×多声/単声×録音品質のサンプリング表を事前作成しセルを埋める  
> フェーズA n=9 / フェーズB n=25–30

これは**MIR研究の層化サンプリング**としては正しい。しかし**事業ベンチの層化軸としては、X上の需要分布と直交している**。

### 1.2 Xから見える「採譜したい曲」の実際の分布

本Roundおよび既存調査で繰り返し出る需要は、学術の「きれいな多声×テンポ」ではなく、**目的×楽器×権利空白**である。

| 実需要クラスタ（Xで高頻度） | 典型行動 | 求める成果物 | ベンチ含意 |
|---|---|---|---|
| **ゲーム音楽・アニソン・東方/ボカロ系ピアノ** | 「楽譜がない→耳コピ→弾いてみた」 | ピアノソロ譜・片手メロ | 市販譜が無い・編曲性が高い |
| **ギターTAB/リフ** | Songsterr AI・手直し・「無いよりマシ」 | GP/TAB | MusicXML→MuseScore前提とズレる |
| **弾き語り・コード譜** | Chordify/手書きコード、市販譜は中古Mercari | コード+メロ、完全譜ではない | フル整譜より**コード精度**がKPI |
| **コピバン（バンドコピー）** | パート分離→各自耳コピ | ステム+パート譜 | Moises前処理が本体、A2Mは後段 |
| **外注（ココナラ）** | 「コピーできない曲を正確に」が売れる | 人の正確譜 | AIの代替候補は**時短下書き**か**失敗** |
| **プログレ/ジャズソロ等の難曲** | プロ外注・専用出品 | 装飾・タイム感まで | 準初級手直し比率の帯判定では測れない |

根拠の例（X実投稿）:

- 市販譜が無い／アレンジが気に入らないので耳コピする、という自己完結需要。
- ゲーム曲はピアノ耳コピが常態。
- 耳コピで「リズム（音価）」がボトルネック——v4の勝負所仮説と整合。
- Songsterr AI採譜は「時短になるが表裏を見失う」「ほぼ手直し」——**ギター縦の現実SOTA**。
- ココナラでゲーム音楽ピアノ採譜が継続購入されている——**人による正確譜への支払意思は存在**。
- メロ譜・コード譜・リードシートが商品単位——フル五線譜が唯一KPIではない。

### 1.3 辛口指摘（選曲設計の欠陥）

1. **層化軸が「研究」で「用途」ではない。**  
   ユーザーは「BPM 90・多声・スタジオ録音」で曲を選ばない。「推しの楽譜がない」「ギターで弾きたい」「バンドで合わせたい」で選ぶ。

2. **v2にあった「ソロピアノ/弾き語り/ポップバンド各3曲」は用途軸に近かったのに、v4で学術軸に薄まった。**  
   回帰ではなく後退。

3. **出力フォーマットが単一（MusicXML→MuseScore想定）なのに、需要の相当割合はTAB/コード。**  
   手直し比率≤25%でも、顧客が欲しいのがTABなら**事業価値ゼロ**。

4. **日本語圏の「権利空白コンテンツ」（東方・同人・ゲーム未譜面化）がセル定義にない。**  
   ここがv4の言う「時間耐久性のある堀（日本語UX）」の唯一の戦場なのに、ベンチ曲リストに明示されていない。

5. **n=9は「全ジャンルが揃ったら決着」と言うが、ジャンル定義が用途クラスタと一致していない。**  
   フェーズAが「全ジャンル撤退帯」と出ても、ギターTAB市場ではSongsterrが別分布——偽の撤退/偽の勝負所が出る。

### 1.4 投資委員会要求（選曲の再封緘条件）

封緘前に**用途マトリクス**を必須化せよ。最低限:

| セル | 曲種例 | 評価する成果物 | 失敗の定義 |
|---|---|---|---|
| U1 ピアノカバー（譜なし） | ゲーム/ボカロ/東方 | 2段ピアノ譜 | 右手メロが聴き取れない |
| U2 弾き語り | J-pop弾き語り想定 | メロ+コード | コード進行の誤認定 |
| U3 ギターTAB | ロック/メタル | 6線+運指 | 表裏（ピック方向/弦）崩壊 |
| U4 バンド（多楽器） | ポップバンド | パート分離後A2M | 声部分離崩壊 |
| U5 ライブ/低品質録音 | スマホ録 | 同上 | ノイズ耐性 |

学術軸（テンポ×多声×録音）は**副次ラベル**に落とすこと。主軸は用途。

## 2. 需要ゲート：インタビュー5–10件の対象はどこにいるか／X上の実在コミュニティ

### 2.1 v4の設計

> B2C想定層（マイナー曲愛好家・弾き語り・コピバン）へのインタビュー5〜10件  
> 「¥5,000以上の買い切りに前向き」が**5件中0件なら撤退**

方向性は正しい（demandの指摘採用）。**実装仕様が甘い。**

### 2.2 対象者の「実在の巣」（発見可能性は高い）

| ソース | 誰がいるか | アプローチ | バイアス |
|---|---|---|---|
| **X（#耳コピ #採譜 #弾いてみた）** | ピアノ耳コピ勢・ギター勢 | 直近投稿者にDM、15分Zoom | 発信できる層＝やや上級 |
| **ココナラ出品者の「購入者」** | 実際に金を払った人 | 出品者経由 or 自分で依頼後フォローアップ | **最強のrevealed preference** |
| **ココナラ出品者本人** | 需要の中身を週次で見ている | インタビュー1件で需要マップを買う | 競合視される |
| **MuseScore日本語ユーザー** | 手作業採譜→再生で直す層 | XのMuseScore+採譜投稿、フォーラム | 無料記譜文化 |
| **Songsterr利用者** | TAB需要・AI採譜既体験 | X「Songsterr AI」言及者 | ギター偏り |
| **軽音/コピバン圏** | パート譜需要 | 大学軽音Discord、X「コピーバンド」 | 学生・時限 |
| **ぷりんと楽譜／楽譜@エリーゼ利用者** | 市販譜がある曲は買う人 | 「無い曲どうしてる？」 | 公式譜依存 |
| **BOOTH/同人譜サークル** | 東方ピアノ譜など供給側 | まらしぃ系譜面文化の周辺 | 供給側視点 |
| **スコアメーカー文化圏** | 国内手直しレイヤ | 既存調査の延長 | 高齢・買い切り |
| **中国語圏（参考）** | 扒谱需要はWeChat/B站/小紅書偏重 | Xだけでは薄い（既存調査通り） | 本PJスコープ外なら明示 |

X上で観測できる「コミュニティらしき塊」:

- **個人クリエイター経済**: ゲーム音楽ピアノYouTuber＋ココナラ採譜が、商業楽譜の空白を埋めている。
- **MuseScore運用者**: 耳コピ結果をMuseScoreに落としてリズム修正——**まさに疎結合ハブの実ユーザー**。
- **Songsterr AI体験者**: 既に「AI採譜の品質感」を持っている比較可能な被験者。
- **コード勢**: Chordify課金・Suno音源にコード付け——フル譜面プロダクトとは別市場。
- **供給側コミュニティ**: 耳コピ職人・浄書・同人譜。支払意思の代理指標になる。

### 2.3 辛口指摘（需要ゲートの穴）

1. **「5件中0件」は統計でなくメンタリティチェック。**  
   便利な撤退スイッチとしては良いが、**偽陰性**（本当は需要があるのにたまたま0）と**偽陽性**（丁寧なインタビュー相手は前向きに言いやすい）の両方に弱い。最低でも  
   - スクリーニング10件 → 本インタビュー5件、または  
   - **過去12ヶ月に採譜・外注・有料楽譜で実支出した人のみ**  
   を条件にせよ。

2. **¥5,000買い切りアンカーが市場とズレる可能性。**  
   代替は「無料Basic Pitch」「Chordify月額」「Songsterr」「ココナラ数千〜数万」「諦め」。  
   人の正確譜に1万円払う層と、AI下書きに5,000円払う層は**別人**。混同するとゲートが無意味。

3. **3ペルソナ混合は信号を薄める。**  
   マイナー曲ピアノ／弾き語り／コピバンは課題が違う。5件を3分割したらセルあたり1–2件。  
   **先行1ペルソナに固定**（投資委員会推奨: 「譜のない曲をピアノで弾きたい層」——X密度最大）し、他は探索インタビュー。

4. **「不満があり前向き」は願望質問。**  
   撤退基準は次に書き換えよ。  
   - **弱い:** 前向き発言  
   - **強い:** 前払い（LP/Stripeテスト）または「今週ココナラに発注した」  
   v4は「LP or 口頭」とあるが、口頭を認めた瞬間にG2は骨抜き。

5. **インタビュー対象リストが文書に無い。**  
   Round 4で「需要インタビュー設計」をやるなら、**今週DMする20アカウントのリスト**まで落として封緘。そうでなければG2は机上のゲート。

### 2.4 実行可能なリクルート手順（委員会の最低要件）

1. X高度検索: `耳コピ OR 採譜` + `楽譜がない` / `Songsterr` / `ココナラ`（直近30日）  
2. 各クラスタから5人スクリーニング（支出の有無でフィルタ）  
3. 本インタビュー8件（1クラスタ濃厚5 + 他3）  
4. 並行でLP前払い（価格帯: 3,000 / 5,000 / 9,800）  
5. **口頭の「買いたい」は通過条件に使わない**

## 3. 競合再定義の漏れ（新興・中国・インド・隣接代替）

### 3.1 v4のAMT直接競合集合

> Songscription / Klangio / AnthemScore / Basic Pitch ＋「→MuseScoreで編集」疎結合ハブ  
> MuseScore = オプション脅威

### 3.2 致命的漏れ（優先度順）

#### ★致命: MuseScore はすでに AMT ベータを出している

MuseScore Studio **4.7.4（2026-07-07）** リリースノート:

> Import audio to score: Shortcut in the File menu to our new **online MP3 to MSCZ converter**, which is currently in beta.  
> リンク: `musescore.com/upload?format=audio2score`

フォーラムではエンジン推測として **Basic Pitch（速度）/ Omnizart（精度）** まで議論済み。  
**「AMT参入したら詰み」の条件付き未来形は、すでに現在完了形に近づいている。**  
v4の「オプション脅威に格下げし定点観測」は、**格下げしすぎ・観測開始が遅すぎ**。

→ 投資委員会要求:  
- 撤退判定用ツールに **MuseScore audio2score を第4本として即追加**  
- 品質がSongscription級なら「時間窓18–24ヶ月」ではなく**時間窓0–6ヶ月**に修正

#### ★重大: オープンソース前線がv4から消えている

| 漏れ | なぜ重要か | 根拠 |
|---|---|---|
| **MuScriptor（Kyutai × Mirelo）** | 多楽器フルミックス→楽器別MIDI。2026-07にOSS公開でX過熱 | X公式・紹介多数。プロジェクト自前の `x-grok-survey.md` でも主戦場 |
| **Omnizart** | MuseScore実装候補として名指し | 4.7.4フォーラム |
| **Open Music AI** | 日本語ユーザーがBasic Pitchの待ち時間から鞍替え | X実報告 |

「安く勝てるか」列でOSS後処理を前提にするなら、**ベンチからMuScriptorを外すのは設計矛盾**。v2では触れられていたのにv4で落ちている。

#### ★重大: ギター縦の支配者 Songsterr AI

日本ユーザーが実利用・「時短になるが手直し必須」と評価。  
TAB市場ではKlangio/Songscriptionより**体感競合**。v4の「疎結合ハブ＝MuseScore」は**ピアノ/記譜勢に偏った世界観**。

#### ★重要: コード市場 Chordify

日本語圏で「耳コピ不要」の成功報告が多い（既存X調査）。  
フル譜面ではなく**コード抽出で需要の大半が満たされる**層が厚い。ここを無視した「整譜勝負」は、そもそも顧客がいない戦場で勝とうとしている可能性。

#### ★重要: 前処理層（代替労働）

| ツール | 役割 |
|---|---|
| Moises | ステム分離＝耳コピ労働の本体削減 |
| RipX / Melodyne | プロA2M |
| Demucs系 | 無料前処理 |

X日本語の成功パターンは既存調査どおり  
**(1) ステム分離 → (2) A2M → (3) コード** の組み合わせ。  
単体AMT勝負の世界観はユーザーの現実ワークフローと不一致。

#### 中国（Xは薄いがWeb一次では厚い）

| 名前 | 位置づけ | 含意 |
|---|---|---|
| **AudioJam** | AI扒谱・伴奏分離、月¥13〜年¥98帯、1M+訴求 | 価格破壊・モバイル練習特化。v3でも触れたがv4競合集合から消えた |
| **爱扒谱 (aibapu.cn)** | 音声→五線譜・分離・MIDI一式の中文Web | 「フルスタック中文」 |
| **PianoTrans** | ピアノ独奏MIDI、効率10倍訴求（X中文） | 単楽器特化 |
| **MoChord** | コード/指板/五線一体 | コード市場 |

中国は「AMT単体SaaS」ではなく**扒谱＝練習OS**として売っている。  
v4の「中国向けは買断一択」は課金形態の話で、**競合マップには載せていない**。

#### インド

本RoundのX・Webでは**インド発の強力な商用AMTプロダクトの社会証明は薄い**。  
ただし「不在＝安全」ではない。sargam/非西洋記譜はv2–v3で仮説として潰れているが、**英語圏グローバルツールのインドユーザー浸透**は別問題。現時点では**ウォッチリスト低優先**でよいが、「調査したので不在」と書くなら出典を残すこと。

#### その他ウォッチ

- Logic / Ableton 内蔵 Audio-to-MIDI（DAWユーザーの離脱）  
- NeuralNote  
- StemCraft型（AI生成曲の逆解析・TAB化）  
- 台湾 Academia Sinica Omnizart

## 4. MuseScore AMT参入の定点観測方法（v4の「例外デスク調査」を更新）

### 4.1 前提の修正

| v4の言い方 | 委員会の言い方 |
|---|---|
| 参入したら個人余地消滅／定点観測 | **2026-07-07にベータUIが製品に載った。観測は「兆候」から「品質・課金・デフォルト化」へ格上げ** |

### 4.2 観測プロトコル（週次15分で足りる）

| 頻度 | 何を見るか | シグナル定義 | アクション |
|---|---|---|---|
| **週次** | `audio2score` に固定3曲（U1/U3/U4）を投入し手直し比率メモ | 準初級手直し≤40%が2曲以上 | G1の「明確撤退」に準じて**案件凍結レビュー** |
| **週次** | MuseScore.org リリースノート / MuseHub | ベータ表記削除・オフライン化・無料化 | 時間窓を「閉鎖間近」に更新 |
| **週次** | GitHub `musescore/MuseScore` main（Studio 5）と audio/transcription 系PR | ネイティブ統合・VST/MIDI連携と同時進行 | 疎結合戦略の前提崩壊 |
| **隔週** | フォーラム「Import audio to score」スレ | エンジン開示・精度苦情の質 | エンジンがSOTA寄りなら即撤退議論 |
| **月次** | Muse Group全体（Ultimate Guitar / Songsterr AI / NoteVision OMR） | AMT+OMR+TABのバンドル | 「挟み撃ち」がOMR誤解ではなく**本当に形成** |
| **イベント時** | ISMIR/製品発表/資金調達 | MuScriptor級の内製or提携 | T4堀（日本語）が無効化される速度を再計算 |

### 4.3 「兆候」から「kill switch」への昇格条件（事前封緘せよ）

次のいずれかで**プランBを打ち切り、プランA相当のクローズ**に戻すことを文書化せよ。

1. MuseScore audio2score が、自前ベンチの撤退判定用3本と同等以上  
2. 無料枠で商用利用可能なA2Sがデフォルトオン  
3. Songsterr AI × MuseScore の公式連携で MusicXML/GP 双方向がワンクリック  

v4は「定点観測を継続タスク化」と書くだけで、**kill switchが無い**。オプション価値の監視に見えて、実態は**見て見ぬふり用TODO**。

### 4.4 観測チャネルの優先順位

1. 一次: 自前でベータを叩く（最重要。フォーラムより雄弁）  
2. 一次: [musescore.org/en/4.7.4](https://musescore.org/en/4.7.4) と後続リリースノート  
3. 一次: GitHub Projects / main branch  
4. 二次: X `MuseScore (audio OR transcription OR 採譜 OR audio2score)`  
5. 二次: MusicRadar級レビュー（Klangio同様）  

「フォーラムと公式発表だけ見る」は**弱い**。すでにUIショートカットが製品にある。

## 5. プランB全体への構造批判（採択済み前提）

プランBを否定はしない。**設計の甘さを放置したまま実行するな**、が委員会の要求。

### 5.1 G0「条件付き通過」はゲートのふりをした通過儀礼

整譜SOTA自社開発は却下、OSS+3–6人月のみ可——これはゲートではなく**制約の再掲**。  
真のG0は「MuseScoreベータ＋MuScriptor公開後でも、まだ3–6人月の隙があるか」の**再計算**である。未実施。

### 5.2 ゲートの独立性神話

v4は「1つでも撤退帯なら全体撤退」「順不同」と書くが:

- G1で「勝負所濃厚」→G2で需要ゼロ、は意味がある  
- しかし **G2需要ゼロでもG1をフルでやる**のは機会費用の無駄  
- demandが言った「インタビュー2.5時間がデスクリサーチ5周を上回る」は、**G2をG1より先または同時の“軽い先出し”**にすべき根拠  

プランB採択下の推奨順序:

```
G0' MuseScoreβ+MuScriptorの30分机上アップデート（半日）
  → G2 スクリーニング+前払いLP（3–5日、低コスト）
  → G1 フェーズA（用途マトリクスでn=9）
  → G3 弁護士3問（G1/G2が撤退帯でない時のみ）
```

v4の「G1とG2並行」は悪くないが、**G0'欠落**が致命。

### 5.3 手直し比率25%/60%は未パイロットの魔法数

封緘は正しい。数値がRound 4で決まるのも正しい。  
ただし**測定者2名×耳コピ基準線**を取る前に帯を決めると、事後的に帯が動く。  
パイロット1曲で分散を見てから封緘、を義務化。

### 5.4 「安く勝てるか」列は観測不能に近い

誤り種別から「規則的後処理で埋まるか／モデル刷新が要るか」を分ける——言うは易し。  
**判定者スキルが未定義**。経済性ゲートと技術ゲートの接続が、実際には主観。  
最低限: 誤りログを3名（測定者2+第三者1）で独立ラベルし、一致率を出す。

### 5.5 堀（T4）と時間窓の矛盾

- 堀候補: 日本語UX・国内配布・スコアメーカー文化圏  
- しかしX上の実ユーザーはすでに**英語Webツール＋Moises＋Chordify＋Songsterr**を使っている  
- MuseScoreが無料A2Sを載せた瞬間、日本語UIだけでは堀にならない  

**堀が無いなら18–24ヶ月の時間窓内撤退設計**——v4はそう書いた。  
MuseScoreベータを踏まえれば、**時間窓の起点を2026-07-07にリセット**し、残存を半年以内と見積もるのが保守的。

### 5.6 redteam未反駁のままプランB

委員会はプランBを「条件付きで許容」するが、次を文書の冒頭に残せ。

> 本プランは教育資産とオプション価値の購入である。  
> 4ゲート全通過以外のあらゆる結果は**撤退**であり、スパイク承認の誘惑に負けない。

### 最終メッセージ

v4は「事業仮説は死んでいる。生きているのは安い崩れ方の1シナリオだけ」と正しく自認している。  
問題は、その1シナリオの検証装置が、

- **測る曲がユーザーの曲ではない**  
- **比べる相手にMuseScoreとSongsterrとMuScriptorがいない**  
- **聞く相手の見つけ方と閾値が甘い**  
- **観測対象のMuseScoreが既に製品に手を出している**

という四点で、まだ投資委員会の「実測」に達していないことだ。

プランBを続けるなら、**最初の半日で MuseScore audio2score と MuScriptor を固定3曲に通せ。**  
そこで手直しが軽いなら、残りのゲートは実施せずクローズせよ。  
それがプランBを儀式にしない唯一の方法である。
