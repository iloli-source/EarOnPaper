## (2) 法務線引きの穴

v3の最大の穴は、「タイプ2クラウド」と「AI採譜変換」を同じ箱に入れている点です。文化庁報告書はロッカー型クラウドの4分類を扱っており、タイプ2は利用者持込コンテンツのプライベート・ユーザーアップロード型です。報告書自体も、サービス態様は個別具体的に評価するしかないとしています。

一方、AI採譜SaaSは単なる保存ではなく、音源を解析し、楽譜・MIDI・MusicXMLという別表現へ変換するサービスです。文化庁報告書では「メディア変換サービス」を別類型として検討対象に挙げています。 ここをまたぐなら、v3は「なぜ採譜変換がメディア変換ではなくタイプ2なのか」を説明できていません。

さらにロクラクII最高裁は、ユーザーが録画指示をしても、事業者が管理支配下で入力・複製の枢要行為を担えば、事業者が複製主体になり得るとしました。 サーバー側で音源を受け、モデルを走らせ、MusicXML/PDFを生成するなら、このリスクはかなり近い。

自炊代行も、上告不受理で原告勝訴が確定しています。 つまり「ユーザーが持ち込んだ」「ユーザーのためにやった」だけでは足りません。v3の法務条件は最低でもこう修正すべきです。

- ローカル実行のみ
- サーバー変換なし
- YouTube等URL取り込みなし
- 公開リンク生成なし
- 人手校正・納品代行なし
- ユーザー音源・生成譜面を学習利用しない、少なくともデフォルト学習利用しない
- 教師・教室・法人利用は別扱い

SongscriptionはFAQ上、アップロード音源・採譜結果をAI改善に使うことがあり、Plus/Proはオプトアウト可能としています。 これと「成果物非関与・ソフト提供のみ」は同居しにくいです。

## (3) 参入制約7つの矛盾

矛盾があります。

一番大きいのは、**#4 法務は成果物非関与** と **#5 買い切り+サブスク並走** です。Webサブスクをやるほど、サーバー処理・アカウント保存・再エクスポート・共有・学習利用が入りやすい。これは法務上の「非関与」と反対方向です。

次に、**#3 MusicXML/MIDI疎結合** と **#7 技術的勝負所は整譜** も弱い。整譜が価値なら、ただMusicXMLを吐いてMuseScoreで直してもらうだけでは価値を取り切れません。MuseScore自身もNoteVisionでOMRを自社投入し、1日1,000件超・前年比130%増のアップロード処理を公表しています。 プラットフォーム側に飲まれる筋が濃い。

さらに、**#6 月$15k MRR上限** と **#7 独自データ戦略必須** が経済的に噛み合いません。人間校訂MusicXML、ジャンル別音源、権利処理、評価者確保までやるなら、月$15k上限の事業で回収できる設計ではない。ASAPのような近い公開データも、222譜面規模でクラシックピアノ中心です。

## (4) 実機ベンチの科学的妥当性

「手直し15分」は投資判断のスモークテストとしては有用ですが、科学的ベンチではありません。理由は明確です。

- 15分が任意。v3本文では「30分以内で納品可能」を測ると言いながら、撤退条件は「15分」にすり替わっている。 [conclusions-v3.md](<repo>/docs/research/conclusions-v3.md:83)
- 曲数9曲では少なすぎる。ジャンル、録音品質、楽器、曲長、拍子で分散が大きい。
- 手直し時間は、AI品質だけでなく、編集者の熟練度、使用エディタ、曲への既知性に左右される。
- 「納品可能」の定義がない。音高、リズム、拍子、小節割り、連桁、休符、声部、TAB、コード、演奏可能性を別々に採点すべき。
- 競合のどれか1社が1ジャンルで15分以内なら即撤退、は過剰。標的セグメントを決めずに全体撤退条件にしている。

最低限、ベンチはこう直すべきです。

1. 事前に対象セグメントを1つに絞る。例: 日本語ポップ弾き語り、またはソロピアノ。
2. 曲長あたりの修正時間、例 `分/音源1分` に正規化する。
3. 修正前はmir_eval系のOnset/Offset/Pitch F1で機械評価する。mir_evalは採譜評価で音符一致・F値・オーバーラップを扱う。([mir-eval.readthedocs.io](https://mir-eval.readthedocs.io/latest/api/transcription.html))
4. 修正後は第三者2名以上でブラインド評価する。
5. 撤退条件は「対象セグメントで、主要競合2社以上が、品質基準を満たし、中央値で手直し時間が自社目標以下」にする。
