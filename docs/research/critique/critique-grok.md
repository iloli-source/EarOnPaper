## 総評（1段落で済ませる）

この結論は「ピアノの音高はだいたい取れるらしい」「整譜が大変らしい」「ココナラに人がいるらしい」「Piascoreがあるらしい」を、**因果とTAMとユニットエコノミクスなしに「勝ち筋」へ接続**している。

## A. 技術結論への批判  
### 「音高検出は実用域（特にソロピアノ）」— 定義の詐術

**何をもって「実用域」か未定義。** ここが一番たちが悪い。

| 計測レイヤ | 典型的な話 | 顧客が買うもの |
|---|---|---|
| frame / note F1（ベンチ） | ピアノで note-level F1 80%前後などが論文で報告 |  quantize・拍節・声部分離・運指・可読譜面ではない |
| 演奏可能スコア | リズム・拍子・小節線・声部・装飾・pedal | **これ** |
| 商用「聴いて弾ける」 | 人間が直した完成譜 | 代行市場の現実 |

研究側でも、SOTAピアノでも **frame 90%超でも note-level 80%前後**といった層差が明示されている。辛口レビューのため、結論の根拠を潰す反証をWeb・Xから集めます。# 投資委員会向け 辛口レビュー  
## AI採譜プロジェクト「現在の結論」への反証・棄却意見

**結論を先に言う。**  
この「現在の結論」は投資判断に使える水準に達していない。論理の飛躍、指標の取り違え、法務の単純化、そして**実測ゼロの断定**が重なっている。いま承認すれば、金と時間を「正しそうに見える仮説の実装」に溶かす可能性が高い。

## 0. まず潰すべき前提：F（調査手法）

> デスクリサーチのみ（X・GitHub Issues・Webレビュー）で A〜E を「結論」にした

これは調査報告ではなく**読み物**である。

| 問題 | なぜ致命的か |
|------|----------------|
| **検証プロトコル不在** | onset F1、meter accuracy、手直し時間（分/曲）、納品可率など、投資に必要なKPIが一切ない |
| **生存者バイアス** | X/レビューは「動いた」「がっかりした」が極端に残りやすい。中間層（静かに解約）は見えない |
| **宣伝混入** | ベンダー比較記事・公式ヘルプ・アフィ系「Best of 2026」が一次情報と同等に扱われやすい |
| **GitHub Issuesの偏り** | 使っている人のバグ報告＝市場需要ではない。需要があるなら有料ユーザーとNPSが要る |

**投資委員会の立場：** 実機ベンチ（同一音源セット×複数商用×人手採譜時間）がない段階で「実用域」「勝ち筋」「有望」と書いた時点で、調査の独立性を疑う。  
Fが崩れている以上、A〜Eはすべて**仮設のまま格下げ**されるべきである。

## 1. A. 技術結論への批判

### 1.1 「音高検出は実用域（特にソロピアノ）」——条件付きの自明を誇張している

- ソロピアノは AMT 研究史上もっともデータが揃い（MAESTRO等）、**もっとも易しいドメイン**である。ここで「まあまあ動く」は差別化でも機会でもない。
- 「実用」の定義がない。  
  - 研究指標で F1 0.8？  
  - 演奏者がそのまま弾ける？  
  - 採譜代行者が手直し30分以内？  
  この3つは別物である。調査はそれらを混同している。
- 実世界音源（YouTube圧縮、残響、テンポ揺らし、ペダル、重なり和音、アコースティック以外）への**劣化曲線**が示されていない。クリーン録音での「実用」は投資価値の過大評価になりやすい。

### 1.2 「ボトルネック＝リズム・整譜」——因果が逆かもしれない

調査は「音高はOK → 残りは後処理」という直線物語を採用している。だが現場の不満はしばしばこうだ：

- 拍がずれているのではなく、**そもそも何が演奏意図の音符か**が曖昧（装飾・ロール・ペダル・倍音・アタック弱い音）
- 整譜は「検出誤差の化粧」であり、**上流の誤りを消えない**
- 商用ですら「notes incorrect / sections completely wrong」「utterly useless」系の声が残る。Klangio は比較記事でも「品質より網羅・連携」と評価されることがある。

つまり「整譜さえすれば勝てる」は**未検証の因果仮説**である。  
ボトルネックが整譜だとしても、それは「勝てる場所」ではなく「**誰も完全には解けていない地獄**」かもしれない。

### 1.3 「マルチ楽器・ドラムは未成熟」——正しいが、事業的には罠

未成熟を「参入余地」と読むのは早計。  
未成熟＝**ユーザーが金を払わない／払っても解約する領域**である可能性が高い。  
さらに 2026-07 に Kyutai/Mirelo の **MuScriptor**（オープンウェイトの多楽器→MIDI）が公開され、マルチ楽器の**基礎レイヤは急速にコモディティ化**している。調査が「未成熟」と止めた地点は、すでに動いている。

## 2. B. 「OSS空白＝勝ち筋」——投資委員会が最も嫌う飛躍

### 2.1 空白は「機会」ではなく「死体置き場」であることが多い

論理：

> 整譜は商用（Klangio Arrangement Mode 等）が先行し、OSSが空白 → そこが最大の勝ち筋

**反転解釈（こちらの方が Occam）：**

1. 整譜はドメイン知識が重く、汎用OSSメンテナが食えない  
2. 価値がユーザー側の**最終エディタ（MuseScore / Dorico / Sibelius）**に回収される  
3. 単体プロダクトとしてARPUが立たない  
4. だからOSSが空いている  

「誰もやっていない」≠「やれば勝てる」。  
誰もやっていない理由の検証（支払意思・競合の利益率・解約理由）がない。

### 2.2 「空白」そのものが過大評価

- MIDI→MusicXML、量子化、表記変換は **music21 / MuseScore import / 各種変換ツール**で部分的に存在する。  
- 空白なのは「完璧な演奏可能楽譜への自動整譜」であり、それは**未解決の研究課題**であって、スタートアップが6ヶ月で埋める穴ではない。
- Klangio の Arrangement Mode は「正確な採譜」ではなく、難易度付きの**アレンジ生成**である。これを「整譜の先行」と同一視するのはカテゴリ誤認。アレンジと忠実採譜は権利も評価軸も違う。

### 2.3 勝ち筋が「後処理」なら、参入障壁はほぼゼロ

後処理をOSSで公開すれば：

- 既存の Basic Pitch / MT3 系 / MuScriptor 系の上に誰でも載る  
- 商用各社も同じ後処理を真似できる  
- **データ・ブランド・配信許諾・DAW/API導線**を持たない後発が勝てない  

調査は「技術的空白」と「経済的モート」を混同している。

### 2.4 モデル進化による陳腐化リスク（調査自身が挙げるMuScriptor問題）

MuScriptor は推論コード MIT、重みは CC BY-NC 等で商用制限があるが、方向性は明白：

- 多楽器MIDI化の**公開ベースラインが上がる**  
- 自前エンジン開発のROIが下がる  
- 差別化は「モデル」ではなく**UX・許諾・信頼性・人間品質保証**に移る  

にもかかわらず調査の勝ち筋は「整譜OSS」。  
これは**基礎モデルが上がるほど相対価値が下がる層**に張る戦略になりかねない。

## 3. C. 日本市場仮説のデータの弱さ

### 3.1 「カテゴリ実績5.1万件」は需要指標ではない

ココナラの件数は典型的に：

- **出品数（供給）**または累計表示の一種  
- GMV、成約率、リピート率、客単価中央値、返品・クレーム率ではない  

5.1万は「市場が熱い」ではなく「**供給過多・価格競争が起きやすいスキル市場**」のシグナルにも読める。

### 3.2 「1曲2,000〜3,000円」は事業機会というより赤旗

- 低価格帯は**副業・スキル練習・客寄せ**価格の混在が常態  
- ここをAIで効率化するなら、理論上は単価下落圧力がさらに強まる（供給曲線が右にシフト）  
- ハイブリッド（AI下書き→人手仕上げ）は、**人件費が残る**。AIが8割やっても、残2割が難しい箇所なら工数半減にならない（80/20の逆）

### 3.3 ハイブリッド代行は「有望」ではなく「誰でも明日始められる」

- フリーランスは既に AnthemScore / Klangio / 無料モデルを下書きに使える  
- プラットフォーム側（ココナラ）の手数料・価格競争・レビュー経済が支配  
- 自社が勝つ理由（集客、品質保証、権利処理、ブランド）が調査にない  

「市場が存在する」≠「あなたが取れる」。  
取れるのは**既存の上位出品者**か、**出版社/教育機関のB2B**であり、2,000円帯の延長線ではない可能性が高い。

### 3.4 TAMの不在

- 年間の有料採譜需要（曲数×単価）  
- うち「著作権クリアで合法に回せる」割合  
- SaaS課金に転換可能な層（自分で直せるユーザー）の比率  

これがないのに「有望」はマーケティング用語。

## 4. D. 著作権——最大級の過小評価

調査の整理：

> 採譜自体は私的複製なら許諾不要。配布・販売は許諾必須。Piascore等が使える。

これは**個人の耳コピメモ**の話としては概ね通じるが、**プロダクト/事業の法務**としては危険なまでに薄い。

### 4.1 「私的複製」はSaaSの免罪符にならない

- 著作権法30条の私的使用は、**個人的・家庭内等の限られた範囲**での使用を想定した権利制限。  
- クラウド上で第三者が処理するサービス、代行納品、共有リンク配布は、単純に「ユーザーの私的複製」では片付かない領域に入る。  
- 代行ビジネスは「依頼者の私的利用目的」をうたっても、**業としての複製・提供**として別評価されうる。  
- 調査は「私的ならOK」を前面に出し、**サービス設計上のグレーゾーンを解消したように見せている**。これは法務DDで即指摘される。

### 4.2 アレンジと採譜の混同

- Arrangement Mode 的な「易しいピアノアレンジ」は、忠実採譜ではなく**編曲（翻案）**に寄りやすい。  
- 日本法では編曲権（27条）・同一性保持の論点が出る。  
- Piascore自身も、編曲・替え歌等は**あらかじめ権利者許諾が必要**と明示している。  
  「Piascoreがあるから安心」は**逆**で、Piascoreは管理楽曲の販売導線であり、編曲の自由許可証ではない。

### 4.3 Piascoreの経済性・制約を無視している

- 販売主取り分は権利状況で大きく変動（オリジナル高め、管理曲・外国曲で低下）  
- 管理楽曲は販売地域制限などの実務制約  
- 手数料・著作権料控除後の**実質粗利**が、2,000円代行や安価PDF販売と両立するか未計算  

「スキームがある」≠「事業として回る」。

### 4.4 見落としがちな権利の束

| 権利・論点 | 調査の扱い | 実態リスク |
|-----------|------------|------------|
| 複製権 | 配布時のみ注意 | サーバー生成・キャッシュ・学習用途でも論点 |
| 編曲・翻案 | ほぼ無視 | 難易度調整アレンジで直撃 |
| 原盤・実演（隣接権） | 触れず | 音源アップロード型で追加リスク |
| 学習データ | 触れず | 自前学習するなら炎上・契約リスク |
| プラットフォーム規約 | 触れず | YouTube URL入力系は別途利用規約違反リスク |

**法務を「配布時に許諾」の一文で閉じた時点で、この調査は事業化準備として不合格。**

## 5. E. 技術構成候補——時期尚早のアーキテクチャ・コスプレ

> モジュラーモノリス＋エンジン別コンテナ＋ジョブキュー ＋ TDD/DDD/クリーン/EDD/issue駆動

### 批判の要点

1. **プロダクト仮説未検証の段階で、運用コストの高い分離構成を先に選んでいる**  
   - まず測るべきは「どのエンジンがどの音源で何分の手直しを減らせるか」  
   - コンテナ分離は、それが分かってからでよい  
2. **手法の羅列はリスクヘッジに見えて中身がない**  
   - TDDもDDDもクリーンも有用だが、同時並記は「真面目にやる感」の演出  
   - EDD（評価駆動）こそ最優先なのに、調査自体が評価データセットを持っていない——自己矛盾  
3. **採用技術の話で投資を通そうとしている**  
   - 委員会が買うのはアーキテクチャではなく、再現可能な優位性

この節は「エンジニアが作りたいものリスト」であり、事業結論を補強していない。

## 6. 観点別の総括（要求された6視点）

### (1) 論理の飛躍・因果の取り違え

| 調査の主張 | 飛躍 |
|-----------|------|
| OSS空白 → 勝ち筋 | 空白＝低収益の可能性を無視 |
| 音高OK → 残り整譜 | 上流誤差と下流整形を分離できていない |
| ココナラに案件あり → ハイブリッド有望 | 供給指標を需要・自社TAMに読み替え |
| Piascoreあり → 権利問題は実務的に解ける | 編曲許諾・私的複製の境界を過小評価 |
| 手法が揃っている → 実行可能 | 実行可能性≠投資合理性 |

### (2) データの弱さ・サンプルバイアス

- n数・抽出条件・除外基準なし  
- 商用レビューは二極化（「思ったより良い」vs「utterly useless」）で、**中央値と分散**がない  
- ココナラ件数の定義曖昧  
- 単価レンジのみで粗利・CAC・LTVなし  

### (3) 見落とし・反証になりうる事実

- **Songscription / ScoreCloud / AnthemScore / Ivory** 等、整譜・表記・編集を含む競合は既に複数。Klangio一強ではない。  
- ScoreCloud は「黒箱検出＋音楽認知モデルで拍・フレーズを解釈」を差別化として打ち出しており、「整譜空白」物語と矛盾する。  
- **MuScriptor** により多楽器MIDIの公開性能が跳躍。調査の「マルチ未成熟→後でやる」タイムラインは短縮されている。  
- プロ実務では「AIは掃除前提、合奏は耳コピの方が速い」系の評価が残る。自動化が常に優位とは限らない。  

### (4) 事業リスク

- **市場：** 小さなスキルマーケットの価格破壊帯に寄るとスケールしない  
- **ユニットエコノミクス：** GPU推論＋人の仕上げ＋プラット手数料＋権利コストで、2,000〜3,000円は即死ラインになりうる  
- **陳腐化：** 基礎転写のOSS/研究が進むほど、自前モデルの価値は下がる  
- **参入障壁：** 後処理アルゴリズム単体にモートなし。許諾網・出版社関係・教育機関契約の方が障壁  
- **競合反応：** 既存商用はAPI・DAW・モバイルを既に持つ。後発OSSが奪えるのは「無料ユーザー」中心になりやすい  

### (5) 法務リスクの過小評価

前述。特に **SaaS×著作物音源×アレンジ生成×代行納品** の組み合わせは、調査が扱うより重い。

### (6) 実機検証ゼロで結論を出す危険

- 偽陽性：ベンチマーク音源では良く、顧客音源で崩壊  
- 偽陰性：整譜の価値を過大/過小評価  
- 開発ロードマップが「物語」にロックされ、反証データを後から拒否する組織病理  
- 投資家・社内向けに「もう結論が出た」空気が作られ、**kill すべき案件が生き残る**

## 7. この結論のまま進めた場合に、最も高くつく間違いトップ3

### 1位｜「ボトルネックは整譜」を所与にフルスタック開発する（数百万〜数千万円＋半年）

実測なしで後処理に張り、音源多様性で破綻。  
結果：デモはピアノの綺麗な曲だけ、本番は手直し地獄。  
**最も高いのは「作ってから気づく」機会損失。**

### 2位｜「OSS空白＝モート」で後処理/整譜にブランドを賭け、基礎モデル陳腐化に直面する

MuScriptor系の公開進化と商用のArrangement/編集機能の間で潰される。  
OSSを出しても、収益は編集ソフト・プラットフォーム・権利者に流れ、自社に残らない。  
**技術的に正しくても、株主ホルダーとして負ける。**

### 3位｜著作権を「私的複製／配布時許諾／Piascore」で済ませた事業設計（代行 or SaaS）

- 代行：グレー運用が常態化し、スケールした瞬間に権利者・プラットフォームから叩かれる  
- SaaS：ユーザーが著作物を上げ、生成物を共有・販売する二次利用を止めきれない  
- アレンジ系機能：編曲許諾をすっとばすと停止・返金・信用毀損  

法務・事業停止・ブランド毀損は、エンジニアリング負債より高い。

## 8. 投資委員会としての判定

| 項目 | 判定 |
|------|------|
| 調査の投資適格性 | **不合格**（Fが致命傷） |
| A 技術結論 | **保留〜弱気**（ソロピアノは自明域、因果未検証） |
| B 勝ち筋 | **棄却寄り**（空白＝機会の誤謬、モートなし） |
| C 日本市場 | **証拠不十分**（供給指標の過大解釈） |
| D 著作権 | **重大な過小評価** |
| E 技術構成 | **時期尚早**（検証基盤が先） |

### 承認条件（これを満たすまで予算ゼロが妥当）

1. **公開可能な評価セット**（ジャンル×音質×編成）で商用3つ＋人手の手直し時間を実測  
2. 「整譜モジュール単体」の**限界効用**（手直し分数の削減幅と分散）を示す  
3. ココナラ等は出品数ではなく、**成約・粗利・リピート**の一次データ

5. ユニットエコノミクス：推論コスト＋人件費＋手数料＋権利コスト ＜ 支払意思価格
