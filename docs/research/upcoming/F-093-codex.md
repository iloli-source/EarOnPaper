# F-093 人手仕上げ作業パッケージ出力 調査メモ（論文＋WEB）

対象: 採譜 / Pitchsieve の「人手仕上げ作業パッケージ出力」機能。自動採譜の結果を外部（プロ）採譜者に引き継ぐための一式で、(1) 区間音源、(2) 下書き MusicXML、(3) 信頼度ハイライト（自動採譜のどこが不確かかを標示）から成る。

調査日: 2026-07-21
調査手段: mcp__codex__codex（read-only, gpt-5系）＋ WebSearch による裏取り。
検索方針: 論文・技術文献優先、英語・中国語中心。実在 URL のみ。捏造禁止（確証のない引用は本文で明示）。**失敗例、とりわけ「信頼度をどう人間に伝えるか」の失敗を最大化**して列挙する。

---

## 結論（先に）

1. **最大リスクは MusicXML の変換損失ではなく、信頼度表示が人間の注意配分を誤誘導すること。** ASR/HCI 文献では、低信頼度ハイライトは「当たった誤り」の発見率を上げる一方、「外した（ハイライトされなかった）誤り」の見落としを増やし、全体の速度・精度を改善しないと繰り返し報告されている（Vertanen & Kristensson 2008、Kuhn et al. 2025）。
2. **下書き MusicXML 単体を納品物にするのは危険。** MusicXML は「共通西洋記譜の交換形式」であって「完全保存形式」ではない。Library of Congress が明示的に "round-trip import/export should not be expected" と述べており、タイ・連符・移調・声部・レイアウトが開くソフト（MuseScore/Finale/Sibelius/Dorico）で崩れる。
3. **信頼度は `p(correct)` と呼んではいけない。** AMT の frame/note posterior は過信（miscalibration）しやすく、しかも「音価・声部・拍節の正しさ」を表さない。confidence は単一スカラーでなく `pitch/onset/offset/duration/voice/tuplet/alignment` の軸に分解すべき。
4. **区間音源は切り方で人を誤誘導する。** 小節境界ぴったりで切ると、アタック・先行装飾・ペダル残響・タイの文脈が消え、逆に「採譜が間違って見える」。時刻原点（clip ↔ MusicXML ↔ MIDI ↔ 波形）の不一致は致命的。
5. **良いデータセット（MAESTRO/GuitarSet/Slakh 等）は、音だけでなく対応関係・由来を丁寧に保存している。** パッケージは単一 XML ではなく複数 annotation layer + provenance を持つべき。

## 1. MusicXML 交換・ラウンドトリップ失敗

MusicXML は交換形式として成功しているが、ネイティブ譜面の完全保存形式ではない。Michael Good の "MusicXML: An Internet-Friendly Format for Sheet Music" は、Finale/Sibelius 等の構造に合わせた**交換**形式として設計した経緯を述べる。Library of Congress のフォーマット記述は「全ソフトが全機能を支持するわけではなく、round-trip import/export should not be expected」と明記している。

### 実務コミュニティで観測される破綻（WebSearch 裏取り済み）

- **ベンダー間で生成 XML が別物**: 「MusicXML generated by Sibelius differs from Finale, and the MusicXML from MuseScore and Notion are all different, often quite different」。importer 側が差異を吸収しきれず、開いた後に必ずクリーンアップが要る（VI-CONTROL "MusicXML the state of play in 2023/4"）。
- **オクターブシフト（ottava）**: Sibelius 由来の octave-shift を MuseScore が誤ってインポートする既知 issue（musescore/MuseScore #24711）。
- **連符（tuplets）**: triplet が import を壊す事例（Dorico）。回避には exporter が `<divisions>` を「連符を含む全音価の最小公倍数」で定義する必要がある。
- **バージョン非互換**: Sibelius は MusicXML 4 を主に Dolet 経由で扱い、import 時に 3 / 3.1 へダウングレードが要る場合がある。
- **Finale の "disjointed" import**: MakeMusic 公式ヘルプに、MusicXML import が highly "disjointed" になるという相談スレがある。

### 仕様レベルで壊れやすい要素（Codex 整理）

| 要素 | 失敗機序 |
|---|---|
| tie / tied | MusicXML では `<tie>` が音響上、`<tied>` が視覚上のタイ。片方だけ正しいと「再生はつながるが譜面上は切れる」等が起きる（W3C MusicXML 4.0 Notation Basics）。 |
| tuplets | `<time-modification>`（音価比）と `<tuplet>`（表示）の分離。AMT 由来の粗い量子化から作ると普通音価に崩れる。 |
| divisions | 四分音符あたりの整数単位。低すぎると三連・五連・装飾音が表現不能、高すぎると互換・編集性が落ちる。 |
| key / transposition | `<transpose>` は written→sounding の差分。Cl/Sax/Hr で concert/transposed の意図がメタデータにないと実音・記譜音がずれる。 |
| voices / staves / order | ポリフォニックパートでは文書順と音楽順が食い違い得る。時刻順で単純 XML 化すると声部・符尾・休符補完・クロススタッフが壊れる。 |
| layout / slur / text | レイアウトは持てるが出版品質の浄書やソフト固有のテキストボックスまでは安定しない。 |

**Pitchsieve への含意**: handoff XML は「最終譜面」ではなく「機械可読な下書き」と明記し、manifest に `source_app`, `musicxml_version`, `concert_or_transposed`, `quantization_grid`, `divisions`, `voice_assignment_confidence` を入れる。MusicXML 3.1 系での書き出しを既定にし、Dorico/Sibelius の import 制約に合わせる。

## 2. AMT 信頼度・キャリブレーション失敗

Onsets and Frames（Hawthorne et al., ISMIR 2018）は onset/frame detector の sigmoid 出力を使い、推論時に 0.5 閾値で判定する。しかしこれは「採譜者に見せてよい校正済み確率」ではない。Guo et al. "On Calibration of Modern Neural Networks"（ICML 2017, arXiv:1706.04599）は、現代 DNN の確率出力がしばしば過信的で temperature scaling 等の後処理を要すると示す。

AMT 固有の失敗:

- **posterior が note correctness を表さない**: フレーム単位で高確率でも、音価・声部・連符・拍節上は誤り得る。
- **onset は高信頼、offset は低信頼になりやすい**: ピアノでは減衰・ペダル・重音で offset が難しい。
- **評価指標と人間判断がずれる**: Ycart et al. "Investigating the Perceptual Validity of Evaluation Metrics for Automatic Piano Music Transcription"（TISMIR）は、AMT 指標と人間の聴感が、似たスコア同士や低品質同士で一致しにくく、リズム次元が重要だと示す。
- **モデル間一致も過信できない**: PiJAMA（TISMIR）は 2 つのピアノ転写モデルの agreement を弱い信頼度信号に使うが、同種の訓練データ・アーキテクチャは octave errors 等の**共通誤り**を共有し得ると警告する。
- **ドメイン外で崩れる**: MT3（Gardner et al.）はマルチタスク AMT の成果と同時に、評価指標・dataset alignment の一貫性不足を課題に挙げる。クラシックピアノで校正した confidence を、歌・ギター・民族楽器・バンド音源に移植するのは危険。

**Pitchsieve への含意**: confidence を `p(correct)` と呼ばず `model_uncertainty_signal` として扱う。note-level だけでなく `pitch_conf`, `onset_conf`, `offset_conf`, `duration_conf`, `voice_conf`, `meter_quantization_conf`, `alignment_conf` に分けて出す。

## 3. 信頼度ハイライトの「人間への伝達」失敗（最重要）

ASR 文献が最も直接的で、そのまま音楽採譜に効く。

- **Vertanen & Kristensson, "On the Benefits of Confidence Visualization in Speech Recognition"（CHI 2008）**: 低信頼度の下線/色/不透明度表示は、**全体の速度・精度を改善しなかった**。正しく下線が付いた誤りは見つけやすくなる一方、**下線が付かなかった誤りは見つけにくくなる**。ハイライトは注意を移すだけで、漏れた誤りを不可視化する。
  - PDF: http://pokristensson.com/pubs/VertanenKristenssonCHI2008.pdf
- **Kuhn et al., "Evaluating ASR Confidence Scores for Automated Error Detection in User-Assisted Correction Interfaces"（arXiv:2503.15124, CHI EA 2025）**: 最新 E2E ASR の confidence は単語誤り検出に中程度以下で、ハイライトは訂正効率を改善せず、参加者にも有用と評価されなかった。閾値を上げると false positive、下げると false negative が増える。
- **eyes-free 文脈での逆の示唆**: "Using Confidence Scores to Improve Eyes-free Detection of Speech Recognition Errors"（arXiv:2410.20564）は、視覚が使えない状況では confidence 提示が誤り検出を助け得るとする。→ **提示モダリティと作業状況で効き方が反転する**。音楽採譜は視覚併用なので Vertanen 側の警告が優勢。

HITL/annotation 側の失敗:

- **automation bias / anchoring**: 事前アノテーション（pre-annotation）はモデル提案側にラベル分布を寄せる。annotator は速くならないのに自信だけ上がる、というパターンが繰り返し報告される（Schroeder et al. 系、Bias in the Loop = arXiv:2509.08514）。
- **訂正コスト設計の罠**: AI 抽出値の誤り修正で「正しい値の入力」まで要求すると、参加者が誤りを受け入れやすくなる（Beck et al. 系）。
- **信頼度提示が信頼を下げる場合**: HITL フィードバックを求めるだけでユーザーの信頼と精度認知が下がる（Honeycutt et al., HCOMP 2020）。
- **能動学習の分布偏り**: 「最も不確実なサンプルだけ」を人間に回すと、データ分布の狭いスライスに収束し重要だが希少なケースを取り逃す（diversity-aware sampling が必要）。

**Pitchsieve への含意（設計原則）**:
- 低信頼度を**赤一色で塗って「間違い」扱いにしない**。表示は「この音が怪しい」ではなく「この判断軸が未確定: pitch/onset/offset/voice/tuplet/alignment」と分解する。
- ハイライトは lattice/候補併記型にする（Vertanen 系の反省）。「赤い不安マーク」単独より、候補音高・候補拍位置・根拠音声（区間再生）を併記した方がよい。
- **非ハイライト部分にも「未検査の誤りが残る」と明示**し、抜き取り検査を促す。
- 「モデル間不一致」「alignment 不安定」「量子化不安定」「MusicXML 変換損失」を confidence とは別レイヤーにする。

## 4. 区間音源・stem handoff 失敗

- **onset 前後の文脈喪失**: Onsets and Frames は訓練分割で active note を切ると onset detector が onset を見失うため、長め（例 20 秒）分割や zero-crossing を考慮している。handoff clip も小節境界ぴったりで切ると、アタック・先行装飾・ペダル残響・タイの文脈が消える → pre-roll / post-roll を必須化。
- **alignment drift**: score-performance alignment は誤った local minimum に落ちることがあり、manual bootstrapping / anchor points が cascading errors を防ぐ（Meron & Hirose; Peter et al. nASAP, TISMIR）。
- **tempo map 不備**: clip・MusicXML・MIDI・波形表示の時刻原点が不一致だと、採譜者は「音は合っているが譜面位置が違う」に陥る。
- **無音トリム/encoder delay で全部ずれる**: 先頭パディング・ノーマライズ・mp3 encoder delay で原点が変わる。offset をメタデータ化し、clip と note JSON の duration 差を検証・警告する。
- **stem の偽精度**: Slakh のような合成 stem/MIDI は完全整列しやすいが、実録音の bleed・reverb・rubato・mix 処理とは違う。**分離アーティファクトを confidence に混ぜない**。

**Pitchsieve への含意**: 各 clip に `sample_start`, `sample_end`, `audio_sha256`, `sr`, `time_origin`, `bar_range`, `beat_range`, `tempo_map`, `click_track`, `pre_roll_ms`, `post_roll_ms`, `alignment_confidence` を付ける。

## 5. MIR/音声の handoff・annotation package 設計

良いデータセットは、音だけでなく対応関係と由来を保存している。

- **MAESTRO**: 約 200 時間のピアノ audio/MIDI、約 3ms 精度 alignment、composer/title/year/split メタ。v3 で誤混入した弦楽四重奏伴奏つき録音を削除 → **データセット自体も後日修正が要る**という実例。
- **MusicNet**: スコアを録音へ warping してラベル化 → 「score-derived annotation は近似」と扱うべき。
- **GuitarSet**: hexaphonic pickup で弦別信号を取り、pitch contour, MIDI note, string/fret, beat, tempo, chord を JAMS に保存 → **単一 XML でなく複数 annotation layer** が要る。
- **Slakh**: stems + aligned MIDI + metadata。合成音源・重複バグ・loudness normalization など生成過程メタが重要。
- **PipaSet / TEAS**: 中国伝統撥弦楽器 pipa 向けに audio/notation/video/optical sensor + annotation platform。非西洋・奏法依存の採譜では**音高以外の奏法・表情注釈が不可欠**。
- **M4Singer**: Mandarin singing corpus で musical scores を手作業整備。歌唱では歌詞・音節・音高・譜面の対応が handoff に必要。

## 推奨パッケージ仕様

最小 bundle:

- `clips/`: WAV/FLAC、pre-roll/post-roll 付き、click あり/なし両方
- `draft.musicxml`: 下書き（final でないと metadata 明記、MusicXML 3.1 既定）
- `draft.mid`: timing 確認用。MusicXML と同一 note id を共有
- `confidence.json`: note id ごとの pitch/onset/offset/duration/voice/tuplet/alignment confidence
- `alignment.json`: sample time ↔ beat ↔ measure ↔ MusicXML note id
- `review.html` または DAW/譜面ソフト用プロジェクト: 低信頼度箇所・候補・根拠音声へジャンプ
- `README.md`: 既知の弱点・推奨確認順・ソフト互換注意
- `provenance.json`: model version, thresholds, calibration set, source separation 有無, quantization settings

ハイライト設計:

- 赤一色で「間違い」扱いしない
- 低信頼度以外に「モデル間不一致」「alignment 不安定」「量子化不安定」「MusicXML 変換損失」を別レイヤー化
- ハイライトされていない箇所にも抜き取り検査を促す
- 採譜者の修正理由を `correction_type` で残す: pitch, rhythm, voice, notation, articulation, layout

### 実装時チェックリスト

- MusicXML を MuseScore / Finale / Sibelius / Dorico の 4 種で開いて、tie/tuplet/移調/声部/ottava が壊れないか回帰確認しているか。
- confidence を単一スカラーにせず軸分解しているか。`p(correct)` と誤称していないか。
- 低信頼度ハイライトが「非ハイライト＝正しい」という誤読を生まないよう、未検査注意を出しているか。
- clip の時刻原点（無音トリム/encoder delay）を記録し、note JSON と duration 検証しているか。
- pre-roll/post-roll でアタック・ペダル・タイ文脈が残っているか。
- 合成 stem のアーティファクトを confidence に混ぜていないか。
