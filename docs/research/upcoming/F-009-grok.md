# X調査レポート：音楽採譜における「解析ヒント入力」  
（テンポ・キー・楽器・拍子・チューニング・カポの事前指定）

**調査日:** 2026-07-21（JST）  
**対象:** X（旧Twitter）上の実務者／開発者／研究者寄りの投稿（英語・中国語中心、日本語補完）  
**焦点:** 事前ヒントで精度が上がる／上がらない実体験、失敗例、限界、ベストプラクティス、最新トレンド  

---

## 1. 調査の前提と用語のズレ

製品機能名としての **「解析ヒント入力」** は、X上ではそのままのラベルではほぼ語られません。  
代わりに、同種の議論は次の言葉で現れます。

| 製品側のヒント | X上で近い実務語 |
|---|---|
| テンポ | BPM / beat grid / tempo map / Assign Tempo / half-time・double-time |
| キー | key detection / force to scale / scale quantization / pitch reference |
| 楽器 | instrument / stem / multi-instrument / drum mapping |
| 拍子 | time signature（直接のAMT議論は薄い） |
| チューニング | pitch reference / master tuning / half-step down / alternate tuning |
| カポ | capo + tab / 移調の不一致 |

**重要:** 「ヒントを入れるUI」そのものより、**(A) 自動推定が外れる失敗** と **(B) 推定後にスケール／量子化で補正する成功** の両方が多いです。  
「事前指定で精度が上がった」明示的な成功談は、**フル自動AMTそのものより後処理・前処理パイプライン**に偏ります。

## 2. 失敗例（重点）

### 2.1 「Audio-to-MIDIはまだ壊れている」——実務者の総括

ミュージシャン **John Maus** は、長年経っても audio-to-MIDI が「だいたいダメ」だと明言しています。Melodyne 等に統合されても品質は低く、自分のハブでは **Chordino（Sonic Visualizer）** のほうがハーモニー用途でマシだった、と回顧。

生成AI／モーショングラフ系の実務者 **Heba AI** も、「何年も audio-to-MIDI をやってきたが精度が低く、**Ableton と Logic の両方ともかなり悪い**」と述べ、比較検証の意志を表明。

→ **失敗の型:** 「DAW標準のAudio-to-MIDIをヒントなし・全自動で使う」こと自体が、実務では期待外れになりやすい。

---

### 2.2 商用採譜ソフトの「宣伝と実物の乖離」（AudioScore）

**AudioScore** について、ユーザーは「**まったく役に立たない／金の無駄**」「約束された能力に遠く及ばない」と強い否定。PhotoScore（楽譜スキャン）は動くが AudioScore は別物、と対比。レビュー要約として、

- 楽器の誤認識（1本なのに複数扱いなど）
- 出力がランダムで編集不能
- サポートが薄い
- 過剰広告

が列挙されています。

→ **失敗の型:** 自動楽器識別＋フルミックス採譜の過信。  
→ **ヒント示唆:** 楽器・パートを人間が固定しないと、推定エラーが譜面全体に連鎖しやすい。

---

### 2.3 キー自動検出の不一致（同一曲で3ツール3キー）

DJ prep アプリ開発者 **Yannick Feige（Ora DJ）** は、

> 同じ曲を Rekordbox / Serato / Mixed In Key に入れると **3つとも違うキー** が返ることがある。  
> 昔は「2つがバグ」と思っていたが違う。**キー検出は本質的に難しく、電子音楽は特に酷い。**

と報告。

→ **失敗の型:** キーを自動推定に任せると、ツール間で矛盾。  
→ **ヒント示唆:** キーの**事前指定／手動確定**は「精度向上」ではなく「推定競合の切断」として有効。

---

### 2.4 テンポの half / double 誤検出

**Shazam** の曲情報で、聴感 140BPM 級のダブステップが **79BPM** と出る、と指摘（half-time 側に落ちる典型）。

別投稿でも、AI映像ツールのビート検出が **EDM（120BPM・四つ打ち）に過学習**しており、ジャズ／クラシック／アンビエントでは **3–5フレームずれ**る、と実開発者が報告。ジャンル非対応の検出モデルが失敗の原因。

→ **失敗の型:** テンポの「2倍／半分」曖昧性、ジャンルバイアス。  
→ **ヒント示唆:** BPM の事前指定・ジャンル条件付けは、譜面の符割り・クオンタイズの土台を救う。

---

### 2.5 拍間隔の誤差が「積分」してズレる（中国語圏の技術議論）

中国語投稿では、拍間隔の計測／表現が不正確だと誤差が**逐拍累積**し、譜面は「拍間隔の積分」なので単位が BPM でも対数でも関係なく精度が本質、と指摘。**耳コピ／帯の解析でも原曲と合わない**経験として言及。

→ **失敗の型:** 局所テンポ誤差 → 長尺でグリッド全体が崩れる。  
→ **ヒント示唆:** テンポマップ／セクションBPMの事前固定は、累積誤差対策として合理的。

---

### 2.6 Ableton の「誤ノート追加」とスケール矯正の限界

**RAM DOS** は、バンドメイト録音に対し Ableton の Audio-to-MIDI（ハーモニー）が **誤ったノートを足す** ことがあり、手直しの代わりに **スケールに強制**して近似すると書いている。

→ **失敗:** 多声音の過剰検出。  
→ **部分成功だが限界:** スケール強制は「近い線」を得る近道であって、正しい採譜の保証ではない（スケール外音・転調・テンションを潰す）。

---

### 2.7 ドラム MIDI の「楽器は分かってもマップが合わない」

Mirelo のフルミックス Audio-to-MIDI に対し、ユーザーが **EZDrummer / Addictive Drums / Logic ドラム** 等へのマッピングを質問。開発側は「ドラムは consolidated MIDI stem なので **手動マッピングがまだ必要**」「ユニバーサルな標準マップはあるか？」と返す。

→ **失敗の型:** 楽器ラベル成功 ≠ 演奏可能MIDI。  
→ **ヒント示唆:** 楽器指定に加え、**ドラムマップ／音色セット指定**がないと実務では未完。

---

### 2.8 ポリフォニー・フルミックスの本質的難しさ

**Liz Zhang** は「音楽転写はまだ Whisper 級の転換点を欠く。**ポリフォニーが本丸**で、n 本の楽器が同じスペクトルを奪い合う」と述べ、密なジャズ録音で検証予定と宣言。

MIR 系アカウント **Orchestral Pit** は、**PitchBench** で audio-language モデルが **基本的なピッチ聴き取りに失敗**するベンチ結果を共有。

→ **失敗の型:** 事前ヒントなしのフルミックス多楽器は、研究でもまだ脆い。

---

### 2.9 チューニング／基準音を無視した「正しい音程操作」が破綻

**Celemony（Melodyne）** は公式に、

- ノートを触る前に **正しい pitch reference（基準音）** を確認せよ  
- ボーカルをチューニングする前に **曲の master tuning** を確認せよ  

と繰り返し警告。

→ **失敗の型:** A440 前提で解析→全体が「正しいが原曲とズレる」。  
→ **ヒント示唆:** チューニング事前指定は採譜・修正の前提条件。

---

### 2.10 カポ／変則チューニングと「譜面／TABの不一致」

ギター側の実務・文化的失敗例：

- TABサイトが **「4フレット・カポで原曲キーに合わせろ」** と指示 → 低音域の歌手が困る、という皮肉投稿。  
- **Frank Iero** は「スタンダードなら2フレット・カポ、**自分は半音下げなので3フレット**」と、チューニングとカポの相互作用を説明。  
- 風刺記事タイトル級の定番失敗: *「TABはだいたい正しいが、テンポ・チューニング・音選びだけ違う」*。  

→ **失敗の型:** 聞こえた音高をそのまま fret に落とすと、**カポ／半音下げ／ドロップD** で「弾ける譜面」にならない。  
→ **ヒント示唆:** カポ・チューニングは「精度」以前に **記譜系（concert pitch vs 実フレット）** の選択問題。

---

### 2.11 パイプライン後段の「MIDIは出たがリズムがダメ」

中国語の実務パイプライン（Basic Pitch → MuseScore）では、**無料で動くが初期MIDIはリズム誤りが多く微調整が必要**と明記。全自動スクリプトも「結果に不満、正確性を調整しきれなかった」と自己評価。

手動耳コピでも **bE6 を誤り音感が狂った**と再投稿する失敗が共有されている。

Logic 経由の Basic Pitch ワークフローでは、**IAC 経由でノート取りこぼし**が起きる未解決問題も。

→ **失敗の型:** ピッチOKでも時間構造・MIDI I/O で壊れる。

---

### 2.12 失敗例の整理（パターン化）

| パターン | 典型症状 | 関連ヒント |
|---|---|---|
| P1 キー競合 | ツールごとにキーが違う | キー事前確定 |
| P2 半倍テンポ | 79 vs 140 等 | BPM／half-time 指定 |
| P3 累積ズレ | 後半ほどグリッド崩壊 | テンポマップ |
| P4 過剰ノート | スケール外のゴミ音 | キー／スケール |
| P5 楽器誤認 | パートが混線 | 楽器・ステム |

## 3. 成功例（実投稿ベース）

### 3.1 フルミックス多楽器転写の現場成功（指弾きギター）

スタジオ側 **Spackl** は、故メンバーの **複雑フィンガーピッキング**・録音品質の悪いバッキングを Mirelo で転写し、**ほぼ完璧**、Ample Guitar で再生検証できたと報告。「他にこのパズルを解ける手段がなかった」。リアル奏者の代替ではないが、学習／進行の道が開けた、と評価。

→ 成功条件に近いもの: 単一メイン楽器寄りの素材、検証可能なギター音源、人手での用途判断。

---

### 3.2 「スケール／時間量子化」を後段ヒントとして使う

**NeuralNote**（Basic Pitch ベース）紹介では、

- 聴きながら転写設定を調整  
- **scale と time quantization を適用**  

が主要機能として強調されている。

Ableton 実務でも「誤ノートを手で直す代わりに **スケール強制**」が早道とされる。

→ これは厳密には「事前」より **解析後ヒント／制約** だが、製品の「解析ヒント」と同じ設計思想（探索空間の拘束）。

---

### 3.3 ステム分割 → 単体で MIDI 化（精度が上がる、という実務定石）

日本語圏の実務者 **錆人形/RE_DO** は、

1. Ultimate Vocal Remover / Audacity OpenVINO でステム分割  
2. **ステム単体**で Spotify Basic Pitch / Neural Note / WaveTone 等  

とすると MIDI 検出精度が上がる、と複数回説明。Basic Pitch 普及後に Audio-to-MIDI 機能が各所に増えた、とも観察。

中国語の組み合わせ案でも **Demucs で人声分離 → Basic Pitch** が定番パイプライン。

→ **楽器ヒントの代理:** ステム分離＝「この帯域／パートは○○楽器」という強い事前条件。

---

### 3.4 テンポ・コード・基準音を「解析の土台」にする（Melodyne 公式実務）

Celemony は Audio-to-MIDI を推しつつ、実務では

- **Assign Tempo Mode**（テンポマップ作成・クォンタイズの土台）  
- **chord grid** をチューニング参照に  
- **pitch reference / master tuning** を先に  

と「ヒント相当の手順」を公式に繰り返し教育している。

→ 成功の型: **先に時間構造と基準音を確定 → その後に音高操作／MIDI化**。

---

### 3.5 DJ／リミックス文脈：キー合わせ失敗の回避

プロデューサー **PRINCE KAYBEE** は、曲キーとアカペラキーを合わせる準備をせよ、キー検出ソフトを使え、と助言（自分もよくミスった）。

→ 採譜本体ではないが、「キーを確定してから後工程」という **ヒント文化** の成功パターン。

---

## 4. 限界（投稿から抽出）

1. **自動キーは一意に決まらない**（特に電子音楽・モーダル・Ambiguous）。  
2. **テンポは half/double とジャンル依存で破綻**。  
3. **ポリフォニー／フルミックスは依然ハード問題**。  
4. **スケール強制は誤検出を消すが melodic truth を削る**。  
5. **楽器が分かれてもドラムマップ等の意味層は別問題**。  
6. **カポ／チューニングは「音高正解」と「奏法正解」が衝突**。  
7. **データ不足が AMT のボトルネック**（kyutai: MT3以降データが律速、170k録音・11k時間で改善）。  
8. **商用でも「宣伝精度」≠「編集可能な譜面」**。

## 5. ベストプラクティス（実投稿から合成）

優先度の高い順（実務投稿の収束点）:

### BP1. ステム／パートを先に切る（楽器ヒントの最強実装）
フルミックス直投げより、人声・ベース・ギター等を分けてから MIDI 化。

### BP2. テンポ（グリッド）を先に固定する
Melodyne の Assign Tempo、DAW の tempo map、half/double の人手判定。累積誤差を防ぐ。

### BP3. キー／スケールを「制約」として与える
検出結果を鵜呑みにせず、確定キーへ scale quantization。ただし転調・ブルーノートは手動例外。

### BP4. 基準チューニングを先に合わせる
A440 決め打ち禁止。pitch reference / master tuning を確認。

### BP5. 解析コンポーネントをバラして再実行
Ora DJ のように BPM・key・energy・waveform を**個別再解析**できる設計は、現場の失敗修復と相性が良い。

### BP6. ドラムは「転写＋マップ指定」までセット
General MIDI 前提は危険。ターゲット音源のマップを指定 or 人手割当。

### BP7. ギターは concert pitch と fret/capo 表記を分離
半音下げ＋カポの相互作用を先に宣言しないと TAB が死ぬ。

### BP8. 出力は「下書き」扱い、リズムは必ず耳で直す
無料パイプラインでも初期リズム誤りは前提。

## 6. 最新トレンド（2025–2026 投稿から）

### T1. フルミックス多楽器 AMT の再加熱（MuScriptor / Mirelo × kyutai）
- フルミックスから voice / drums / bass / keys 等を分離MIDI化  
- **コード・キー・テンポも検出**して文脈を返す  
- 従来の「1音源ずつ」との差別化を強調  
- 合成MIDI事前学習 → 実データ fine-tune → **人手検証300曲のRL**  
- 開発側は **instrument presence conditioning** も研究キーワードに  


→ トレンドの含意: ヒントの一部（楽器存在・キー・テンポ）を **モデルが内製推定**する方向。  
→ ただし投稿上、ドラムマップ等 **実務ヒントはまだ人手**。

### T2. Basic Pitch エコシステムの定着
- Spotify Basic Pitch → DAW／NeuralNote／自作ONNX移植  
- 「ステム → Basic Pitch」が共通語  


### T3. 後処理ヒント UI の標準化
- scale / time quantization をスライダーで当てる  
- 「再解析を部分実行」  
NeuralNote・DJ解析ツールで同型。

### T4. キー検出の「ジャンル特化 CNN」など、ヒント代替の専用モデル
Ora DJ が pitch-equivariant CNN over CQT で key detection を自作し、市販級に寄せたと報告。

### T5. 記譜ソフト側は「移調楽器」など人間向けヒントを地道に実装
ScoreTail が Bb クラリネット／Fホルン等の **transposing instrument** 対応を「地味だが使える記譜の本質」と build in public。

→ AMT の解析ヒントと対になる **記譜ヒント** の潮流。

| 観点 | X上の実務合意 |
|------|----------------|
| **ヒントは有効か** | **有効**。ただし多くは「事前入力UI」ではなく **後処理の強制（scale/time quantize）** や **前処理（stem分離）** として実装・運用されている |
| **失敗が特に多い領域** | ①キー検出の不一致 ②BPM半／倍速誤検出 ③フルミックス多声 ④商用Audio-to-MIDIの過大広告 ⑤ドラムMIDIマッピング ⑥カポ／変則チューニングのタブ化 |

## 2. 成功例（実投稿）

### 2.1 フルミックス転記が「難曲・低品質音源」で通った例
スタジオ側の実務報告。故バンドメンバーの **複雑フィンガーピッキング**、しかも録音品質が悪いバッキングをデモで転記 → Ample Guitar Taylor で検証し「完璧に近い」と評価。他ツールでは解けなかったパズルだった、と明言。

> 含意: 楽器種別が「ギター」に寄せられ、結果が **演奏可能なMIDI** になった成功例。  
> 「楽器ヒント／楽器認識」が効いた可能性が高い（製品側も楽器別MIDIを売りにしている）。

### 2.2 自動でキー・テンポ・コード文脈を返す（生産者向け成功訴求）
Mirelo × kyutai の Audio-to-MIDI は、フルミックスから楽器別MIDIに加え **chords / key / tempo** を検出すると公式が説明。

開発側（Paula Garcia Domingo）も「既存のほぼ全ツールは一度に1音源」「自社はフルミックスでギター・声・ピアノ・ドラムまで同時」と対比。

### 2.3 「スケール強制」で雑MIDIを実務レベルに salvage
Ableton の Audio-to-MIDI は「少し間違って悪い音を足す」→ **スケールに強制**すると素早く近づく、というバンド実務ハック。

> これは「キー／スケールの事前ヒント」の最も実務的な形。  
> **検出が不完全でも、ユーザー知識で拘束すれば使える**。

### 2.4 NeuralNote：転記後に scale / time quantize を調整
NeuralNote（Basic Pitch 系）は、聴きながら設定を調整し **scale と time quantization を適用**してから DAW へ、と紹介されている。

> 製品UI上は「解析ヒント入力」に近い **後段拘束UI**。事前指定と同型の制約最適化。

### 2.5 ステム分割 → Basic Pitch で精度が上がる（実務定石）
日本語圏だが実務密度が高い投稿。

- ステム分割後に単体で読ませるとMIDI検出精度が上がる、と伝聞・経験として共有。  
- WaveTone / NPlay24 / Basic Pitch を **併用**し、いずれもステム分離すれば精度が上がる、と明示。

中国語圏でも同様の **組合せパイプライン**が語られる。Basic Pitch でMIDI → MuseScore。ただし **初期MIDIはリズム誤りがあり手動微調整が必要**とも正直に書かれている。

### 2.6 Melodyne：テンポマップ／基準音／コードグリッド（開発元の成功設計）
Celemony 公式:

- **Assign Tempo Mode** でテンポマップ作成・クリック無し録音の量子化など。  
- チューニング前に **正しい pitch reference（基準音）** を確認せよ。  
- ボーカルを触る前に **曲の master tuning を確認**（未確認だと「全部合わせたのに変」になる）。  
- コード分析結果を **chord grid として視覚的チューニング基準**にする。  
- Audio-to-MIDI で記譜ソフトへ、という正攻法導線。

> 開発元視点では「ヒント」はオプションではなく **必須前処理** として教育されている。

### 2.7 フィンガーピッキング／学習用途の「使える」評価
中国語プロモ投稿でも、MuScriptor は管弦楽など多楽器で「七七八八（だいたい）取れる」と実機感を訴求。完璧ではなく **実務の一次ドラフト** としての成功。

## 3. 失敗例（重点：投稿を多めに）

### 3.1 「何年経っても Audio-to-MIDI はまだ悪い」
- 音楽家 John Maus: 「今ごろ Audio to MIDI は解けているはずだと思わない？ Melodyne 等に載っていても **全部かなり悪い**」。旧 Chordino の方がハーモニーではマシだった、と。  
- GenAI／Ableton 実務者 Heba AI: 「Audio-to-MIDI を何年もやってきたが **とても不正確**。Ableton も Logic も **かなり悪い**」。

> 失敗の本質: ヒント無し・フルミックス・汎用設定のまま期待すると **本番不可**。

### 3.2 商業転記ソフトの「約束と現実」崩壊（AudioScore）
Neuratron AudioScore に対する強い否定:

- 「**utterly useless**」「金の無駄」「約束の足元にも届かない」  
- 楽器誤認（1本なのに複数／その逆）、出力が **random / unusable**、サポート不満、過剰宣伝。

> 「楽器を自動認識する」マーケは、**楽器ヒントをユーザーが出せない／出せても効かない**と失敗体験になる。

### 3.3 キー自動検出の不一致（同一曲で3通り）
DJアプリ開発者 Yannick Feige（Ora DJ）:

> 同じトラックを Rekordbox / Serato / Mixed In Key に入れると **3つの違うキー**が返る。バグだと思っていたが違う。**キー検出は本質的に難しい**。電子音楽は特にタチが悪い。

> 採譜ヒントとして「キー」を自動任せにすると、**入力時点で既に間違った拘束**が入り、全ノートが半音／モード単位でズレる連鎖失敗が起きる。

### 3.4 BPM 半速／倍速の誤検出
- Shazam 批判: Dubstep 系なのに **79BPM** と出るが、体感は **140BPM** 級。半速誤認の典型。  
- リズムゲーム文脈でも手動BPMチャートは「どうやってもズレる」不満（自動検出への期待＝現状の自動が当てにならないことの裏返し）。  
- ジャンル偏り: 多くの検出が EDM 最適化で、Jazz / Classical / Ambient は **3–5フレームずれ**る、という開発者観察。

> テンポヒントが無い／誤ると、記譜の小節線・符割り・量子化が根こそぎ壊れる。

### 3.5 拍間隔誤差の累積（中国語・技術的失敗）
中国語投稿:

> 拍間隔の測りが不正確だと誤差は必ず **逐拍累積**する。BPMか対数単位かは関係ない。精度が本質。**扒带（耳コピ／タイム同期）でも原曲が合わない**のはこのため。譜面は拍間隔の積分だから。

> 拍子・テンポヒントが「平均BPM 1値」だけでは足りない。**テンポマップ／局所BPM** が要る、という失敗の構造説明。

### 3.6 人手採譜でも音高ミス（ツール以前の失敗）
中国語: 昨日の扒谱を削除再投稿。「**bE6 まで行く音高で間違い**。音感が少し死んでる…」。

> 自動ツールでも「オクターブ誤認」と同型。**音域／楽器ヒント**が無いと高音域で破綻しやすい。

### 3.7 ドラムは転記できても「マップ」が死ぬ
Mirelo 公式も認める限界: ドラムは consolidated MIDI stem で、**まだ手動マッピングが必要**。ユーザーは EZDrummer / Addictive / Logic 等への正しいマップを要求。

> 「楽器＝drums」と指定しても、**キットマッピングという第二の楽器ヒント**が無いと演奏不能。

### 3.8 多声・フルミックスがボトルネック
- 「Music transcription はまだ Whisper 的瞬間を迎えていない。**Polyphony が難しい**。n 本の楽器が同じスペクトルで喧嘩する」。MuScriptor 投稿への反応。  
- 研究側 PitchBench: **Audio-language models は基本的な pitch hearing テストに失敗**（楽器・条件横断）。

### 3.9 パイプライン成功談の「裏側失敗」
中国語の Basic Pitch → MuseScore フロー自体が成功例として書かれつつ:

- 初期生成MIDIは **节奏错误（リズム誤り）で微調整必須**  
- 全自動スクリプトも「**结果不满意，准确性一直没调好**」と自己評価。

日本語の Basic Pitch 実務でも、Logic でノート修正 → IAC 経由で取りこぼし、と **二次加工で再び壊れる**。

### 3.10 カポ／チューニング由来の「譜が合わない」
- ギタリスト Frank Iero: 標準チューニングならカポ2、**自分は half step down なのでカポ3**。  
- OSP: タブサイトが「**カポ4でないと原曲キーに合わない、じゃないと低すぎ**」— 声域とカポ指示のズレをネタ化。  
- The Hard Times 風刺見出し: *Guitar Tab Mostly Right Except for Tempo, Tuning, Note Choices*（テンポ・チューニング・音選択以外はだいたい正しい＝全部違う）。

> 自動採譜でも同じ: **演奏ピッチ（聴こえ）と奏法表記（カポ付き指板）は別問題**。ヒント無しだと「正しい音高・使えないタブ」か「正しい形・ズレたキー」のどちらかに落ちる。

### 3.11 テンポ変更と MIDI タイミングの罠
日本語実務: MIDIは発音位置の「相対微調整」がテンポ変更で崩れる、という構造的失敗。

> テンポヒントを後から変えると、先に「人間タイミングで直した」転記が壊れる。**テンポは先に固定すべき**、という失敗から来るBP。

### 3.12 移調楽器の記譜ギャップ
ScoreTail 開発者: Clarinet in B♭ / Horn in F / Alto sax の移調対応を「地味だが記譜ツールが実用になる条件」として出荷報告。

> 楽器ヒントが「実音」前提だと、**パート譜としては失敗**する。

## 4. 限界（投稿から抽出した構造）

| 限界 | 根拠となる投稿群 | ヒント入力への含意 |
|------|------------------|-------------------|
| キーは一意に決まらない | 3ツール3キー問題 | ユーザー上書き必須。自動キーをヒントにすると汚染 |
| BPM は半速／倍速の多義 | Shazam 79 vs 140 | テンポは「候補選択UI」向き |
| 平均BPMでは足りない | 拍誤差累積 | テンポ**マップ**／区間指定が必要 |
| フルミックス多声 | polyphony / PitchBench | 楽器ヒント＋ステム or 条件付け |
| ドラムは音高≠キット | Mirelo 公式 | ドラムマップ・プリセットが第2ヒント |
| データ不足が歴史的ボトルネック | kyutai: MT3(2022)以来データが律速 | モデル改善でもドメイン外は崩れる |
| カポは「表記の話」 | Iero / OSP / 風刺 | チューニング＋カポを **pitch offset と fingering に分離**しないと失敗 |
| ジャンル偏り | EDM最適化でクラシックがズレる | ジャンル／拍子ヒントが必要 |
| 「検出」と「記譜規約」は別 | 移調楽器、簡譜 vs 五線 | 出力フォーマットヒントも必要 |

kyutai 側の技術開示（限界の裏返し）:

- データ不足が AMT を長らく縛っていた  
- 合成MIDI 150万 → 実データ fine-tune → **人手検証済み300曲での RL** が品質ドライバー  
- 5秒チャンクの autoregressive、長尺は chunk-by-chunk  
- 論文・モデル: MuScriptor（arXiv 2607.08168 等）

別要約投稿では **instrument presence conditioning（楽器存在の条件付け）** が手法に含まれる、と整理されている。これはまさに「楽器ヒント」のモデル内実装。

## 5. ベストプラクティス（実務投稿から再構成）

優先度順（失敗コストの高いものから）:

### BP1. テンポを最初に固定する
- Melodyne の Assign Tempo Mode 的に **テンポマップを先に確立**。  
- 後からテンポをいじると MIDI 微調整が壊れる。  
- 半速／倍速が怪しいジャンル（ダブステップ、ハーフタイム・メタル等）は **手動BPM上書き**。

### BP2. 基準音・マスターチューニングを先に
- チューニング／ピッチ補正の前に pitch reference を確認。  
- 曲全体の master tuning を確認してから各トラック。  
- A=440 以外・テープ変速・ライブ録音では **チューニングヒント無し＝全半音ズレ**。

### BP3. キー／スケールは「検出結果を信じず、拘束として使う」
- 自動キーはツール間で食い違う前提。  
- Ableton 等で雑に取れたら **force to scale**。  
- NeuralNote のように **scale + time quantize を後段で聴きながら適用**。

### BP4. 楽器ヒント＝「単一ステムで食わせる」が最強の実務解
- ステム分割 → Basic Pitch / WaveTone 等。精度が上がる、が複数実務者の一致。  
- 中国語フローも Demucs 系分離 → Basic Pitch。  
- フルミックス一発モデルは進化中だが、**失敗時の切り分け**はステム経路が強い。

### BP5. ドラムは「楽器」指定＋「マップ」指定の二段
- 一括ドラムMIDIのあと、Toontrack 等のマップへ手動 or プリセット。

### BP6. ギターは「聴こえキー」と「カポ／チューニング表記」を分離
- half-step down とカポ位置の関係を人間が明示。  
- ヒントUIなら: `tuning=Eb standard`, `capo=3`, `output=sounding pitch | tab relative` の両方。

### BP7. 拍子は「4/4前提」を疑う
- 11/8 など変拍子は人間でもカバーを躊躇する難所。自動量子化が 4/4 グリッドに潰すと壊滅。  
- 拍子ヒント or 複合拍子モードが無いと、符割りが永久に直らない。

### BP8. 解析はコンポーネント単位で再実行
DJ分析アプリの設計示唆: BPM / key / energy / waveform を **個別に re-run**。全体再解析より安全。

> 採譜ソフトのヒントUIも「キーだけ差し替えて再量子化」「テンポだけ再マップ」が望ましい。

### BP9. DJ／コラボ文脈では「キー一致」を事前に
キー検出ソフトで **曲キーとアカペラキーを合わせる**、というDJ実務アドバイス。

> 採譜パイプラインでも「素材間のキー整合」はヒント共有の価値が高い。

### BP10. 期待値を「清書」ではなく「草稿＋拘束」に置く
成功例でも「人間ギタリストの代替ではないが道が開く」。  
失敗例の多くは **ワンクリック完成** を期待したときに起きる。

## 7. 機能設計への示唆（投稿から逆算）

「解析ヒント入力」を実装／評価するなら、X実務から見える必須セット:

1. **テンポ**: BPM 単値＋半速／倍速トグル＋テンポマップ  
2. **キー／スケール**: 自動候補を出しつつ **ユーザー確定がデフォルト**、force-to-scale  
3. **楽器**: 単一楽器モード／フルミックス多楽器／ドラムキットマップ  
4. **拍子**: 単純拍子＋複合／変拍子、量子化グリッドと連動  
5. **チューニング**: A4 Hz、ドロップ／半音下げ、平均律以外の余地  
6. **カポ**: sounding pitch と tablature pitch の切替  
7. **再実行粒度**: キーだけ／テンポだけ／量子化だけ  
8. **失敗可視化**: 「キー信頼度」「BPM半速候補」「オクターブ曖昧ノート」をUI表示

失敗例が多い領域ほど **ヒントのROIが高い**:

- キー（ツール間不一致）  
- BPM（半速／倍速）  
- チューニング基準（A≠440）  
- カポ／変則チューニング（表記と実音）  
- ドラムマップ
