# 音楽採譜ソフトの「バッチ処理（複数ファイル一括採譜キュー）」  
X（旧Twitter）実務者・研究者・開発者投稿調査（英語・中国語中心）

**調査日:** 2026-07-21  
**対象:** 自動音楽採譜（AMT）／Audio-to-MIDI／楽譜化パイプラインにおける **複数ファイル処理・ジョブキュー・チャンク処理**  
**方針:** 実投稿ベース。失敗・限界を厚めに。Speech-to-Text（Whisper等）は「キュー設計の類推」のみ補足。

---

## 1. 調査サマリー（結論先出し）

| 観点 | 現場の実態（X上） |
|------|------------------|
| **バッチ需要** | 強い（フォルダ投入・夜間一括・CLI/キュー）。ただし「数時間で完成譜が並ぶ」期待はほぼ裏切られる |
| **失敗の主因** | ①精度不足 ②チャンク間の楽器割当不一致 ③並列実行によるOOM ④混音・ポリフォニー・ジャンル依存 ⑤後処理（量子化/ドラムマップ）欠落 |
| **成功条件** | 単一楽器（特にピアノ独奏）、事前ステム分離、結果を「下書き」と割り切る、1ジョブずつ＋復旧付きキュー |
| **最新トレンド** | **フルミックス→多トラックMIDI一発**（MuScriptor等）と、**マルチトラック評価データセット**（MulTTiPop）。SOTAでも「十分」ではない |
| **製品示唆** | バッチ機能は「高速一括正解」より **ジョブ管理・失敗隔離・QCワークフロー・制約の明示** が価値の本体 |

---

## 2. 調査上の注意（ソース特性）

1. **「transcription」ノイズが大きい**  
   X上の `batch transcription` は大半が **音声文字起こし**（Whisper / Azure / Cohere等）。音楽採譜とは別問題。本レポートは音楽側を優先し、音声バッチは「キュー設計の失敗例」としてのみ扱う。

2. **「専用製品のバッチUI」を語る英語/中国語投稿は相対的に少ない**  
   代わりに多いのは、(a) 研究評価での系統的失敗、(b) 現場パイプライン（Demucs→MIDI）、(c) 新モデル試用の精度批判、(d) 中国圏の「批量转MIDI」宣伝＋制約注記。

3. **中国語**は宣伝投稿が多く、批判投稿は **Songsterr AI / PianoTrans制約** などに偏る。英語は **CMU G-CLef / Kyutai-Mirelo** 周辺が情報密度が高い。

## 3. 失敗例（厚め）

### 3.1 研究ベンチで見える「バッチしても直らない」構造失敗

**CMU G-CLef の Nathan Pruyne らが MulTTiPop（ポップ・マルチトラック評価）を公開。** 既存AMTを評価した結果が、バッチ機能設計に直結する。

- **MT3:** パート間で **一貫した編曲（arrangement）を保てない**（同一曲を通して楽器役割がブレる）。  
- **YourMT3+:** 編曲とリズムを **過剰に単純化**する。  
- **MuScriptor（最新フルミックス系）:** 最も一貫性が高いが、なお **チャンク横断で楽器パートが不一致**になることがある。  
- 量的にも **Onset F1 は約38%帯** など「改善余地が大きい」との要約投稿あり。  
  出典: [@pruynathan スレッド](https://x.com/pruynathan/status/2075772813462450389) / [chunk不一致](https://x.com/pruynathan/status/2075772820190003374) / [@chrisdonahuey コメント](https://x.com/chrisdonahuey/status/2075810524609101972) / [@OrchestralPit 要約](https://x.com/OrchestralPit/status/2075543197284000100)

**製品への含意:**  
バッチで100曲流しても、**「チャンク継ぎ目でギターが消えピアノが増える」**ような失敗は **スケールでは消えない**。むしろ夜間バッチほど、目視QC前に **サイレントな品質崩壊**が積み上がる。

---

### 3.2 最新SOTAでも現場試用は「参考にならない／絶望」

Mirelo × Kyutai の Audio-to-MIDI（MuScriptor）は大バズだが、実務試用の失敗報告が明確。

| 試用者の評価 | 内容 |
|--------------|------|
| 人力でもできる曲 | 精度 **約50%** |
| 人力では絶望的な曲 | 精度 **約10%**（認識結果 vs 原曲比較） |
| 既存Audio-to-MIDI全般 | **実用レベルに達していない** |
| コード推定系（deCoda / chord.tube等） | **悲惨で信用していない** |

出典: [試用詳細](https://x.com/CabbageLettuce1/status/2078348054697246827) / [精度良くない](https://x.com/CabbageLettuce1/status/2077330927487971635) / [既存ツール批判](https://x.com/CabbageLettuce1/status/2077289312421626276)

**プロ寄りの冷静評価:** 楽器検出は「まだまだこれから」。音高の下書きとしては助かるが、完成譜ではない。  
出典: [@AraiSanshiro](https://x.com/AraiSanshiro/status/2077383551046345215)

**バッチ含意:** 成功率50%以下のジョブをキューに積むと、**後工程の修正コストがボトルネック**になる。「一括処理」より **失敗分類・再試行ポリシー**が重要。

---

### 3.3 「局所的には正しそう、全体構造は崩壊」

音楽AI全般への批判として、**局所は正しいが大規模構造が取れない**という指摘。採譜でも「小節単位はそれっぽいが、楽章・フォーム・繰り返し構造が壊れる」類の失敗と親和性が高い。  
出典: [構造失敗の指摘](https://x.com/Blarg08125613/status/1896355878338748716)

---

### 3.4 ポリフォニー・ペダル・密集ジャズがバッチ精度を破壊

- **ポリフォニーが本丸の難しさ**（複数楽器が同じスペクトルを奪い合う）。密集ジャズで検証する、という開発者投稿。  
  出典: [@helloLizZhang](https://x.com/helloLizZhang/status/2075615962091729343)
- **ピアノ:** ポリフォニー＋ペダル＋重なり音で **大半のモデルが壊れる**。  
  出典: [@irshit0](https://x.com/irshit0/status/2071641834087215303)

**バッチ含意:** フォルダ一括投入時に **ジャンル/楽器メタデータなし**だと、失敗率が曲ごとに乱高下し、平均SLAが意味をなさない。

---

### 3.5 現場DAWの「Audio to MIDI」は微妙に間違い・余分な音を足す

Ableton等のAudio-to-MIDIは **少し間違えて悪いノートを足す**。スケール強制で寄せる、という応急処置が語られる。  
出典: [@dj_irl](https://x.com/dj_irl/status/2016891106236256452)

元プロ作編曲家も、SunoのStemを直したいが **Audio-to-MIDIの精度がイマイチで結局耳コピ**。  
出典: [@YUJIRO34160841](https://x.com/YUJIRO34160841/status/2047965174355972534)

**バッチ含意:** 「全自動完了」をUXにするとクレームになる。バッチ完了後は **必ず人間編集キュー**が必要。

---

### 3.6 中国語圏：AI一鍵扒譜は便利だが「反人类和弦」

SongsterrのAI一鍵扒譜は参考として便利だが、**論理が乱雑で反人類的な和音**が出る。修正はしたが理論に自信がなく大改変できない、というユーザー投稿。  
出典: [@aaaaa39391](https://x.com/aaaaa39391/status/2012102517442887961)

**バッチ含意:** 批量生成は **「譜面のゴミ山」を高速生成する危険**がある。QCなしの公開/配布はブランド毀損。

---

### 3.7 チャンク処理固有の失敗（長尺＝暗黙のバッチ）

Kyutai側の技術説明: MuScriptorは **5秒のmel断片を自己回帰トークン予測**し、長い音声は **chunk-by-chunk**。  
出典: [@kyutai_labs](https://x.com/kyutai_labs/status/2075540050930991170)

研究側はまさにここで **楽器割当のチャンク間不一致**を指摘（3.1）。  
→ 1曲処理すら「内部バッチ」であり、**複数曲キューの外側問題以前に、内側分割が壊れる**。

---

### 3.8 出力後工程の失敗（MIDIが出ても製品にならない）

| 失敗 | 内容 | 出典 |
|------|------|------|
| ドラム | 統合MIDIステムのため **手動マッピング必須**。標準マップが曖昧 | [Mirelo公式返信](https://x.com/MireloAI/status/2075624374338465847) |
| 多Voice MIDI | 1トラックに2Voice以上があると **レンダリング不良** | [@hmking_works](https://x.com/hmking_works/status/2077027936977526827) |
| ハードウェアMIDI | 安物USB-MIDIは **遅延・データエラー**（SysEx時） | [@DraTohru_XLN](https://x.com/DraTohru_XLN/status/2078729215604867218) |

**バッチ含意:** 採譜キュー完了＝終了ではない。**MIDI正規化・楽器マップ・Voice分割・量子化**の後段ジョブが必須。

---

### 3.9 キュー設計そのものの失敗（並列＝OOM）※音声だが設計教訓は同一

複数録音を同時にFolder Actionで並列Whisper→ **MacがOOM**。対策として **1件ずつのjob queue + crash recovery + dedupe** を実装。  
出典: [@omriariav](https://x.com/omriariav/status/2053931747742540050)

**音楽採譜への転用:** Demucs + AMT はWhisper以上にVRAM/メモリを食う。**「バッチ＝並列最大」は最悪パターン**。

---

### 3.10 研究史的「失敗」：構造を入れてもデータ量に負ける／でもMT3も現場では不足

PhDが和声構造を明示したモデルを作り、Googleの **MT3（構造ゼロ＋大量データ）**に完敗した「bitter lesson」回顧。  
出典: [@jxmnop](https://x.com/jxmnop/status/1927385194601886065)

一方で現在の評価では **MT3でも編曲一貫性が崩れる**（3.1）。  
→ **スケールで勝っても、バッチ運用で信頼できるわけではない**。

## 5. 限界（投稿から抽出した「超えにくい壁」）

| # | 限界 | 投稿からの根拠 |
|---|------|----------------|
| 1 | **マルチトラック一貫性** | MT3編曲崩壊、MuScriptorもchunk不一致 |
| 2 | **絶対精度の天井** | 現場50%/10%評価、Onset F1にheadroom |
| 3 | **楽器ラベルの難しさ** | 和声/打楽に縮約するとSOTA、厳密ラベルだと中位 |
| 4 | **ジャンル・編成依存** | ジャズ密集、電子、ギター指弾きで分散 |
| 5 | **長尺チャンク境界** | 5秒分割の継ぎ目問題 |
| 6 | **後処理必須** | 量子化、スケール矯正、ドラムマップ、Voice分割 |
| 7 | **計算資源** | 並列OOM、GPU前提 |
| 8 | **単一楽器前提の製品が多い** | PianoTransは純ピアノのみ等 |
| 9 | **「譜面」≠「MIDIノート」** | 記譜論理・声部分割・可読性は別問題（Songsterrの論理崩壊） |
|10 | **評価データ自体の希少** | MulTTiPopが「評価インフラ」として注目されるほど、現場バッチQAが困難 |

## 6. ベストプラクティス（投稿群からの実務合意）

### 6.1 キュー設計

1. **デフォルトは直列（concurrency=1）**。並列はプロファイル後にだけ。OOM経験談が根拠。  
2. **crash recovery / dedupe / 再試行** をジョブ単位で。  
3. **ジョブ状態:** `queued / running / failed / needs_review / approved`。`done`を自動完了にしない。  
4. **失敗の隔離:** 1曲が落ちてもキュー全体を止めない（ただしVRAM破損時は例外）。

### 6.2 入力制約の先出し（中国PianoTrans型）

- 対応: 純ピアノ / 単旋律 / ステム済み、等  
- 非対応: 密なフルミックス、極端な歪み、電子パーカッション等  
- バッチUIで **ファイルごとの適合スコア**を出すと失敗が減る

### 6.3 パイプライン段階化

```
[Ingest] → [可选: Source Separation] → [AMT/MIDI] → [Normalize]
  → [Quantize/Scale] → [Score Engraving] → [Human QC] → [Export]
```

- フルミックス一発モデルは前進だが、研究も **まだheadroom大**。  
- 難しい曲は **分離→単一楽器モデル**にフォールバック。

### 6.4 品質ゲート（バッチの命）

- チャンク間の **楽器一貫性チェック**  
- Onset密度・音域外れ・連続休符・不協和の急増をフラグ  
- 自動「完成」禁止。**要レビュー率をKPI**に

### 6.5 人間側ワークフロー

- AIは **耳コピの下書き**（音高当てスキップ用）  
- 悪いノートは **スケール強制で寄せる**応急が現場にある  
- 最終は記譜ソフト/DAWで量子化・声部整理

## 7. 最新トレンド（2025–2026、X上）

| トレンド | 内容 | 代表投稿 |
|----------|------|----------|
| **A. フルミックス多トラックAMT** | ステム不要で楽器別MIDI | Mirelo/Kyutai MuScriptor |

## 8. 製品機能「一括採譜キュー」への設計チェックリスト

投稿ベースの失敗を機能要件に落とすと:

| 必須機能 | 理由（失敗対応） |
|----------|------------------|
| 直列キュー＋VRAM監視 | 並列OOM |
| 曲ごと独立ログ/成果物 | サイレント品質崩壊の追跡 |
| チャンク継ぎ目の一貫性検査 | MuScriptor/MT3系の既知弱点 |
| 入力プロファイル（ピアノ/ステム/フルミックス） | ドメイン外バッチの大量失敗 |
| 後処理プリセット（量子化/スケール/ドラムマップ） | MIDIが出ても使えない |
| `needs_review` 必須ステート | 50%精度帯での誤完了防止 |
| フォールバック経路（分離→単一楽器） | フルミックス失敗時 |
| バッチ統計（成功率・要修正率・平均編集時間） | 「速いが使えない」を可視化 |

## 10. 総括（バッチ機能の「正解」定義）

X上の実務者・研究者が示しているのは次の一点に収束する。

> **音楽採譜のバッチ処理で失敗するのは、キュー実装が下手だからではない。  
> 単曲ですら一貫性と精度が足りないのに、複数曲を「完了」として積む設計が失敗する。**

したがって、採譜/記譜ソフトのバッチ機能の勝ち筋は:

1. **スループット自慢ではなく、失敗の観測可能性**  
2. **制約の先出し（楽器・ジャンル・品質SLA）**  
3. **チャンク一貫性・後処理・人間QCを含むジョブ状態機械**  
4. **フルミックスSOTAと、分離→単一楽器のフォールバックの二系統**

「夜間に100曲を流して朝には完成譜」は、2026年時点のX実務・研究の合意からは **まだマーケティング側の物語**に近い。現実的な価値は **「100曲の下書きと、修正すべき30曲のリストを朝に渡す」** 側にある。
