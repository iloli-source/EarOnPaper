3. ストーリーは「呪い（つらさ）」と「ギフト（作曲・耳コピ）」の両面を持つと共感が広がる。

### 観測された訂正パターン

| 誤解（世間） | 当事者・識者の訂正 | 言語 | 時期 |

### 複数音・微分音

- 和音が一発で取れる能力への**憧れ**（「和音もパッと分かる絶対音感が欲しい」）。  
- 微分音をやるなら**相対音感の方が役立つ**という当事者（日・2026）。  
- 英語側: 異なるチューニングに対し、自動不快ではなく「質問として固定しないと反応しない」という記述——**選択的絶対音高処理**。

### アプリ仕様への示唆（テーマ2）

1. **信頼度（confidence）と「近似音名」**を必須に。倍音ノイズには「音名不可／帯域のみ」を返す。  
2. 単音→複音→コードの**段階的モード**が、実体験の能力分布と一致。  
3. A=440 固定は危険。**基準ピッチ校正（440/442/可変）**とセント表示が「本物感」を出す。  
4. 加齢・基準ドリフトの物語は、ユーザーが「自分の耳をアプリに預ける」正当化にも使える（キャリブレーション機能）。

## 6. フィールド録音モード F-108 仕様への厚め示唆

コンセプト整合: **雑音混じり日常録音から、音程成分を選択抽出して音符化** ＝ 実在する絶対音感の「注意＋選択的ラベル化」モデルに近い。

### 6-1 何を音符化すべきか（優先度）

| 優先 | 音源タイプ | 理由（X上の体験根拠） |
|------|------------|------------------------|
| P0 | **準定常的な純音・狭帯域トーン**（家電ハム、警報、電子ビープ、靴の電子音、信号音） | 音名列挙の体験が多い／デモ映え |
| P0 | **意図的楽音**（口ずさみ、楽器、発車メロ、街のBGM断片） | ユーザーの主用途 |
| P1 | **打撃音の基音が明確なもの**（コップ、机、金属） | 「何の音？」あるある——ただし自信度必須 |
| P1 | **明確なメロディ輪郭を持つ環境音**（鳥の一部、サイレンのグライドを階段近似） | 「日常が音楽」体験の核 |
| P2 | **複数同時の楽音（和音・重なり）** | 人間でも難易度が高い→段階表示 |
| P3 | **話し声のイントネーション** | 興味はあるが音符化は誤解を生みやすい。オプション |

### 6-2 何を無視／非音符化すべきか

| 無視 or 別扱い | 理由 |
|----------------|------|
| 広帯域ノイズ（雨、ざわめき、交通のブロードノイズ） | 音名が付かない／付いても無意味。絶対音感者も通常ラベル化しない |
| 倍音が立ちすぎた打撃・衝突 | 「ドレミに落とせない」あるある |
| 連続グライドのみ（ドアきしみ、一部サイレン） | 単一音名より contour／gliss 表現が適切 |
| 部屋の残響・エコーの二次反射 | 偽ピーク |
| 人声のフォルマント主体（子音・雑音部） | 音高成分抽出は pitched 部分のみ |
| 超低域のエアコン等で「不快ノイズ」としてだけ意味を持つ音 | 音符化より「 hum 検出・マスキング提案」がユーザビリティ |

### 6-3 F-108 が実装すべき「絶対音感エミュレータ」挙動

1. **選択的注意 UI**  
   - タイムライン上で「ここを聴く」タップ／ブラシ選択。  
   - デフォルトは全自動ではなく **候補ハイライト → ユーザー確定**（ござ氏の匂いモデル）。

2. **音事件の分類タグ（音符以外も返す）**  
   - `pitched_stable` / `pitched_transient` / `noisy` / `speech` / `poly` / `inharmonic`  
   - noisy・inharmonic は楽譜に落とさず「音響オブジェクト」として残す。

3. **信頼度とセント偏差**  
   - 「ソ#っぽい」回答をそのまま UI に。  
   - 440 からのズレを可視化（加齢・チューニング体験に対応）。

4. **単音ファースト、和音はオンデマンド**  
   - リアルタイム全和音分解をデフォルトにしない（失望防止）。  
   - 「もう一度聴く（ループ分析）」が人間の自己報告と一致。

5. **フィールド用プリセット**  
   - 家電ハム採譜 / 街のメロディ断片 / ノック音ゲーム / 発車メロ抽出 / 雨＋メロ（メロのみ）  
   - プリセットごとに **無視マスク** を変える。

6. **相対モード併設**  
   - 最初の音をドとみなす移動ド出力。相対派・プロへの橋渡し。

7. **「つらさ」ではなく「コントロール」**  
   - 常時ラベル地獄の再現はアクセシビリティ事故。  
   - オンデマンド絶対音感が、体験談の健全な側と一致。

8. **失敗を物語に内包**  
   - 「この音は倍音が多すぎて音名にできません」はバグではなく**仕様の誠実さ**（くいなちゃんあるあるの製品化）。

### 6-4 F-108 の受け入れ基準（体験ベース）

- ノック音デモで「当たる／外れる／判断不能」が説明可能であること。  
- 雨＋口ずさみで、雨を無視して口ずさみだけ五線に載ること。  
- エアコン 50/60Hz ハムを「低いドに近い」等で近似しつつ、**楽譜の主旋律に自動混入しない**こと。  
- テンポ不定のフィールド音で、無理な melodic grid を強制しないこと（既存AI採譜の失望点）。

## 7. 「絶対音感エミュレータ」物語の市場通用度

### 判定: **条件付きで通用 — スコア B+（日本市場は A-、英語圏は B、中華圏 B-、南アジアは当面 C）**

| 観点 | 判定 | 理由 |
|------|------|------|
| 物語の直感性 | **高** | 「全部がドレミ」は誇張だが、一般人の理解コストが低い。日本では特に強い |
| 当事者からの科学的反発 | **中** | 「常時全部」は否定される。**選択的エミュレータ**と言えばむしろ当事者に近い |
| 相対音感派からの反発 | **中** | 「絶対音感崇拝」と見られると教育者層が離反。相対併記で緩和可 |
| 既存AI採譜との差別化 | **高（設計次第）** | スタジオ曲の自動採譜競争ではなく、**雑音環境での選択抽出**は空白に近い |
| 英語マーケ | **中** | perfect pitch の比喩汚染と、生成AI批判文脈での “perfect pitch” ネガが混在 |
| 中国語 | **中** | 走音苦痛・才能語彙はあるが、生活音採譜ニーズの言説は薄い |
| ヒンディー圏 | **低（現時点）** | X上の一次需要シグナル不足。インド古典音楽の shruti 文化とは別設計が必要 |

### 物語の勝ち筋（推奨フレーミング）

> **「絶対音感者のように、雑音の海から“音程のあるもの”だけをつまみ出せる耳」**  
> — 全部を楽譜にする魔法ではなく、**注意と選択のエミュレーション**。

これは X 上の当事者訂正（匂いの比喩、質問しないとラベル化しない、和音は難しい）と整合し、かつ非当事者の憧れ（欲しい／耳コピしたい）とも接続できる。

### 物語の負け筋（避ける）

- 「AIが本物の絶対音感を獲得」など**生物学的誇称**  
- 「生活音がすべて美しいメロディに」などの**ファンタジー過多**（当事者はつらさも語る）  
- 既存採譜AIと同じ土俵での「100%正確」約束（失望ログが既に厚い）

## 8. 見つからなかった／薄かった領域（正直報告）

1. **ヒンディー語・Hinglish の音楽的絶対音感日常談** — 実質未検出。  
2. **中国語での家電・駅・雨などの具体音名列挙** — ほぼ未検出（走音・加齢・相対十分論はあり）。  
3. **英語の「冷蔵庫が〇音」系の高頻度一次投稿** — 今回の期間・クエリでは少なく、ノックあるあるとチューニング反応が中心。比喩ノイズが調査を阻害。  
4. **AI採譜を明示的に「絶対音感」と呼ぶ大量言説** — 生成AIの完璧ピッチ皮肉の方が多い。採譜は「期待と耳コピ回帰」が主。  
5. **プロオーケストラ奏者の長文スレッド** — 断片的。相対音感十分論はギタリスト・教育者・管楽器勢が目立つ。  
6. **ヒンドゥスターニ／カルナータカの shruti と absolute pitch の衝突談** — X上では今回捕捉できず（学術・YouTube 寄りの可能性）。
