# NF-043 推論ランタイム別受入ゲートのベンチ設計（Issue #21）

**設計日:** 2026-07-19
**目的:** NF-043（Must）「推論ランタイム別受入ゲート」の実行設計。「量子化・GPU/NPU利用で必ず高速化・同品質という前提を置かない」を検証プロトコルとして明文化する。
**実証済みの根拠（本プロジェクトでの実例）:** Basic Pitch の **ONNX 経路がこの環境で壊れた出力**（無音入力にノート事後確率 mean 0.60）を返し、**TFLite 経路は正常**（無音で max 0.16）だった（G0'台帳§7.1、2026-07-19実測）。同一モデルでもランタイムで結果が変わることは既に自分たちで踏んだ — 本ゲートはその再発防止の制度化である。

---

## 1. 対象ランタイムと確認項目

| ランタイム | 主用途 | 演算子対応・fallbackの確認手順 |
|---|---|---|
| ONNX Runtime | クロスプラットフォーム標準 | ①モデルロード時のログで未対応op・CPUfallbackの有無を記録 ②EP（CoreML EP/CPU EP）別に同一入力の出力を比較 ③**無音・既知信号でのサニティチェックを必須化**（ONNX破損事例の直接の教訓） |
| Core ML | macOS/iOSのNPU活用 | ①coremltoolsの変換ログで演算子変換失敗を記録 ②compute unit指定（CPU/GPU/ANE）別に出力一致を確認 |
| TFLite | 現行の正規経路 | ①delegate（XNNPACK等）有無での出力一致 ②現行基準線としてリグレッション監視 |

**共通サニティテスト（ゲート入口・自動化）:**
1. 無音入力 → ノート事後確率が閾値未満（壊れたONNXはここで即検出できた）
2. 既知の合成メロディ → spike回帰テストの合成E2Eと同一の正解突合
3. ランタイム間の出力差分 → 同一入力でノートイベント集合のF1が0.99以上（それ未満は「同じモデル」とみなさない）

## 2. 量子化精度劣化の測定手順（品質KPIとの接続）

- **判定器:** AIの耳ハーネス（4指標）＋ PD正解つきベンチ（Note F1）。量子化前後のモデルを**同一曲セット**にかけ、差分を測る
- **手順:** ①FP32基準線を確定（PDベンチF1・AIの耳スコアを記録）②量子化版（INT8等）で同一セット再実行 ③劣化幅を記録（F1低下・幽霊オンセット増加数）④受入判定: **F1劣化≤2pt かつ サニティテスト全パス**で採用可（閾値は初回実測後に封緘）
- **禁止事項:** 「量子化したから速いはず」で採用しない。速度・品質の**両方を実測**してから判定（ONNX量子化が古い端末でむしろ遅くなる報告があることは要件調査済み）

## 3. 電力・発熱の測定手順（NF-036接続）

- macOS: `sudo powermetrics --samplers cpu_power -i 1000` を採譜実行中に併走させ、平均/ピーク電力を記録。連続解析30分でのサーマルスロットリング有無（クロック低下）を確認
- 記録様式: ランタイム×モデル×曲長の表（W平均・W峰値・処理時間・スロットリング有無）
- Windows実装時は HWiNFO 等の相当手段を追記（現段階はmacOSのみ）

## 4. 受入ゲート判定表（ひな形）

| 判定項目 | ONNX RT | Core ML | TFLite(基準) | 合格条件 |
|---|---|---|---|---|
| サニティ（無音・既知信号） | | | ✅実測済 | 全パス必須 |
| ランタイム間出力一致（F1） | | | 基準 | ≥0.99 |
| PDベンチF1劣化 | | | 基準 | ≤2pt（仮・初回実測後封緘） |
| 処理時間（3分曲・CPU） | | | 基準 | 実測記録（速い前提を置かない） |
| 電力・発熱 | | | 基準 | スロットリング非発生 |
| 演算子fallback | | | — | fallback内容をログ記録・未対応opゼロが望ましい |

**運用:** 新ランタイム・新量子化・モデル更新のたびにこの表を1枚起こし、回帰評価CI（NF-021）の成果物として保存。表が埋まらないランタイムは採用しない（デフォルト不採用）。

## 5. 残課題

- 閾値（F1劣化≤2pt等）は初回のFP32/INT8実測後に封緘（現状は仮置き）
- Core ML経路は未着手（macOSアプリ実装段階で本ゲートを通す）
- 自動化スクリプト化は spike のbench群に統合予定（Issue化して実装時に対応）
