# 採譜エンジン実精度 改善ロードマップ（Note F1@100ms 0.765 → ?）

**作成:** 2026-07-22
**目的:** 実採譜精度（Note F1@100ms=0.765 / score_rhythm=0.410）を、**盲目的チューニング
（それ自体が偽成功）を避けて**段階的に上げるための調査記録と手順。外部レビュー(grok)採点で
「エンジン実精度 6/15」が残る唯一の大きなギャップであり、テスト/GUI/E2E の磨き込みでは動かない。

## 大原則: 測ってから直す
「取りこぼし 0.235」が**どこで生まれているか**を分解しないと、何を直すべきか分からない。
エラーは必ず次の4種のどれか。まずこれを bench(`spike/ear-pipeline/bench/bench_pd.py`)で
曲ごとに%集計する（エラー・タクソノミー）。

| エラー種別 | 典型原因 | 対処の重さ |
|---|---|---|
| 音高ミス | pYIN/basic-pitch のオクターブ誤り(f0の半分/2倍)・倍音誤検出 | 軽〜中 |
| オンセットずれ(>100ms) | 量子化グリッドへのスナップ／**先頭無音トリムによる全体オフセット** | 軽（重要） |
| 取りこぼし(miss) | 弱音・速い音・重なり | 中 |
| 余分な音(extra) | ゴースト音・残響尾・倍音 | 中 |

## 第1次プローブの発見（2026-07-22・①②→全13曲タクソノミー）

> 調査の経過を正直に残す: 仮説→易しい曲で一旦反証に見えた→深掘り診断で**最悪外れ値の
> 主因として検証**された。これが盲目チューニング回避の実例。

### 検証された発見: 先頭無音トリムが「無音を持つ曲」の F1 を壊滅させる（測定artifact）
- 全13曲を poly(basic-pitch)+timing=raw で採点。**平均 F1@100ms = 0.748**（報告0.765とほぼ一致）。
- **kojo_no_tsuki だけ F1=0.051 の壊滅的外れ値**（P=0.057/R=0.046, gt=131/pred=106）。
- 診断: この曲は GT が t=0.24s 開始（前奏の無音あり）。エンジンが `trimmed_leading_sec=0.159` で
  先頭無音を**正しくトリム**した結果、採譜MIDIの時間軸が GT に対し ~0.16s 早くずれる。
  F1@100ms(tol=0.1)は 0.16>0.1 なのでほぼ全滅。**時間オフセット +0.10s を掛けると一致 6→83**。
- 他12曲は GT が t=0 開始で `trim=0` → 影響なし（だから最初の3曲テストでは反証に見えた）。

### 影響の定量（トリム量だけ pred を整列して再採点）
| | base | +trim整列 | Δ |
|---|---|---|---|
| kojo_no_tsuki | 0.051 | **0.700** | +0.649 |
| 13曲平均 | 0.748 | **0.798** | **+0.050** |

- **これは engine の質の問題ではなく bench の測定欠陥**: bench は GT-MIDI を無音込みで wav 化し、
  エンジンは正しくトリムするのに、採点前に `trimmed_leading_sec` の再整列をしていない。
  一定の全体オフセットは音の相対タイミングを保つので、note精度指標が罰するのは不適切。
- **是正は正当**（数字の細工ではない）: bench で pred を `trimmed_leading_sec` だけ整列してから
  採点する（または GT にも同じ前処理をかける）。これで報告 F1 が 0.748→0.798 に正直に上がる。
  ただし**採譜品質そのものが上がったわけではない**（kojoの音は元々ほぼ正しく、時間軸だけずれていた）。

### 反証された仮説: grid vs raw
実コーパスで `timing=grid` と `raw` の F1 は**完全同一(Δ=0.0)**。格子量子化はオンセットを
100ms以上ずらしていない。→ raw/grid は F1 に効かない（仮説B反証）。オクターブ誤りの寄与も小
(gnossienneで+0.019のみ)。→ オクターブ後処理の ROI は低い。

### トリム是正の後に残る「本物の」失点（難曲・engine品質の課題）
| 曲 | F1 | 診断 | 主因 |
|---|---|---|---|
| traumerei | 0.633 | precision低 | 余分音(extra) — 残響/倍音/ゴースト |
| sakura | 0.708 | recall低 | 取りこぼし(miss) — 多声の重なり |
| gymnopedie1 | 0.746 | precision低 | 余分音 |
| gnossienne1 | 0.769 | 均衡 | 混合 |
| moonlight_1st | 0.781 | precision低 | 余分音 |

→ **残る本当の課題は「余分音(precision)」が優勢**。次手は残響尾/倍音/ゴースト音の抑制後処理。

## 改善方針（検証済みの優先順）
1. **bench のトリム整列是正**（実装容易・報告0.748→0.798・engine非改変）→ **`bench_pd.py` main() に
   実装済み**（`_align`: pred を `trimmed_leading_sec` だけ GT 時間軸へ戻して採点）。
   → 「測定が正しくなる」だけで品質向上ではない点を明記して報告する。
   → follow-up: 他のベンチ変種(adaptive/score-checks の note_f1 site 281/364/445)にも同整列を
     適用して一貫させる（本コーパスでは無音を持つのは kojo_no_tsuki のみなので影響は限定的）。
2. **余分音(precision)抑制**: traumerei/gymnopedie/moonlight の extra を減らす後処理
   （残響尾トリム・整数倍音抑制・最小音長/エネルギー閾値の曲非依存調整）。実 engine 品質向上。
3. **取りこぼし(recall)対策**: sakura 等の多声重なり。basic-pitch 感度/ステム分離前処理。
4. 頭打ちなら**モデル差し替え**（新しめの AMT）。統合・ライセンス要検討。

## 重い改善（上が頭打ちになってから）
6. モデル差し替え（basic-pitch は2021年世代。新しめの AMT／mono用CREPE強化）。統合・ライセンス要検討。
7. ドメイン適合の fine-tune。データが要る・利得不確実。**現時点では非推奨。**

## 正直な注意（偽成功を避ける）
- **利得は不確実**。「必ず○点」とは言えない。測定→狙い撃ち→再測定の反復が必須。
- 着手前に X/論文/WEB で**先行研究（特に失敗例）を先行調査**する（プロジェクト方針）。
- 最終判定は「同じ曲に聞こえるか」の**聴感テスト**が要る。機械指標だけ上げて満足しない。
- 発見A/Bはいずれも**まず測定・評価を正す**話で、engine 改変前の必須ステップ。
