# 採譜エンジン実精度 改善ロードマップ（Note F1@100ms 0.765 → ?）

**作成:** 2026-07-22
**目的:** 実採譜精度（Note F1@100ms=0.765 / score_rhythm=0.410）を、**盲目的チューニング
（それ自体が偽成功）を避けて**段階的に上げるための調査記録と手順。外部レビュー(grok)採点で
「エンジン実精度 6/15」が残る唯一の大きなギャップであり、テスト/GUI/E2E の磨き込みでは動かない。

## 大原則: 測ってから直す
「取りこぼし 0.235」が**どこで生まれているか**を分解しないと、何を直すべきか分からない。
エラーは必ず次の4種のどれか。まずこれを bench(`spike/ear-pipeline/bench/bench_pd.py`)で
曲ごとに%集計する（エラー・タクソノミー）。

| エラー種別 | 典型原因 | 対処の重さ |
|---|---|---|
| 音高ミス | pYIN/basic-pitch のオクターブ誤り(f0の半分/2倍)・倍音誤検出 | 軽〜中 |
| オンセットずれ(>100ms) | 量子化グリッドへのスナップ／**先頭無音トリムによる全体オフセット** | 軽（重要） |
| 取りこぼし(miss) | 弱音・速い音・重なり | 中 |
| 余分な音(extra) | ゴースト音・残響尾・倍音 | 中 |

## 第1次プローブの発見（2026-07-22・①②実施）

### 発見A（重要・オンセット系）: 先頭無音トリムが GT に対して**全体を時間シフト**させる
既知オンセット `[0.30, 0.85, 1.40, 1.95, 2.50, 3.05]` の単旋律を合成し `transcribe_file` に通した結果:

| timing | 検出オンセット(先頭) | 100ms以内一致 |
|---|---|---|
| grid | `[0.00, 0.548, 1.096, 1.644, ...]` | **0/6** |
| raw  | `[0.011, 0.568, 1.113, 1.659, ...]` | **0/6** |

全オンセットが約 **0.29〜0.30秒 早い方向へ一様シフト** している（true 0.30 → pred ~0.01）。
これは**先頭無音の自動トリム**（既定ON・可読性向上のため。`trimmed_leading_sec` で報告）が、
GT（トリムしていない正解）に対して採譜結果の時間軸を丸ごとずらすため。F1@100ms は
オンセット時刻の一致を見るので、トリム量 >100ms のとき**系統的に大量失点**する。
→ **grid/raw の差(~20ms)より、トリム由来のオフセット(~300ms)の方が支配的。**

### 発見B（測定系）: bench は F1 を**格子量子化オンセット**で測っている
`bench_pd.py:135` は `transcribe_file(wav, out_midi=spike_mid, engine="poly")` を既定
`timing="grid"` で呼ぶ。F1@100ms は**生オンセット時刻の指標**なのに、格子スナップ後の
時刻で採点している。エンジンは `timing="raw"`（実タイミングMIDI）を持つのに評価に使っていない。

## 安い改善候補（ROI順・先に試す）

1. **評価アラインメントの是正（最有力・測定の是正）**
   - bench で採譜結果を GT に合わせて `trimmed_leading_sec` だけ時間シフトしてから採点する
     （または eval 時のみトリムを無効化／GT にも同じ前処理をかける）。
   - 発見Aの通り、現状の失点の相当部分は**エンジンの質ではなく評価のアラインメント欠如**の
     可能性が高い。ここは engine を触らずに F1 が上がりうる。**まずここを潰して再測定する。**
   - 注意: これは「数字を上げるための細工」ではなく、指標が本来測るべきものを正しく測る是正。
     アラインメント後に残る失点こそが本当の精度課題。

2. **F1 を raw タイミングで測る**: bench の spike 予測を `timing="raw"` で生成して比較。

3. **オクターブ誤り後処理**: 局所音高連続性＋想定音域で f0 の半分/2倍ジャンプを補正。音高F1に直効。

4. **閾値スイープ**: basic-pitch の onset/frame/最小音長を bench 上で振って F1 最大点を探す
   （`--sensitivity` の実測最適化）。

5. **倍音・重複抑制**: 同時発音の強い音の整数倍音を除去。

## 重い改善（上が頭打ちになってから）
6. モデル差し替え（basic-pitch は2021年世代。新しめの AMT／mono用CREPE強化）。統合・ライセンス要検討。
7. ドメイン適合の fine-tune。データが要る・利得不確実。**現時点では非推奨。**

## 正直な注意（偽成功を避ける）
- **利得は不確実**。「必ず○点」とは言えない。測定→狙い撃ち→再測定の反復が必須。
- 着手前に X/論文/WEB で**先行研究（特に失敗例）を先行調査**する（プロジェクト方針）。
- 最終判定は「同じ曲に聞こえるか」の**聴感テスト**が要る。機械指標だけ上げて満足しない。
- 発見A/Bはいずれも**まず測定・評価を正す**話で、engine 改変前の必須ステップ。
