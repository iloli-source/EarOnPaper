一次資料: [tab-notation-grok-x.md](tab-notation-grok-x.md) / [tab-notation-codex-papers.md](tab-notation-codex-papers.md) / [tab-notation-web.md](tab-notation-web.md)

## 3方向が一致した核心

1. **LLM直出しTABは捏造で使えない**（grok X: 英中で多数の不満）。**音響解析ベース**が正道 → 本エンジンは音響→音程イベントベースなので方向性は正しい
2. **運指の非現実性・ポジション飛びが「使えないTAB」の典型**（grok X）。運指最適化＝手の移動最小化は論文でも定石（codex: Sayegh 1989のDP、Itoh 2004、Radisavljevic 2004の学習コストDP、Hori 2013のHMM、Edwards 2024 MIDI-to-Tab）
3. **チューニング表記は必須**（Web: 「無いTABは調号なし五線譜と同じ」）
4. **TAB単体は音価（リズム）を表現できない**のが本質的弱点。五線譜併記が標準（Web英語圏・中国語圏aibapu both）
5. **数字が弦線に打ち消されると可読性が落ちる**（Web: Donald Sauter標準化提案）

## 最大の発見: 音数削減がTAB実用化の鍵

今回のテスト曲（ピアノ的多声・音符3994）をそのままTAB化すると、6弦に載せきれず47音ドロップ・14%の数字が重なり、**ギターでは弾けない譜面**になった。調査の全方向がこれを裏付ける:

- grok X: 「音が多すぎ・非現実的な運指」が使えないTABの typical
- codex 論文: playabilityは制約違反率で測る。arrangement研究（Hori 2013・MIDI-to-Tab 2024）は「弾ける範囲へ音を減らす/選ぶ」ことが本質
- Web 中国語圏: aibapuもギター向けには五線譜と別に六線譜を「適配」生成＝ギター向けに整形している

→ **実用的ギターTABには「メロディ/伴奏の分離」または「音数削減してギター向けにアレンジ」する層が必要**。これは二層アーキテクチャの「記譜プロファイル」より上流の「編曲層」であり、現状の耳→そのままTAB化では不足。ユーザーの原体験「コード譜から起こすとローコードになりがち・転調でポジションが飛ぶ」も、この編曲層＋手移動最小化DPで解く問題。

## 推奨する次アクション（優先順）

1. **単音（メロディ）モードのTAB**: `--engine mono` の単旋律をTAB化すれば重なりは激減し「弾けるTAB」になる。既存単音エンジンで即検証可能（最小コスト・高効果）
2. **五線譜＋TAB併記**（音価が読める）: 調査で最も標準的な形。Verovioは五線+TAB併記に対応の可能性→将来オプション
3. **技法記号**（h/p/b/s）とリズム表記: 実用度を上げるが優先度中
4. **コードフォーム認識**（ユーザー要望のローコード偏重回避）: 和音を慣用フォームに割当てる拡張。DPコストにコード形の慣用度を追加
