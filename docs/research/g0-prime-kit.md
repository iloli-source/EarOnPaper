## §1. 実施手順書（合計 約4時間）

### Step 1: 事前セットアップ（約40分）

1. **MuseScore Studio を 4.7.4 以上に更新**（10分）— 起動→ヘルプ→更新確認。File メニューに「Import audio to score」があることを確認
2. **musescore.com アカウント確認**（5分）— audio2score はオンライン処理のため musescore.com へのログインが必要な可能性が高い。既存アカウントの有無を確認し、**要否・無料枠の制限（曲数/曲長/ファイルサイズ）を台帳の備考欄に記録**
3. **MuScriptor 導入**（15分）— ターミナルで:
   ```bash
   pip install muscriptor        # または uv add muscriptor
   muscriptor list-instruments   # 動作確認を兼ねる
   ```
   Python 3.10-3.12 が必要。GPUなしで可（smallモデル）
4. **作業フォルダ作成**（5分）— `~/works/iloli/プロジェクト/採譜/bench/g0-prime/` に `input/` `output-musescore/` `output-muscriptor/` を作成
5. **3曲の音源を input/ に配置**（5分）— 選定基準は §2。MP3/WAV形式

### Step 2: 投入（各曲×2ツール = 約60-90分）

各曲について:

1. **MuseScore audio2score**: File→Import audio to score→音源をアップロード→変換完了を待つ→MSCZを `output-musescore/` に保存。**処理時間・失敗の有無も記録**
2. **MuScriptor**:
   ```bash
   muscriptor transcribe input/<曲>.mp3 -o output-muscriptor/<曲>.mid
   # 多楽器曲は: --instruments で対象楽器を指定（list-instrumentsで確認）
   # CPUが遅ければ --model small
   ```
3. 出力6ファイル（3曲×2ツール）が揃ったことを確認

### Step 3: 手直し感の記録（各出力15-20分 × 6 = 約90-120分）

各出力を **MuseScore Studio で開き**、以下を観察して台帳（§3）に記録:

- 原曲を再生しながら譜面を目視比較（正式測定ではない。タイマー計測不要 — 体感の3段階評価でよい）
- 「この譜面から実用譜に直すのに、ゼロから耳コピするのと比べてどれくらい楽か」を体感で判定: **軽い（≤40%相当）/ 重い（>40%）/ 使用不能**
- 崩れの種類をチェック: 音高 / リズム・拍子 / 整譜（小節線・声部）/ 奏法
- **崩れが規則的か非規則的か**: 同じパターンの誤りが繰り返される（後処理で埋まりそう）か、バラバラ（モデル刷新が要りそう）か
- **耳起因か記譜起因か**（二層構造の検証）: 音高・オンセット自体が拾えていない（耳の失敗）のか、拾えているのに譜面表現が崩れている（記譜の失敗）のか

### Step 4: 判定と記録（約20分）

- §4 の判定フローを適用し、結果を台帳に記入
- 台帳をコミット（このキットと同じフォルダに `g0-prime-ledger.md` として保存）
- 判定結果に応じた次アクション（§4 末尾）を Issue #13 にコメントで記録

## §2. 固定3曲の選定基準（spec §2 準拠: U1×1・U2×1・U4×1）

**著作権上の注意（全曲共通):** ユーザーが合法に保有する音源（購入済み・自作・PD）を使う。処理は私的な検証目的（30条1項の私的使用の範囲）で行い、**出力譜面は公開・配布しない**。台帳には曲名でなく曲IDで記録し、著作権状態列を必ず埋める。

| 枠 | 用途セル | 条件 | 例のタイプ |
|---|---|---|---|
| 曲1 | U1 ピアノカバー | 市販譜が存在しないピアノソロカバー。中庸テンポ（90-140BPM）・中程度の編曲密度・スタジオ品質 | ゲーム/ボカロ/東方系のピアノアレンジ |
| 曲2 | U2 弾き語り | ボーカル＋ギターorピアノの2要素。コード進行が明確なJ-pop系 | アコースティック弾き語り音源 |
| 曲3 | U4 バンド | ドラム・ベース・ギター・ボーカルを含む多楽器。ポップバンド編成・轟音系は避ける | バンドサウンドの市販音源 |

- 3曲とも**尺は2-4分**（無料枠制限に当たりにくく、観察時間も揃う）
- G1本ベンチで再利用できるよう、選んだ曲は台帳の曲メタ列（§3）を全部埋めておく

## §4. 判定基準（spec §2 封緘ルールの転記 — 当日変更禁止）

```
「手直しが軽い（体感で準初級手直し比率≤40%相当）」の曲が 2曲以上
  → 無料・デフォルト搭載の競合が既に到達済み
  → 残りのゲートを実施せず【クローズ】（プランA相当に降格・台帳を教育資産として保存）
  → 記録: Issue #13 に結果を記載してクローズ、全Issueに休眠ラベル、README現在地を更新

上記に該当しない（手直しが重い）
  → 【G2前半へ進む】（Issue #15: 質問文は封緘済み・つてリスト作成から）
  → G0'の観察はG1の選曲・仮説精緻化・出力プロファイルの磨き込み順序（五線デフォルト＋どのオプション形式を先に磨くか）の材料に使ってよいが、G2質問文には反映しない（情報隔壁）
```

- 判定に迷うグレーな体感の場合は**保守側（クローズ寄り）に倒さない** — G0'は粗い観察なので、迷ったら「重い」としてG2前半に進み、正式判定はG1に委ねる（specの帯判定はG1側にある）
- **同時に確認する全停止シグナル（spec §7-1）:** audio2score が「軽い」を連発する場合はシグナル1発火に相当。判定と独立に記録する

## §5. よくある落とし穴

1. **audio2score はオンライン処理** — 音源が musescore.com のサーバーに送信される。これは自社製品の「成果物非関与」とは無関係の**他社サービスの検証利用**であり、私的な検証目的・合法保有音源の範囲で行えば問題は小さいが、未発表の自作音源など秘匿したいものは投入しない
2. **MuScriptor の重みは CC BY-NC 4.0** — 検証（非商用）はOK、**将来の製品組込は不可**。検証結果を「MuScriptorをそのまま使えばよい」という結論に直結させない（Issue #4）
3. **MuScriptor の出力は MIDI** — MusicXML ではないので整譜品質（小節線・声部・記譜）は MuseScore の MIDI インポータの品質と混ざる。「記譜起因の崩れ」を評価する際は、この混入を台帳の所感に書き分ける
4. **ツールが動かない場合** — audio2score が地域/アカウント制限で使えない場合はその旨を台帳に記録し、MuScriptor＋Basic Pitch（`pip install basic-pitch`）の2本で代替実施。MuScriptor が環境問題で入らない場合は Python 3.12 の仮想環境を作り直す
5. **観察のしすぎ** — G0'は半日の粗い観察であり、正式測定（ブラインド化・タイマー計測）はG1の仕事。1出力20分を超えたら次に進む
```
