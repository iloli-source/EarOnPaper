## 役割分担（2026-07-19改訂2: 評価はAIの耳＋ユーザー聴感の二段判定）

ユーザーは楽譜を読まない・音感を必要としない前提で設計する。評価は **AIの耳ハーネス（事前封緘した客観指標）** と **ユーザー自身の聴き比べ（ターゲットユーザー=非音楽家として最適な判定者。楽譜不要）** の二段で行う。音楽関係者のつては、プロダクトが五線譜PDFを出力できるようになった後の製品レビューに温存し、G0'では使わない（ユーザー決定 2026-07-19）。

## §1. 実施手順書（ユーザー実作業 約1.5時間。うち聴き比べ判定 20-30分）

### Step 1: 事前セットアップ

1. 【ユーザー・10分】**MuseScore Studio を 4.7.4 以上に更新** — 起動→メニューの「ヘルプ」→「更新を確認」。File（ファイル）メニューに「Import audio to score」という項目があればOK
2. 【ユーザー・5分】**musescore.com アカウント確認** — audio2score はオンライン処理のためログインが必要な可能性が高い。既存アカウントの有無を確認（なければ無料登録）。画面に出た**制限（曲数/曲長/ファイルサイズ）はそのままAIに伝える**（台帳に記録します）
3. 【AI・15分】**MuScriptor 導入と動作確認** — AIがターミナルで実行:
   ```bash
   pip install muscriptor        # または uv add muscriptor
   muscriptor list-instruments   # 動作確認を兼ねる
   ```
4. 【AI・5分】**作業フォルダ作成** — `bench/g0-prime/` に `input/` `output-musescore/` `output-muscriptor/` を作成
5. 【ユーザー・5分】**3曲の音源を input/ にコピー** — 選定基準は §2（AIが候補条件を提示するので、手持ちの音源から選ぶだけ。迷ったらAIに相談）

### Step 2: 投入（各曲×2ツール）

1. 【ユーザー・計30-60分】**MuseScore audio2score（数クリックの機械操作×3曲）**: File→Import audio to score→音源ファイルを選択→変換完了を待つ→保存先に `output-musescore/` を指定して保存。**処理にかかった時間と、失敗した場合はその画面表示をAIに伝える**
2. 【AI・並行】**MuScriptor 実行×3曲**:
   ```bash
   muscriptor transcribe input/<曲>.mp3 -o output-muscriptor/<曲>.mid
   ```
3. 【AI】出力6ファイル（3曲×2ツール）を整理し、AIの耳ハーネス（`tools/ai-ears/`）への入力と、聴き比べ用の再生ペア（元音源 vs 出力の合成音）を準備する

### Step 3: 評価（AIの耳ハーネス＋ユーザー聴き比べの二段判定）

1. 【AI・30分】**第一判定 — AIの耳ハーネス**（`tools/ai-ears/`）で6出力を評価: 音高一致度（クロマDTW類似度）・音の出だし一致・テンポ整合・譜面健全性を算出し、平易な解説つき評価レポートを生成（閾値は自己検証後に封緘済みのものを使用）
2. 【ユーザー・20-30分】**第二判定 — 聴き比べ**: AIが用意した再生ペア（元音源 vs 出力の合成音）を順に聴き、各出力を「①同じ曲に聞こえる / ②部分的に同じ / ③別物」の3段階で記録するだけ（楽譜は読まない。ターゲットユーザー=非音楽家の体験としてこの判定自体が製品価値の一次検証になる）
3. 【AI】両判定を §3 の台帳に転記・集計。「耳起因/記譜起因」はハーネスの指標分解で判定する（音高・リズム系の不一致が主=耳起因 / 一致率は高いが譜面健全性が低い=記譜起因）

### Step 4: 判定と記録（ユーザー 約10分）

- 【AI】§4 の判定フローに集計結果を当てはめ、判定案を提示
- 【ユーザー】封緘ルール（§4）通りかを確認して最終判断（「軽い」が2曲以上→クローズ / それ以外→G2前半へ）
- 【AI】台帳をコミットし、結果を Issue #13 にコメントで記録

## §4. 判定基準（spec §2 封緘ルールの転記 — 当日変更禁止）

※ハーネスの「到達」閾値は `tools/ai-ears/` の自己検証完了後・実測開始前に封緘し、本節と spec §2 に追記する（spec §0 の二段階封緘）。

```
「到達」（AIの耳ハーネスの複合スコアが封緘閾値以上 ∧ ユーザー聴感が①同じ曲に聞こえる）の曲が 2曲以上
  → 無料・デフォルト搭載の競合が既に到達済み
  → 残りのゲートを実施せず【クローズ】（プランA相当に降格・台帳を教育資産として保存）
  → 記録: Issue #13 に結果を記載してクローズ、全Issueに休眠ラベル、README現在地を更新

上記に該当しない（到達が1曲以下）
  → 【G2前半へ進む】（Issue #15: 質問文は封緘済み・A/B2系統の対象者リクルートから。spec改訂3参照。つては温存方針のためリクルート主導線にしない）
  → G0'の観察はG1の選曲・仮説精緻化・出力プロファイルの磨き込み順序（五線デフォルト＋どのオプション形式を先に磨くか）の材料に使ってよいが、G2質問文には反映しない（情報隔壁）
```

- 聴感判定が②（部分的に同じ）で迷う場合は**保守側（クローズ寄り）に倒さない** — G0'は粗い観察なので、迷ったら「重い」としてG2前半に進み、正式判定はG1に委ねる（specの帯判定はG1側にある）
- **同時に確認する全停止シグナル（spec §7-1）:** audio2score が「軽い」を連発する場合はシグナル1発火に相当。判定と独立に記録する

## §5. よくある落とし穴

1. **audio2score はオンライン処理** — 音源が musescore.com のサーバーに送信される。これは自社製品の「成果物非関与」とは無関係の**他社サービスの検証利用**であり、私的な検証目的・合法保有音源の範囲で行えば問題は小さいが、未発表の自作音源など秘匿したいものは投入しない
2. **MuScriptor の重みは CC BY-NC 4.0** — 検証（非商用）はOK、**将来の製品組込は不可**。検証結果を「MuScriptorをそのまま使えばよい」という結論に直結させない（Issue #4）
3. **MuScriptor の出力は MIDI** — MusicXML ではないので整譜品質（小節線・声部・記譜）は MuseScore の MIDI インポータの品質と混ざる。「記譜起因の崩れ」を評価する際は、この混入を台帳の所感に書き分ける
4. **ツールが動かない場合** — audio2score が地域/アカウント制限で使えない場合はその旨を台帳に記録し、MuScriptor＋Basic Pitch（`pip install basic-pitch`）の2本で代替実施。MuScriptor が環境問題で入らない場合は Python 3.12 の仮想環境を作り直す
5. **観察のしすぎ** — G0'は半日の粗い観察であり、正式測定（ブラインド化・タイマー計測）はG1の仕事。1出力20分を超えたら次に進む
```
