## 1. 競合構図の修正（事実誤りの訂正）

v3の「Songscription×MuseScore挟み撃ち」は**カテゴリ誤認**だった。

- **MuseScore NoteVision（2025年7月）はOMR** = 紙・PDFの楽譜画像を読み取る機能。音声→楽譜（AMT）ではない。よって本件の直接競合ではない（grok。round2-web-bizmodel の記述を訂正）
- **AMT直接競合集合:** Songscription（$5M調達・月次60%成長・Web）/ Klangio（独7名・10年弱）/ AnthemScore（買い切り+サブスク並走）/ Basic Pitch（OSS下限・CLI障壁）＋「→MuseScoreで編集」疎結合ハブ
- **MuseScoreの正しい位置づけ:** 無料記譜ソフトの圧倒的配布力を持つ**オプション脅威**。AMTを自社実装した瞬間に個人参入の余地は消滅する。→ AMTロードマップの定点観測（フォーラム・公式発表）を継続タスク化（Round 3 唯一の残存デスク調査）
- **正式証拠に昇格:** MusicRadar独立レビュー「Songscriptionはリズムが崩壊し、クリーンアップは手書きより大変」（grok発見）。勝負所（リズム/整譜）が現SOTAで未解決であることの第三者証拠

**含意:** 挟み撃ちは(まだ)存在しない。ただし「最も資金力のある競合の次の開発対象＝リズム/整譜」であり（competition）、勝負所が空いていても**時間窓は18-24ヶ月**と見積もる。時間耐久性のある堀（日本語UX・国内配布チャネル・スコアメーカー文化圏接続）を持たない参入案は時間窓の閉鎖で死ぬため、堀の有無を§4のトレードオフ軸に昇格させた（summary修正7）。

## 2. 判定の語調修正と免責軸の統一

### 2.1 「批判の射程が限定されたもの」（v3「反証されたもの」を格下げ）

| 批判の主張 | 射程限定の内容 | 残る有効射程 |
|---|---|---|
| 「買い切りとSaaSは両立しない」 | AnthemScore/カワイは**課金形態の並走**が可能なことを示す（存在証明） | 「成果物非関与の免責構造と成長SaaSの両立」は**未証明のまま**。Webサブスク=クラウド処理は免責の傘の外（logic飛躍1） |
| 「Basic Pitch無料で差別化消失」 | pip/CLI障壁により**現時点では**一般ユーザー未到達 | GUI障壁は埋められやすく（現に無料〜低価格競合が殺到）、**持続的差別化の論証は未達**（logic飛躍2） |
| 「買い切りデスクトップは死んだ」 | 日本のスコアメーカー文化圏は現役 | 役割は「AI下書きの手直しレイヤ」に縮小済み |

### 2.2 免責軸の統一（「ローカル」を要件から除去）

免責構造の判断軸は**「①コンテンツを利用者が用意し ②事業者が成果物に一切関与せず ③自動処理に留まる」の三点セットの充足度**であり、処理のローカル/クラウドは決め手ではない（ロクラクII総合考慮・自炊代行判決。legal穴2）。v3要件文の「ローカル採譜ツール」の語はすべて「**成果物非関与型採譜ツール**」に置換する。ローカル処理は「三点セットを満たしやすい実装」の一つに過ぎず、逆にクラウド処理でも非関与構成が立つ可能性は弁護士3問の回答までオープンとする。

### 2.3 「$15k MRR上限」の限定条件の明示

v3の「月$15k MRRが上限」は**無調達インディー3社（生存者標本）の観察上限**であり、市場の理論上限ではない（logic飛躍3。Songscriptionは調達により別分布）。v4では「無調達・個人開発で参入した場合の現実的な計画上限」として使う。

## 3. 4ゲート構造（「残る問いは1つ」の撤回）

実機ベンチは必要条件の1つを検証するに過ぎない。未決の問いは4つあり、それぞれ独立のゲートとする。**1つでも撤退帯に入れば全体撤退**（順不同で潰せるものから潰す）。

### G0. 経済性ゲート — 今すぐ・机上で判定可能（ベンチ結果と独立）

- **判定基準:** 「整譜をSOTA超えの自社開発（研究水準・約24人月・初期投資2,000万円相当・回収4-7年）で解きに行く」案は、囲炉裏の資金繰り（2026Q2危機・SES機会費用は同工数で確定2,400-3,240万円）に照らし**通さない**（economics）
- **通過できる唯一の形:** 「安く勝てる」場合のみ。すなわちベンチで観測される競合の崩れ方が、**既存OSS＋後処理3〜6人月で越えられる種類**である場合に限る。→ 技術ベンチの判定列に「安く勝てるか」を組み込む（summary修正13）
- **結論: G0は条件付き通過。**「細く長くやるなら整譜は解けない、整譜を解くなら細く長くは無理」という二律背反を、ベンチが「安い崩れ方」を示した場合にのみ回避できる

### G1. 技術ベンチゲート — 設計を§5に全面改訂

### G2. 需要ゲート — ベンチと並行実施（demand指摘の採用）

- ベンチはFeasibilityを測るがDesirabilityを測らない。X調査自身が「耳コピ・外注で満たされた市場、AI採用の社会証明なし」を示している以上、需要は独立検証が必要
- **実施:** B2C想定層（マイナー曲愛好家・弾き語り・コピバン）へのインタビュー5〜10件（計2.5時間・無料）＋前払い意思の確認（LP or 口頭での価格提示）
- **撤退基準（事前固定）:** 「現在の解決手段（耳コピ/外注/諦め)に不満があり、¥5,000以上の買い切りに前向き」が**5件中0件なら、競合が全滅でも撤退**

### G3. 法務ゲート — 弁護士確認を3問に分割（legal指摘の採用）

1. **タイプ2変換型該当性:** 採譜は「保存」でなく「変換」であり、文化庁平成27年報告書はメディア変換型を留保付きグレーとした。成果物非関与型採譜ソフトはこの留保をどう越えるか
2. **私的使用該当性:** 顧客がB2C個人でも、SNS配信・イベント演奏・バンド練習用途は30条1項の「私的使用目的」に収まるか（利用目的レイヤ）
3. **30条の4:** 学習データへの依拠は30条の4（情報解析）で権利制限され得るが、採譜は享受目的併存・楽譜市場代替を主張されうる。ただし書リスクの評価と、契約ベースでのクリーン化の要否
- **実施タイミング:** G1・G2が撤退帯でなかった場合のみ発注（費用最小化）。それまで「免責構造が立つ」は投資前提に置かない
- **付随論点（3問の回答時に併せて確認）:** 配布・アップデートの継続関与性（モデル更新配信は「管理・支配」を補強しうる）、アプリストア申立て＋差止仮処分の合わせ技という**攻撃コスト≪防御コストの非対称性**（legal穴4-5）

## 4. 制約条件の再定義（ANDの撤回）

v3の「7条件を満たさない事業案は検討しない」を撤回する。7条件を同時充足する実在製品はゼロであり（grok）、3組の内部矛盾が未解決だった。v4では**必須ゲート条件2つ**と**トレードオフ軸5つ**に分解する。

### 必須（これを欠く案は検討しない）

- **N1. 成果物非関与構造**（三点セット充足。G3の3問が返るまで暫定）
- **N2. 撤退線の遵守**（無調達・個人開発の計画上限$15k MRR帯を前提に、初期投下は後処理3-6人月・現金支出最小に制限）

### トレードオフ軸（設計時に選択する。両立不能の組合せを明示)

| 軸 | 選択肢と緊張関係 |
|---|---|
| T1. 出口 | 疎結合（MusicXML/MIDI→MuseScore編集）は既存行動と一致するが、**整譜品質の価値が下流の編集で希釈される**（3⊥7）。→ 価値主張は「手直し時間の削減」そのものに置き、エディタは持たない。体験の最終マイルを他社に依存するリスクは需要ゲートで検証 |
| T2. 課金 | 買い切り主軸。**サブスクは課金根拠（何に月額を払うか）が成果物非関与と緊張**（4⊥5）し、モデル更新配信は継続関与性を補強する（legal穴4）→ サブスク採否はG3回答後の設計判断に先送り。中国向けは買断一択 |
| T3. 技術投資 | SOTA超え自社開発は経済性ゲートで却下済み（6⊥7）。許されるのは既存OSS＋後処理の「安い勝ち方」のみ |
| T4. 堀 | 日本語UX・国内配布チャネル・スコアメーカー文化圏接続は**時間耐久性のある唯一の堀候補**（competition）。採用しない場合は18-24ヶ月の時間窓内に撤退可能な設計を要件化 |
| T5. 価値提案 | 「権利」でなく「工数削減」（v3から維持） |

## 5. 実機ベンチ設計 v2（G1の仕様）

**v3設計の欠陥（単一閾値・n=9・測定者未定義・合格線なし・GO側空欄）をすべて修正。この設計はRound 4のレビューを通してから封緘し、封緘後に測定を開始する。**

### 対象（役割分け）

- **撤退判定用（3本）:** Songscription / Klangio / AnthemScore — 到達されていたら撤退
- **参照値（2本）:** Basic Pitch（OSS下限）/ スコアメーカーZERO無料版（国内文化圏の手直しレイヤ）— 客観指標のみ軽く取得
- **対象外:** MuseScore NoteVision（OMRでありAMTではない。§1の定点観測で代替）

### 測定プロトコル

1. **基準線:** 同一整譜者が同一曲をゼロから耳コピ起譜する時間を先に測定。評価指標は絶対分数でなく**手直し比率 =（AI出力の手直し時間）÷（フル耳コピ時間）**
2. **誤り種別分解:** 音高 / 音価 / 拍子・小節線 / 声部分離 / その他 ごとに修正所要を記録（崩れの質で戦略含意が真逆になるため）
3. **測定者2名:** 熟練1名＋想定顧客層に近い準初級1名。独立測定・プロファイル明記・ブラインド化（どのツールの出力かを伏せる）
4. **客観指標併記:** mir_eval（Note F1 / Onset F1）を全出力に付す（社内ベンチ基盤としての正しい用途）
5. **選曲:** テンポ帯×多声/単声×録音品質のサンプリング表を事前作成しセルを埋める。全曲・全出力を曲ID付き台帳で公開（追試可能性）

### 2段階実施

- **フェーズA（3日・n=9）:** 粗い足切りのみ。「明確撤退帯」または「明確勝負所帯」に全ジャンルが揃った場合のみ決着
- **フェーズB（追加1週間・n=25-30）:** フェーズAが中間（最も起こりやすい）なら曲数を拡大し解像度を上げる

### 判定（3帯・事前封緘。数値はRound 4レビューで最終確定のうえ封緘）

| 帯 | 条件（撤退判定用3本のいずれかで） | 帰結 |
|---|---|---|
| **明確撤退帯** | 準初級測定者の手直し比率 ≤25% | 即撤退（先行者の領域） |
| **グレー帯** | 25%〜60% | フェーズB移行。B後もグレーなら「安く勝てるか」列で判定 |
| **勝負所濃厚帯** | 全社が ≥60% かつ 崩れの主因がリズム/拍子/整譜 | G2・G3ゲートへ進む |

- **「安く勝てるか」列（G0接続）:** 観測された崩れ方が「既存OSS出力＋後処理3-6人月で埋まる種類（規則的・後処理可能）」か「学習データ・モデル本体の刷新が要る種類」かを誤り種別分解から判定。後者なら勝負所濃厚でも撤退

### GO側のゲート列（process指摘の採用。プランB採用時のみ有効）

- **G1通過 → スパイク定義:** 「Basic Pitch等の出力→リズム/整譜後処理→MusicXML」プロトタイプ1本。成功指標=同一ベンチ再測で手直し比率を撤退判定用3本の最良値より20pt以上改善。投下上限3人月・現金支出上限10万円。未達なら撤退
- 各ゲートに時間箱と中止条件があり、**どのゲートでも「進める理由が立証できない場合のデフォルトは撤退**」

## 6. 経営判断の提示（ユーザー決定事項・Round 4 開始前に選択を推奨）

## 8. 総合判定

| 項目 | v3 | v4 |
|---|---|---|
| 総合判定 | No-Go継続・残る問いは1つ | **No-Go継続・残る問いは4ゲート**（経済性=条件付き通過済み、技術・需要・法務=未実施） |

**v4の結論を一行で:** 事業仮説は依然すべて死んでいる。生きているのは「安く・小さく・成果物非関与で勝てる崩れ方が実在するか」という確率の低い1シナリオだけであり、それを1日で見切る（プランA）か、ゲート制で最小コスト検証する（プランB）かは経営判断である。

> **経営判断（確定）:** 2026-07-18、ユーザーは**プランB（4ゲート逐次検証）を採択**。プランAは却下（redteamの論証は未反駁のまま記録として保存）。Round 4 は v4 への批判＋4ゲート実行設計（ベンチ台帳・弁護士3問・需要検証手順）のレビューに充てる。
