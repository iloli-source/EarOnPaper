# AIの耳 (ai-ears)

採譜結果を人間の耳なしで評価する自動ハーネス。G0'ゲート（競合ツール出力の品質確認）の事前スクリーニングを担い、将来の品質評価基盤（NF-019/020/021）の最初の実装を兼ねる。

## 構成

| ファイル | 役割 |
|---|---|
| `ears.py` | 計算の耳。元音源と採譜MIDIを数値比較（音高一致・出だし一致・テンポ整合・譜面健全性） |
| `gemini_ears.py` | 第二の耳。Gemini音声入力モデルに聴き比べさせ音楽的判定をJSONで得る |
| `synth_test.py` | ハーネス自体の自己検証（正解つき感度テスト） |
| `validation-report.md` | 自己検証の結果（7/7 PASS・既知の限界の正直な記録） |

## セットアップ

```bash
cd tools/ai-ears
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

## 使い方

```bash
# 元音源 vs 採譜結果の総合評価（Markdownレポートつき）
.venv/bin/python ears.py compare --original 元音源.mp3 --transcription 採譜結果.mid --report report.md

# 譜面健全性だけの単体検査（音源不要）
.venv/bin/python ears.py inspect --transcription 採譜結果.mid

# 音楽的判定（Gemini・クォータ回復後に実行可）
.venv/bin/python gemini_ears.py --original 元音源.mp3 --transcription 採譜結果.mid
```

## G0'での運用フロー

1. 検証3曲（ユーザー保有音源）を MuScriptor 等に投入 → 出力MIDIを取得（AI代行可）
2. 各出力を `ears.py compare` にかけ、曲×ツールのスコア表を作る
3. `gemini_ears.py` で音楽的判定（使える下書きか・手直し量の見立て・**問題が耳起因か記譜起因か**）を重ねる
4. 2つの耳のレポートを G0' 台帳（`docs/research/g0-prime-kit.md`）に転記して判定

## 指標の読み方（音楽用語なしで）

- **音高一致度**: ドレミの流れが元と同じか。0.85以上=ほぼ同じ曲、0.6未満=別物
- **出だし一致**: 音のタイミングが合っているか。precision低=幽霊音符が多い、recall低=取りこぼしが多い
- **テンポ整合**: 曲の速さの解釈が合っているか（倍速/半分は許容）
- **譜面健全性**: 細かすぎる音符・音域外・幽霊音符など「直すのが大変な譜面」の兆候
- **総合**: 0.80以上=高一致 / 0.60-0.80=部分一致 / 0.60未満=低一致

自己検証での実測感度: 同一0.90 → 音高改変0.83 → リズム崩し0.77 → 無関係0.59（validation-report.md）

## 限界（重要）

- このハーネスは**事前スクリーニング**であり、音楽家の代替ではない。「譜面として読みやすいか・演奏できるか・直す苦痛はどれほどか」は測れない
- スコアの絶対値は音色・録音条件に依存する。**同一曲でのツール間の相対比較**に使うのが正しい運用
- 本ハーネスはゲート判定の第一判定（事前封緘した客観指標）を担い、第二判定はターゲットユーザー（非音楽家）の聴き比べが担う。最終検証は**ユーザーテスト段階**（ターゲットユーザーの実利用）で行う。音楽関係者レビューは演奏者向け出力オプション（TAB等）のQAと製品レビューに限定し、プロダクトが「音声→五線譜PDF出力」を実現した後に実施（ユーザー方針 2026-07-19: つてはPDF出力後の製品レビューに温存）
